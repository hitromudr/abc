# abracadabra: Абстракция, Сущность, Протокол, Исполнение

Это когнитивный конвейер из двух агентов, предназначенный для дистилляции хаоса сложных процессов в кристально ясные, исполняемые **Концептуальные Протоколы** — и их автономной реализации.

## Два агента, одна задача

- **abra** (Архитектор) — принимает хаотичную проблему, прогоняет через философские фильтры, фреймворки и инженерные оси, генерирует Концептуальный Протокол.
- **cadabra** (Исполнитель) — получает от abra машиночитаемый контракт (`EXECUTION_STATE.md`), выполняет шаги DAG с обязательной верификацией каждого, честно останавливается при сбое.

Разделение мышления и действия — не случайность. Агент, который одновременно думает и делает, деградирует в обе стороны: анализ становится поверхностным (торопится к коду), код — хрупким (нет архитектурного фундамента).

## Концепция

Любая сложная задача, независимо от предметной области (разработка ПО, бизнес-процессы, инженерия, биология), является частным случаем фундаментального класса проблем.

Цель проекта:
1.  **Принять** хаотичную, многосоставную проблему (источник движения).
2.  **Дезинтегрировать** её, сняв слой предметного контекста.
3.  **Синтезировать** решения через кросс-дисциплинарный анализ (физика, теория систем, биология, логистика).
4.  **Выделить Сущность** — непреложные инварианты решения.
5.  **Сформировать Протокол** — концептуальный, но прагматичный алгоритм действий.
6.  **Исполнить** — автономно реализовать протокол через cadabra с пошаговой верификацией.

## Структура

Репозиторий — моно-репо с двумя модулями:

*   `abra/docs/01_БАЗА_ЗНАНИЙ/` — когнитивное ядро (фильтры, фреймворки, Октагон).
*   `abra/docs/02_ИНСТРУМЕНТЫ/` — шаблоны и алгоритмы обработки задач.
*   `abra/docs/03_РЕШЕНИЯ/` — библиотека кристаллизованных Концептуальных Протоколов.
*   `cadabra/` — системный промпт и инструменты агента-исполнителя.

## Мета-Конвейер: Как это работает

### Фаза Архитектора (abra)

Процесс создания нового протокола проходит 4 стадии абстракции:

1. **Дезинтеграция** — перевод специфической проблемы в абстрактную топологию.
2. **Кросс-дисциплинарный синтез** — поиск изоморфных проблем в других областях знаний.
3. **Выделение Сущности** — 3-5 фундаментальных законов, нарушение которых = провал.
4. **Формирование Протокола** — стандартизированный документ с алгоритмом перехода от проблемы к решению.

### Фаза Исполнителя (cadabra)

После утверждения оператором abra генерирует `EXECUTION_STATE.md`:

1. **Чтение контракта** — загрузка Scope, Kill Box, DAG.
2. **Последовательное исполнение** — шаг за шагом с верификацией каждого.
3. **Честная остановка** — при сбое: ERROR_LOG, status: blocked, STOP. Без героизма.
4. **Интеграционная верификация** — COMPLETION_PROOF доказывает решение задачи целиком.

## Анатомия Концептуального Протокола

Каждый протокол в `abra/docs/03_РЕШЕНИЯ/` строится по жесткой схеме:

1.  **Топология задачи (Ontology):** Фундаментальный класс системы/проблемы.
2.  **Инварианты (Invariants):** Незыблемые правила или законы.
3.  **Точка опоры (Leverage Point):** 20% усилий → 80% результата.
4.  **Векторы энтропии (Degradation paths):** Как процесс разрушается со временем.
5.  **Алгоритм стабилизации (The Protocol):** Пошаговая модель решения.
6.  **Метрика истины (Truth Criteria):** Объективные признаки решённости.
7.  **Эвристики и Мета-Контекст:** Открытые вопросы, тактические предложения.
8.  **Резолюция (Approval Gate):** Развилка для оператора + генерация EXECUTION_STATE.
