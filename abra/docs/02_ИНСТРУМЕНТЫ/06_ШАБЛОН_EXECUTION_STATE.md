# Шаблон: EXECUTION_STATE.md (Контракт Исполнителя)

Этот документ является строго структурированным контрактом (API) между Архитектором (`abra`) и Исполнителем (`cadabra`). 
Архитектор генерирует этот файл ТОЛЬКО после прохождения `Approval Gate` (утверждения варианта оператором). Исполнитель (`cadabra`) читает этот файл, выполняет шаги, обновляет статусы и возвращает управление.

### Диаграмма переходов состояний

```
  abra создаёт          оператор утверждает
  ┌───────┐  ────────────►  ┌──────────┐
  │ draft │                 │ approved │
  └───────┘                 └────┬─────┘
                                 │ cadabra начинает
                                 ▼
                           ┌─────────────┐
                      ┌──► │ in_progress │ ◄──┐
                      │    └──────┬──────┘    │
                      │           │           │
                      │     ┌─────┴─────┐     │
                      │     ▼           ▼     │
                  ┌─────────┐     ┌────────┐  │
                  │ blocked │     │  done  │  │
                  └────┬────┘     └────┬───┘  │
                       │               │      │
          abra правит  │               │      │
          контракт,    │               │      │
          оператор     │               │      │
          утверждает   └───────────────┼──────┘
                                       │
                         оператор      │
                         архивирует    ▼
                                 ┌──────────┐
                                 │ archived │
                                 └──────────┘
```

**Правила переходов:**
- `draft → approved` — только оператор (после ревью контракта)
- `approved → in_progress` — cadabra при начале исполнения
- `in_progress → in_progress` — cadabra при возобновлении после падения сессии (продолжает с первого незакрытого шага)
- `in_progress → done` — cadabra после прохождения всех шагов DAG + COMPLETION_PROOF
- `in_progress → blocked` — cadabra при провале верификации шага
- `blocked → approved` — abra после анализа ERROR_LOG и правки контракта + оператор утверждает
- `done → archived` — оператор (задача закрыта, контракт в архив)
- `blocked → archived` — оператор (задача отменена)
- **Запрещено:** `draft → in_progress` (без утверждения), `blocked → in_progress` (без abra), `done → in_progress` (без нового контракта)

## 2. ЖЕСТКИЕ ОГРАНИЧЕНИЯ (Kill Box / Иммунитет)
*Красные флажки из Уровня 0 (Инварианты).*
- `[MUST_NOT_DO]:` [Чего категорически нельзя делать (например: "не менять схему БД", "не переписывать логику авторизации")]
- `[MUST_NOT_DO]:` [Специфичный анти-паттерн, выявленный Архитектором]

**Правило пропорциональности:** Kill Box защищает от архитектурных ошибок, а не от баг-фиксов. Однострочное исправление документированного бага (нарушение контракта API, опечатка в enum, отсутствие обязательного параметра) — НЕ подпадает под Kill Box, даже если файл упомянут в ограничениях. Архитектор обязан явно пометить такие исключения: `[EXCEPTION]: <файл> — допустим баг-фикс <описание>`.
