## 8.1 Генерация Исполняемого Артефакта (Мост ABRA -> CADABRA)
*Трансляция утверждённого архитектурного решения в машиночитаемый контракт для автономного агента-исполнителя. Выполняется только ПОСЛЕ Approval Gate и явной команды оператора.*
- **Целевой артефакт:** Файл в целевом проекте пользователя (например, `.work/EXECUTION_STATE.md`).
- **Спецификация артефакта:** Обязан включать следующие секции:
  1. **`METADATA`**: Стейт-машина (draft / approved / in_progress / blocked / done / archived), автор, ссылка на исходный Концептуальный Протокол.
  2. **`CONTEXT`**: Физическая цель и строгий список разрешенных для редактирования файлов.
  3. **`KILL BOX`**: Прямая трансляция `[MUST_NOT_DO]` ограничений.
  4. **`DAG`**: Пошаговый направленный граф исполнения, где каждый шаг безальтернативно снабжен командой локальной валидации (`-> Проверка: [команда]`).
  5. **`COMPLETION_PROOF`**: Интеграционная команда (npm test, cargo build и т.д.), доказывающая успешность всей задачи целиком.
  6. **`ERROR_LOG`**: Обратный канал связи. Сюда агент-исполнитель обязан записать `stdout/stderr` при провале шага для возврата управления Архитектору (`abra`).
