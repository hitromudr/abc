# Оркестрация цикла abra ↔ cadabra

Полный жизненный цикл задачи в abracadabra. Три актора: **Оператор** (человек), **abra** (архитектор), **cadabra** (исполнитель). Агенты не общаются напрямую — весь обмен идёт через файл `EXECUTION_STATE.md` и оператора.

## Правила оркестрации

### 1. Оператор — единственный маршрутизатор
Агенты не вызывают друг друга. Оператор решает когда переключить управление. Это не баг — это Approval Gate на каждом переходе.

### 2. Апоптоз сессии между актами
Каждый агент работает в своём чате (сессии). После завершения работы агента оператор закрывает чат. Это защищает от context rot — каждый агент получает чистый контекст.

### 3. EXECUTION_STATE.md — единственный канал
Агенты обмениваются данными только через файл. abra пишет контракт, cadabra его исполняет и пишет ERROR_LOG. Никакой устной договорённости, никаких "подразумевается".

### 4. Оператор не чинит за cadabra
Если cadabra заблокирован — оператор не правит код сам. Он возвращает управление abra, который переосмысливает план. Иначе теряется трассируемость (кто что менял и почему).

### 5. Один контракт — одна задача
Запрещено добавлять новые цели в существующий EXECUTION_STATE. Новая задача = новый контракт. Старый архивируется.

## Где хранить артефакты

```
проект-пользователя/
└── .work/
    ├── EXECUTION_STATE.md        ← текущий активный контракт
    ├── archive/
    │   ├── 001_фича_корзина.md   ← завершённые контракты (status: archived)
    │   └── 002_фикс_авторизации.md
    └── protocols/
        └── ...                   ← Концептуальные Протоколы от abra (опционально)
```

- `.work/` — рабочая директория abracadabra в проекте пользователя
- Активный контракт всегда один: `.work/EXECUTION_STATE.md`
- После `done → archived` контракт переносится в `.work/archive/` с порядковым номером
- Протоколы (выход abra до генерации контракта) сохраняются опционально

## Граничные случаи

| Ситуация | Действие |
|----------|----------|
| cadabra нашёл ошибку в контракте (шаг требует файл вне Scope) | `blocked` + ERROR_LOG: "дефект контракта" |
| cadabra встретил развилку (два способа сделать шаг) | `blocked` + ERROR_LOG: "развилка, нужно решение abra" |
| Оператор хочет изменить требования в процессе | Останавливает cadabra → `blocked` → новый цикл через abra |
| COMPLETION_PROOF прошёл, но оператор видит проблему | Не менять `done`. Создать новый контракт на фикс |
| abra генерирует контракт, но оператор не согласен | Не утверждать (`draft`). Обсудить с abra, переработать |
| Сессия cadabra упала (таймаут, context rot) | Статус остаётся `in_progress`. Новая сессия cadabra подхватит с первого незакрытого шага |
| cadabra исчерпал Retry Budget (3 попытки) на шаге | `blocked` + ERROR_LOG с типом `retry_exhausted` и логами всех 3 попыток |
