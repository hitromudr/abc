# abracadabra

[![Version](https://img.shields.io/badge/version-2.1.0-green.svg)](CHANGELOG.md)
[![MIT License](https://img.shields.io/badge/license-MIT-blue.svg)](LICENSE)

**Когнитивный экзоскелет для LLM — жесткая архитектура вместо промпт-инженерии.**

LLM без ограничений — это океан энтропии: галлюцинации, context rot, слепая угодливость. abracadabra накладывает архитектурные печати — систему когнитивных фильтров, кибернетических ритуалов и матриц выживания, — заставляя модель работать по законам вашей физики.

## Два агента, один ритуал

abracadabra — двухфазный коммит (Two-Phase Commit), воплощающий закон [Двойной Спирали](abra/docs/00_ИНИЦИАЛИЗАЦИЯ/02_ДВОЙНАЯ_СПИРАЛЬ_РАЗРАБОТКИ.md) (Process & Product):

- **abra** (Архитектор) создаёт **Процесс**. Блокирует инстинкт немедленной генерации кода и прогоняет задачу через [8 философских фильтров](abra/docs/01_БАЗА_ЗНАНИЙ/01_АБСОЛЮТНЫЕ_ФИЛЬТРЫ_ВХОДА.md) и 8 инженерных осей [Октагона](abra/docs/01_БАЗА_ЗНАНИЙ/03_ИНЖЕНЕРНЫЙ_ОКТАГОН_ВЫЖИВАНИЯ.md). Если задача — иллюзия, `abra` её уничтожит. Если реальна — выкует архитектурный контракт.
- **cadabra** (Исполнитель) создаёт **Продукт**. Слепой голем-прораб: берёт [контракт](abra/docs/02_ИНСТРУМЕНТЫ/06_ШАБЛОН_EXECUTION_STATE.md) от `abra` и методично исполняет шаг за шагом. При сбое — 3 локальные попытки починки внутри разрешённого Scope, затем остановка и [возврат управления](abra/docs/02_ИНСТРУМЕНТЫ/07_ОРКЕСТРАЦИЯ_ЦИКЛА.md) Архитектору.

```
Оператор → abra init → [Ядро загружено] → abra [задача] → Концептуальный Протокол → Approval Gate
                                                                                         ↓
                                           Оператор утверждает → abra генерирует EXECUTION_STATE.md
                                                                                         ↓
                                           cadabra [путь к EXECUTION_STATE.md] → DAG → верификация → done/blocked
```

## Quick Start

### Внедрение в готовый проект (Основной сценарий)

abracadabra — это мета-фреймворк. Чтобы решать продуктовые задачи, его нужно подключить к вашему рабочему репозиторию.

1. Добавьте `abracadabra` как сабмодуль (или просто скопируйте папку) в корень вашего проекта:

```bash
git submodule add https://github.com/hitromudr/abc.git .abracadabra
```

2. Подключите системные правила к вашему AI-редактору:

| IDE | Настройка (выполнить в корне вашего проекта) |
|---|---|
| **Zed** | `ln -s .abracadabra/abra/core_rules.md .rules` |
| **Cursor** | `ln -s .abracadabra/abra/core_rules.md .cursorrules` |
| **Claude Code** | `cp .abracadabra/CLAUDE.md .` |

### Только анализ (любой web-чат)

1. Скачайте [`03_АВТОНОМНЫЙ_ПАЙПЛАЙН.md`](abra/docs/02_ИНСТРУМЕНТЫ/03_АВТОНОМНЫЙ_ПАЙПЛАЙН.md)
2. Прикрепите в чат ChatGPT / Claude / Gemini
3. Напишите задачу — модель выполнит полный цикл анализа

Это только **abra** (анализ и протокол). Для полного цикла (анализ + исполнение) нужна AI-IDE с доступом к файловой системе.

**Шаг 1 — Инициализация ядра.** В новом чате напишите:

```
abra init
```

Агент загрузит Базу Знаний в контекст и ответит «Ядро загружено. Жду задачу.»

**Шаг 2 — Постановка задачи.** После инициализации напишите:

```
abra [задача]
```

> **Можно ли сразу `abra [задача]`?** Да. Агент сам разобьёт на два такта: загрузит ядро, попросит нажать Continue, и только вторым шагом начнёт анализ. Но `abra init` → `abra [задача]` надёжнее: вы явно контролируете момент перехода.
>
> **Почему два такта?** LLM не может одновременно загрузить 10+ файлов архитектуры и качественно проанализировать сложную задачу. Совмещение вызывает коллапс внимания — модель начинает галлюцинировать фреймворки вместо чтения файлов. Разнесение на два generation-шага устраняет проблему.

**Шаг 3 — Исполнение.** После утверждения протокола оператором, Архитектор сгенерирует контракт (обычно в `.work/EXECUTION_STATE.md`). Откройте новый чат и запустите Исполнителя:

```
cadabra .work/EXECUTION_STATE.md
```

cadabra прочитает контракт и начнёт исполнение DAG.

## До / После

**Задача:** *"У нас микросервисы тормозят, что делать?"*

**Без abracadabra** (типичный ответ LLM):
> "Рекомендую оптимизировать запросы к базе данных, внедрить кеширование Redis, рассмотреть переход на gRPC, использовать connection pooling..."

Шаблонный список без диагностики. Не известно, что именно тормозит.

**С abracadabra** (после прогона через конвейер):
> **Задача отклонена ([Фаза 0](abra/docs/02_ИНСТРУМЕНТЫ/01_АЛГОРИТМ_РАЗБОРА_ЗАДАЧИ.md)).** Отсутствуют метрики: p99 latency, RPS, flamegraph, трейсы. "Тормозят" — это языковая иллюзия ([Фильтр Витгенштейна](abra/docs/01_БАЗА_ЗНАНИЙ/01_АБСОЛЮТНЫЕ_ФИЛЬТРЫ_ВХОДА.md)). Предоставьте: 1) трейс конкретного медленного запроса, 2) график latency за последнюю неделю, 3) текущий CPU/RAM на узлах.

abra заставил модель *отказаться решать несуществующую проблему* и потребовать физические данные.

## Когда НЕ использовать

abracadabra — экзоскелет, а не скафандр. Он для задач, требующих декомпозиции (Complicated/Complex по [Cynefin](abra/docs/01_БАЗА_ЗНАНИЙ/02_КОГНИТИВНЫЙ_ЛАНДШАФТ.md#7-фреймворк-cynefin-сноуден)). Не запускайте, если:

- **Задача простая** — известное решение, не нужно 6 фаз
- **Задача творческая** — нарратив, дизайн, брейншторм (формализация убивает нелинейное мышление)
- **Нужна скорость** — ответ за 10 секунд, а не архитектурный документ
- **Нужен разговор** — обсуждение, размышление вслух
