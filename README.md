# abracadabra v2.0.0: Когнитивный Экзоскелет для LLM

Набор алгоритмов и философских фильтров, превращающий LLM из вежливого болтуна в хладнокровного инженера-архитектора.

**Проблема:** 90% ответов LLM — убедительный, отформатированный мусор. Модель угадывает "Best Practice" из обучающей выборки вместо того, чтобы думать.

**Решение:** abracadabra — два агента, работающих в паре:
- **abra** (Архитектор) — когнитивный конвейер, который принудительно прогоняет задачу через 8 философских фильтров, 13 фреймворков и 8 инженерных осей выживания *до* генерации первого слова ответа.
- **cadabra** (Исполнитель) — автономный прораб, который берёт план от abra и выполняет его пошагово с обязательной верификацией каждого шага.

## Quick Start

### Только анализ (любой web-чат)
1. Скачайте один файл: [`03_АВТОНОМНЫЙ_ПАЙПЛАЙН.md`](abra/docs/02_ИНСТРУМЕНТЫ/03_АВТОНОМНЫЙ_ПАЙПЛАЙН.md)
2. Прикрепите его в чат ChatGPT / Claude / Gemini
3. Напишите задачу — модель выполнит полный цикл анализа

Это только **abra** (Архитектор) — анализ задачи и генерация протокола. Для полного цикла (анализ + исполнение через cadabra) нужна AI-IDE с доступом к файловой системе.

### Полный цикл (AI-IDE)

Клонируйте репо. Дальше зависит от среды:

| IDE | Что делать |
|---|---|
| **Zed** | Ничего. `.rules` подхватывается автоматически. Пишите `abra [задача]` в чат |
| **Cursor** | `.cursorrules` уже в репо (симлинк). Пишите `abra [задача]` в чат |
| **Claude Code** | Ничего. Работает через `CLAUDE.md` (уже в репо) |

В AI-IDE доступны оба агента: `abra [задача]` для анализа, `cadabra [путь]` для исполнения.

## До / После

**Задача:** *"У нас микросервисы тормозят, что делать?"*

**Без abracadabra** (типичный ответ LLM):
> "Рекомендую оптимизировать запросы к базе данных, внедрить кеширование Redis, рассмотреть переход на gRPC, использовать connection pooling..."

Шаблонный список без диагностики. Не известно, что именно тормозит.

**С abracadabra** (после прогона через конвейер):
> **Задача отклонена (Фаза 0).** Отсутствуют метрики: p99 latency, RPS, flamegraph, трейсы. "Тормозят" — это языковая иллюзия (Фильтр Витгенштейна). Предоставьте: 1) трейс конкретного медленного запроса, 2) график latency за последнюю неделю, 3) текущий CPU/RAM на узлах.

abra заставил модель *отказаться решать несуществующую проблему* и потребовать физические данные.

### Два агента, одна задача

**abra** анализирует задачу и генерирует Концептуальный Протокол — архитектурный документ с инвариантами, точкой опоры, алгоритмом стабилизации. После утверждения оператором abra создаёт `EXECUTION_STATE.md` — машиночитаемый контракт для cadabra.

**cadabra** берёт контракт и выполняет шаги DAG один за другим. Каждый шаг верифицируется командой проверки. При ошибке cadabra не пытается чинить — он останавливается и заполняет ERROR_LOG для возврата управления abra.

```
Оператор → abra [задача] → Концептуальный Протокол → Approval Gate
                                                          ↓
                              Оператор выбирает вариант → abra генерирует EXECUTION_STATE.md
                                                          ↓
                              cadabra [путь] → исполнение DAG → верификация → done/blocked
```

## Когда НЕ использовать

abracadabra — экзоскелет, а не скафандр. Он для задач, требующих декомпозиции и структурирования (Complicated/Complex по Cynefin¹). Не запускайте полный цикл, если:

- **Задача простая** — известное решение, не нужно 6 фаз чтобы ответить
- **Задача творческая** — нарратив, дизайн, брейншторм. Формализация убивает нелинейное мышление
- **Нужна скорость** — ответ за 10 секунд, а не архитектурный документ
- **Нужен разговор** — обсуждение, размышление вслух

---
¹ **Cynefin** — фреймворк Дэйва Сноудена: Clear (делай по инструкции), Complicated (нужна экспертиза), Complex (зондируй и адаптируйся), Chaotic (сначала действуй).

## Ключевые механизмы

**8 Абсолютных Фильтров** — пре-чеклист по 8 измерениям невалидности задачи:
1. Витгенштейн (Язык) — реальный конфликт или языковая иллюзия?
2. Декарт (Знание) — в чём я математически уверен?
3. Сенека (Действие) — это в зоне контроля?
4. Аристотель (Цель) — какова истинная цель (Телос) системы?
5. Джеймс (Результат) — решение изменит физическую реальность завтра?
6. Хаммурапи (Причинность) — какой базовый контракт нарушен?
7. Сунь Цзы (Время) — сейчас ли момент действовать?
8. Платон (Масштаб) — я смотрю на правильный уровень?

**Инженерный Октагон** — 8 осей, по которым проверяется любое решение:
Топология → Метаболизм → Кинематика → Наследственность → Иммунитет → Гомеостаз → Мутагенез → Телос

**Двойная Спираль** — каждая задача обязана двигать Product (решение) и Process (улучшение системы) одновременно. Одноразовые фиксы запрещены.

**Кросс-доменные инварианты** — фундаментальные законы из 6 областей (термодинамика, кибернетика, биология, теория игр, когнитивистика, семиотика), заземлённые в инженерные паттерны.
