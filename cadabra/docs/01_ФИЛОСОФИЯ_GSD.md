# Философия cadabra: Управляемый GSD vs Оркестрированный GSD

## GSD — фреймворк для агентского кодинга

GSD (Get Shit Done) — [фреймворк](https://github.com/gsd-build/get-shit-done) для Claude Code: npm-пакет с набором slash-команд, превращающий одного агента в оркестратора пула субагентов. Цикл GSD:

```
Discuss → Plan → Execute (waves) → Verify
```

Ключевые механизмы GSD:
- **Субагент-оркестрация.** 4 research-агента параллельно → planner+checker → executor waves → verifier+debugger. Каждый субагент получает свежий контекст (~200K).
- **Wave-based параллелизм.** Независимые задачи группируются в волны и исполняются параллельно.
- **Spec-driven.** PROJECT.md, REQUIREMENTS.md, ROADMAP.md, STATE.md, CONTEXT.md — цепочка документов, задающих каркас проекта.
- **Debug-агент.** При сбое — автоматическая диагностика выделенным агентом, план фикса, повтор.
- **Atomic commits.** Один коммит на задачу для git bisect.

## Почему abracadabra — не GSD

GSD решает проблему **исполнения**: как эффективно превратить спецификацию в код. abracadabra решает проблему **мышления**: как отличить реальную задачу от иллюзии до написания первой строки кода.

```
GSD:           Spec ─────────────────────→ Code (максимально эффективно)
                                           ↑ оркестрация, параллелизм, debug

abracadabra:   Хаос → [abra: фильтры] → Протокол → Approval → [cadabra: DAG] → Code
                       ↑ 8 фильтров,                           ↑ последовательно,
                         13 фреймворков,                          с верификацией
                         Октагон
```

GSD предполагает, что спецификация уже существует и корректна. abra проверяет саму постановку задачи — не является ли она языковой иллюзией (Витгенштейн), не выходит ли за зону контроля (Сенека), есть ли вообще метрики успеха (Эмпирическое заземление).

## Архитектурные различия

| Аспект | GSD | abracadabra |
|--------|-----|-------------|
| **Агенты** | Один оркестратор + пул субагентов | Два агента с жёстким контрактом |
| **Планирование** | Встроено в цикл (Discuss → Plan) | Вынесено в отдельного агента (abra) |
| **Параллелизм** | Wave-based: независимые задачи параллельно | Последовательный DAG |
| **Context rot** | Свежий контекст на каждый субагент | Разделение мышления (abra) и действия (cadabra) |
| **State** | 5 файлов (PROJECT, REQUIREMENTS, ROADMAP, STATE, CONTEXT) | 1 файл: EXECUTION_STATE.md |
| **Верификация** | Встроена в каждый task | Встроена в каждый шаг DAG |
| **При ошибке** | Debug-агент → fix plan → retry | STOP → ERROR_LOG → возврат abra |
| **Когнитивный слой** | Нет (чистая инженерия) | 8 фильтров, 13 фреймворков, Октагон |
| **Approval Gate** | Нет (агент решает сам) | Да (оператор утверждает вариант) |
| **Установка** | npm пакет, slash-команды | Markdown-файлы, zero dependencies |

## Где GSD сильнее

1. **Параллелизм.** Wave-based execution — принципиально быстрее последовательного DAG cadabra на задачах с независимыми подзадачами.
2. **Автовосстановление.** Debug-агент GSD диагностирует сбой и генерирует fix plan. cadabra при сбое останавливается — ремонт требует возврата к abra и нового контракта.
3. **Субагентная изоляция.** Каждый субагент GSD получает свежие ~200K контекста. cadabra работает в одном контексте, который деградирует по мере исполнения DAG.
4. **Tooling.** npm install, slash-команды, интеграция с Claude Code из коробки.

## Где abracadabra сильнее

1. **Пре-анализ.** GSD начинает с готовой спецификации. abra проверяет саму задачу на валидность до генерации спецификации. 8 философских фильтров отсеивают иллюзорные, невыполнимые или преждевременные задачи.
2. **Blast radius.** cadabra имеет жёсткий Scope (список разрешённых файлов) и Kill Box (список запретов). Файл вне периметра = немедленный STOP. У GSD субагенты работают в рамках task-файлов, но без формализованного Kill Box.
3. **Approval Gate.** abra останавливается при развилке и ждёт решения оператора. GSD-оркестратор принимает архитектурные решения автономно.
4. **Zero dependencies.** Markdown-файлы работают в любой среде с любой LLM. GSD привязан к Claude Code и npm.

## Позиционирование cadabra

cadabra — не конкурент GSD. Это другой уровень абстракции:

- **GSD** = инженерия исполнения (как делать быстро и надёжно)
- **cadabra** = управляемое исполнение с внешним мозгом (как делать ровно то, что нужно)

GSD оптимизирует пропускную способность конвейера. abracadabra оптимизирует качество входа в конвейер.

```
cadabra = GSD − Автономность − Параллелизм + Scope-изоляция + Kill Box + Внешний план (abra)
```

Или: **GSD без свободы воли, с предохранителями и внешним мозгом**.
