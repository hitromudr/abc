# Формат ERROR_LOG

Спецификация обратного канала cadabra → abra. Заполняется в секции `4. ERROR_LOG` файла `EXECUTION_STATE.md` при переводе статуса в `blocked`.

---

## Обязательные поля

```markdown
- **Проваленный шаг:** [номер шага из DAG]
- **Тип блокировки:** [ verification_failed | scope_violation | contract_defect | fork_detected | kill_box_violation | retry_exhausted ]
- **Попытки:** [использованные из Retry Budget]
- **Симптом:** [1-2 предложения]
- **Вывод среды (stdout/stderr):**
\```text
[Полный вывод команды проверки. Без редактирования. Если >200 строк — последние 200]
\```
- **Контекст:** [Какие файлы тронуты, какие изменения внесены до сбоя]
- **Выполненные шаги:** [номера со статусом ✓]
```

## Типы блокировок

| Тип | Когда |
|-----|-------|
| `verification_failed` | Команда проверки вернула ненулевой код |
| `scope_violation` | Шаг требует файл не из Scope |
| `contract_defect` | Контракт содержит ошибку (нет файла, противоречие) |
| `fork_detected` | Несколько способов выполнить шаг, cadabra не выбирает |
| `kill_box_violation` | Выполнение неизбежно нарушит MUST_NOT_DO |
| `retry_exhausted` | 3 попытки исчерпаны, тест красный |

## Правила

1. **Не интерпретируй.** Факты, не гипотезы.
2. **Не обрезай stdout.** abra нужен полный лог.
3. **Фиксируй контекст.** Какие файлы изменены, какие шаги пройдены.
4. **Один ERROR_LOG на блокировку.** При повторной блокировке — предыдущий стирается.
