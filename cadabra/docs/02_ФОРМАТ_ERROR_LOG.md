# Формат ERROR_LOG

Спецификация обратного канала cadabra → abra. ERROR_LOG заполняется в секции `4. ЖУРНАЛ ОШИБОК` файла `EXECUTION_STATE.md` при переводе статуса в `blocked`.

---

## Обязательные поля

```markdown
## 4. ЖУРНАЛ ОШИБОК (ERROR_LOG / Гомеостаз)
- **Проваленный шаг:** [номер шага из DAG]
- **Тип блокировки:** [ verification_failed | scope_violation | contract_defect | fork_detected | kill_box_violation ]
- **Симптом:** [Что именно пошло не так — 1-2 предложения]
- **Вывод среды (stdout/stderr):**
```text
[Сырой лог. Без редактирования. Без обрезки. Полный вывод команды проверки]
```
- **Контекст:** [Что cadabra успел сделать до сбоя на этом шаге — какие файлы тронул, какие изменения внёс]
- **Выполненные шаги:** [список номеров шагов со статусом ✓]
```

---

## Типы блокировок

| Тип | Когда | Пример |
|-----|-------|--------|
| `verification_failed` | Команда проверки шага вернула ненулевой код или неожиданный вывод | `npx eslint src/cart.ts` → 3 errors |
| `scope_violation` | Шаг DAG требует файл не из секции Scope | Шаг 3 требует `src/auth.ts`, но он не в Scope |
| `contract_defect` | Контракт содержит ошибку: нет команды проверки, ссылка на несуществующий файл, противоречие между шагами | Шаг 2: "Изменить файл X", но файл X не существует |
| `fork_detected` | Два или более способов выполнить шаг, cadabra не имеет права выбирать | "Добавить обработку ошибок" — try/catch или Result type? |
| `kill_box_violation` | Выполнение шага неизбежно нарушит MUST_NOT_DO | Шаг требует изменить схему БД, Kill Box запрещает |

---

## Правила заполнения

1. **Не интерпретируй.** Записывай факты, не гипотезы. "Тест упал с exit code 1" — факт. "Видимо, проблема в зависимости" — интерпретация (запрещена).

2. **Не обрезай stdout.** abra нужен полный лог для диагностики. Если лог длиннее 200 строк — записывай последние 200, но укажи "обрезано, полный лог: [количество] строк".

3. **Фиксируй контекст.** Какие файлы уже изменены? Какие шаги уже пройдены? abra должен понимать, в каком состоянии находится проект.

4. **Один ERROR_LOG на одну блокировку.** Если cadabra блокируется повторно после разблокировки abra — предыдущий ERROR_LOG стирается, пишется новый.

---

## Пример заполненного ERROR_LOG

```markdown
## 4. ЖУРНАЛ ОШИБОК (ERROR_LOG / Гомеостаз)
- **Проваленный шаг:** 3
- **Тип блокировки:** verification_failed
- **Симптом:** Unit-тест cart.test.ts падает — функция addItem возвращает undefined вместо обновлённого массива
- **Вывод среды (stdout/stderr):**
```text
$ npm test -- cart.test.ts

FAIL src/__tests__/cart.test.ts
  ● addItem › should return updated cart

    expect(received).toEqual(expected)

    Expected: [{"id": 1, "qty": 2}]
    Received: undefined

      12 |   const result = addItem(cart, { id: 1, qty: 1 });
      13 |   expect(result).toEqual([{ id: 1, qty: 2 }]);
         |                  ^
      14 | });

Test Suites: 1 failed, 1 total
Tests:       1 failed, 2 passed, 3 total
```
- **Контекст:** Создал файл `src/cart.ts` (шаг 1 ✓), написал интерфейсы (шаг 2 ✓). На шаге 3 реализовал функцию addItem, но проверка показывает неверный return.
- **Выполненные шаги:** 1 ✓, 2 ✓
```
