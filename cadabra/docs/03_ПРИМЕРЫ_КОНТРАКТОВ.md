# Примеры контрактов EXECUTION_STATE.md

Три сценария: успешное выполнение, блокировка на верификации, блокировка на дефекте контракта.

---

## Пример 2: Блокировка на верификации (status: blocked)

```markdown
# EXECUTION_STATE.md

## 0. METADATA (Кинематика)
- **Status:** blocked
- **Source Protocol:** Протокол "API эндпоинт /orders"
- **Updated:** 2026-03-10T16:45:00

## 1. КОНТЕКСТ ДЛЯ CADABRA (Топология)
- **Цель:** Создать REST эндпоинт POST /orders с валидацией и сохранением в БД.
- **Разрешенный Периметр (Scope):**
  - `src/routes/orders.ts`
  - `src/validators/order.ts`
  - `src/routes/orders.test.ts`

## 2. ЖЕСТКИЕ ОГРАНИЧЕНИЯ (Kill Box / Иммунитет)
- `[MUST_NOT_DO]:` Не менять схему БД (миграции)
- `[MUST_NOT_DO]:` Не менять middleware авторизации

## 3. ГРАФ ИСПОЛНЕНИЯ (DAG)
- [x] **Шаг 1:** Создать `src/validators/order.ts` со схемой валидации Zod
  - `Проверка:` `npx tsc --noEmit src/validators/order.ts`
- [x] **Шаг 2:** Создать `src/routes/orders.ts` с хэндлером POST /orders
  - `Проверка:` `npx tsc --noEmit src/routes/orders.ts`
- [ ] **Шаг 3:** Создать тесты `src/routes/orders.test.ts`
  - `Проверка:` `npm test -- orders.test.ts`

## 4. ЖУРНАЛ ОШИБОК (ERROR_LOG / Гомеостаз)
- **Проваленный шаг:** 3
- **Тип блокировки:** verification_failed
- **Симптом:** Тест POST /orders → 500 вместо 201. Хэндлер пытается вставить запись в таблицу `orders`, но колонка `customer_id` имеет NOT NULL constraint, а в тестовых данных это поле отсутствует.
- **Вывод среды (stdout/stderr):**
```text
$ npm test -- orders.test.ts

FAIL src/routes/orders.test.ts
  ● POST /orders › should create order

    expected 201 "Created", got 500 "Internal Server Error"

    Error: insert into "orders" ("product_id", "qty")
    - null value in column "customer_id" violates not-null constraint

Test Suites: 1 failed, 1 total
Tests:       1 failed, 1 passed, 2 total
```
- **Контекст:** Валидатор (шаг 1) и хэндлер (шаг 2) не включают поле customer_id — его нет в контракте. Вероятно, ошибка в спецификации (контракт не учёл обязательное поле БД).
- **Выполненные шаги:** 1 ✓, 2 ✓

## 5. ИНТЕГРАЦИОННАЯ ВЕРИФИКАЦИЯ (COMPLETION_PROOF)
- **Команда проверки:** `npm test`
- **Ожидаемый результат:** All tests passed
- **Статус верификации:** pending
```

## Пример 3: Блокировка на дефекте контракта (status: blocked)

```markdown
# EXECUTION_STATE.md

## 0. METADATA (Кинематика)
- **Status:** blocked
- **Source Protocol:** Протокол "Рефакторинг логгера"
- **Updated:** 2026-03-10T11:20:00

## 1. КОНТЕКСТ ДЛЯ CADABRA (Топология)
- **Цель:** Заменить console.log на структурированный логгер (pino) в модуле payments.
- **Разрешенный Периметр (Scope):**
  - `src/payments/processor.ts`
  - `src/payments/webhook.ts`

## 2. ЖЕСТКИЕ ОГРАНИЧЕНИЯ (Kill Box / Иммунитет)
- `[MUST_NOT_DO]:` Не менять бизнес-логику процессинга платежей

## 3. ГРАФ ИСПОЛНЕНИЯ (DAG)
- [x] **Шаг 1:** Заменить console.log на logger.info в `src/payments/processor.ts`
  - `Проверка:` `grep -r "console.log" src/payments/processor.ts | wc -l` → должно быть 0
- [ ] **Шаг 2:** Заменить console.log на logger.info в `src/payments/webhook.ts`
  - `Проверка:` `grep -r "console.log" src/payments/webhook.ts | wc -l` → должно быть 0
- [ ] **Шаг 3:** Добавить конфигурацию pino в `src/logger.ts`
  - `Проверка:` `npx tsc --noEmit src/logger.ts`

## 4. ЖУРНАЛ ОШИБОК (ERROR_LOG / Гомеостаз)
- **Проваленный шаг:** 2
- **Тип блокировки:** scope_violation
- **Симптом:** Файл `src/payments/webhook.ts` импортирует логгер из `src/logger.ts`. Но `src/logger.ts` ещё не существует (шаг 3 идёт после шага 2). Чтобы шаг 2 прошёл проверку, нужно сначала выполнить шаг 3. Это ошибка порядка шагов в DAG.
- **Вывод среды (stdout/stderr):**
```text
$ npx tsc --noEmit src/payments/webhook.ts
src/payments/webhook.ts:1:25 - error TS2307: Cannot find module '../logger' or its corresponding type declarations.

1 import { logger } from '../logger';
                          ~~~~~~~~~~
```
- **Контекст:** Шаг 1 выполнен, но processor.ts тоже ссылается на '../logger'. Однако проверка шага 1 (grep) не ловит ошибку импорта — она проверяет только отсутствие console.log.
- **Выполненные шаги:** 1 ✓ (проверка прошла, но с латентным дефектом)

## 5. ИНТЕГРАЦИОННАЯ ВЕРИФИКАЦИЯ (COMPLETION_PROOF)
- **Команда проверки:** `npx tsc --noEmit && npm test`
- **Ожидаемый результат:** No errors
- **Статус верификации:** pending
```

**Что abra должен исправить:** Переупорядочить DAG — шаг 3 (создание logger.ts) должен быть первым. Также усилить проверку шага 1: вместо grep добавить `npx tsc --noEmit`.
