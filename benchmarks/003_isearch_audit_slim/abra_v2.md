## 1. Топология задачи (Ontology)

- **Мета-тип:** Аудит целостности распределённой системы с ML-конвейером и персистентным состоянием (векторная БД).
- **Ключевой конфликт:** Быстрая итерация фич (семантический поиск, кластеризация, граф зависимостей) vs отсутствие контрактов на границах компонентов (API ↔ VectorStore ↔ Embedder). Система работает в happy path, но не защищена от отклонений.

## 2. Инварианты (Invariants)

1. **Закон детерминизма ID:** Идентификатор точки в векторной БД обязан быть стабильным между перезапусками процесса для одних и тех же данных. Нарушение → дубликаты, раздувание индекса.
2. **Предел атомарности индексации:** Параллельная индексация одного проекта не должна быть возможна. Нарушение → повреждение индекса.
3. **Принцип замкнутости периметра:** Имя проекта, переданное через URL, не должно позволять выход за пределы `PROJECTS_BASE_DIR`. Нарушение → чтение произвольных файлов.

## 3. Точка опоры (Leverage Point)

- **Фокус внимания:** Слой VectorStore (генерация ID + управление коллекциями) — единственная точка, где данные обретают персистентность. Все остальные дефекты вторичны, если данные в индексе повреждены или недетерминированы.
- **Игнорируемый шум:** Стилистические замечания (мёртвый код в `index.py`/`search.py`, дублирование комментариев) — не влияют на работоспособность.

## 4. Векторы энтропии (Degradation Paths)

- **Анти-паттерн А — Раздувание индекса:** Каждый перезапуск процесса (новый `PYTHONHASHSEED`) генерирует новые ID для тех же чанков → дубликаты накапливаются. Поиск деградирует (дубли в результатах, рост latency).
- **Когнитивное искажение Б — "Работает на моей машине":** `.env` содержит конфликтующие значения (`QDRANT_URL=./qdrant_local_data`), что маскирует баг конфигурации. Разработчик переключается между docker-режимом и local-режимом, забывая обновить `.env`.

## 5. Алгоритм стабилизации (The Protocol)

### Фаза 3: Интеграция — тесты на пограничные случаи

- Добавить тест на конкурентную индексацию.

## 6. Метрика истины (Truth Criteria)

- **Состояние успеха:** Повторная индексация того же проекта после перезапуска процесса не создаёт дубликатов в Qdrant. `project_name=../../etc/passwd` возвращает 400/422. Параллельные запросы на индексацию одного проекта сериализуются.
- **Ложный сигнал успеха:** "Все тесты проходят" — текущие тесты не покрывают ни один из критических дефектов (нет тестов на ID-стабильность, path traversal, конкурентность).

## 7. Реестр дефектов

### CRITICAL

#### C-1. Недетерминистичные ID точек → дубликаты в индексе
- **Файл:** `src/vector_store.py:90`
- **Суть:** `abs(hash(p.get('source_file', '') + p.get('text', '')))` использует встроенный `hash()`, который рандомизирован по `PYTHONHASHSEED` (по умолчанию в Python 3.3+). При перезапуске процесса тот же чанк получит другой ID. `upsert` с новым ID = новая точка, а не обновление → раздувание коллекции дубликатами.
- **Импакт:** Экспоненциальный рост размера коллекции при каждом re-index. Деградация качества поиска (дубли в результатах). Рост потребления RAM/диска Qdrant.
- **Дополнительно:** `abs(hash(...))` также даёт коллизии: два разных чанка могут получить одинаковый ID → один перезапишет другой, потеря данных.

#### C-2. Path Traversal через project_name
- **Файл:** `src/main.py:246,271,354,514,534`
- **Суть:** `project_name` из URL подставляется напрямую в `projects_base_dir / project_name` без валидации. Значение `../../etc` пройдёт проверку `.is_dir()` и даст доступ к произвольным директориям ФС через эндпоинты `/projects/{project_name}/index`, `/search`, `/graph`.
- **Импакт:** Чтение содержимого произвольных файлов через индексацию + поиск. Информационная утечка. При наличии CORS `*` — эксплуатируемо из браузера.

#### C-3. Отсутствие qdrant-client в зависимостях
- **Файл:** `pyproject.toml`, `requirements.txt`
- **Суть:** Пакет `qdrant-client` не указан ни в `pyproject.toml`, ни в `requirements.txt`, хотя используется как прямая зависимость в `vector_store.py:4` и `main.py:33`.
- **Импакт:** Чистая установка через `pip install` не установит qdrant-client → `ImportError` при запуске.

### HIGH

#### H-1. Race condition при параллельной индексации
- **Файл:** `src/main.py:230-253`
- **Суть:** Нет блокировки per-project. Два одновременных POST на `/projects/X/index` запустят две параллельные задачи, которые будут одновременно сканировать файлы, удалять точки и вставлять новые → повреждение индекса.
- **Импакт:** Потеря данных, неконсистентный индекс.

#### H-2. Unbounded memory growth в TASK_STATUS
- **Файл:** `src/main.py:51`
- **Суть:** `TASK_STATUS: Dict[str, Dict[str, Any]] = {}` — in-memory словарь без TTL и eviction. Каждый запрос на индексацию добавляет запись.
- **Импакт:** При длительной работе сервиса → unbounded memory growth → OOM kill.

#### H-3. CORS allow_origins=["*"] + allow_credentials=True
- **Файл:** `src/main.py:176`
- **Суть:** Спецификация CORS запрещает эту комбинацию. Фактически открывает API для запросов с любого origin.
- **Импакт:** Внешний сайт может удалить все индексы, запустить индексацию произвольных путей (в сочетании с C-2).

#### H-4. Параметр reindex объявлен, но не используется
- **Файл:** `src/main.py:233`
- **Суть:** `reindex: bool = Query(False)` принимается API, но не передаётся в `run_indexing_task()`. Кнопка "Full Re-index" на дашборде отправляет `?reindex=true`, но backend его игнорирует.
- **Импакт:** Нарушение API-контракта. Пользователь ожидает полную переиндексацию, получает инкрементальную.

#### H-5. Scroll limit 10000 в кластеризации без пагинации
- **Файл:** `src/main.py:449`
- **Суть:** `vector_store.client.scroll(..., limit=10000)[0]` — жёсткий лимит без пагинации. Проекты с >10000 чанков — данные усечены.
- **Импакт:** Некорректные результаты кластеризации для крупных проектов.

### MEDIUM

#### M-1. Конфликтующая конфигурация в .env
- **Файл:** `.env`
- **Суть:** Два присвоения `PROJECTS_BASE_DIR` — последнее (`..`) перезаписывает первое. `QDRANT_URL` задан как путь, а не HTTP URL.

#### M-2. Перепутаны комментарии к портам docker-compose
- **Файл:** `docker-compose.yml:8-9`
- **Суть:** Комментарий `# gRPC port` на 6333, `# REST API port` на 6334. В реальности наоборот.

#### M-3. Отсутствие payload index на file_type
- **Файл:** `src/vector_store.py:51-65`
- **Суть:** Нет payload index для `file_type` → full scan при фильтрации scope=code/docs.

#### M-4. delete_points_by_source — O(n) фильтр с should
- **Файл:** `src/vector_store.py:181-200`
- **Суть:** `should` (OR) фильтр со всеми файлами. При 1000+ файлов → огромный OR-фильтр.

#### M-5. Мутация score при keyword boosting
- **Файл:** `src/main.py:397-400`
- **Суть:** `top_hit.score = min(1.0, top_hit.score + boost_factor)` — прямая мутация Pydantic-объекта. Boost `+0.1` может радикально изменить ранжирование.

#### M-6. Мёртвый код: index.py и search.py
- **Файлы:** `src/index.py`, `src/search.py`
- **Суть:** Broken imports (`from embedder` без `src.`), устаревшая логика, не вызываются из main.py.

- **Суть:** `torch`, `sentence-transformers`, `fastapi` — без закреплённых версий.

#### L-2. Health check не проверяет связность с Qdrant

#### L-3. Нет HTTPS/аутентификации
