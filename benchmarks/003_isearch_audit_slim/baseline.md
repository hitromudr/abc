## Аудит кодовой базы проекта `isearch`

### Отчёт Senior Software/System Engineer

**Дата аудита:** 2024-07-28
**Версия проекта:** `isearch` (slim abra)

---

### Общее впечатление и архитектура

Проект `isearch` представляет собой хорошо структурированное приложение для семантического поиска и анализа графов кода, построенное на FastAPI, SentenceTransformers и Qdrant. Архитектура выглядит модульной, с чётким разделением ответственности между сервисами (эмбеддинги, векторное хранилище, индексация, построение графов, API). Использование Click CLI (`manage.py`) для управления сервисами и взаимодействия с API улучшает удобство разработки и эксплуатации.

Однако, в процессе глубокого аудита были выявлены несколько критичных и высокоприоритетных уязвимостей, касающихся целостности данных, масштабируемости, безопасности и эффективности ML-пайплайна.

---

### Классификация дефектов и оценка критичности

#### 1. Дефекты целостности данных и векторного хранилища (Qdrant)

*   **Название:** Недетерминированные ID точек Qdrant из-за `hash()`
    *   **Описание:** В `src/vector_store.py` для генерации уникальных ID точек Qdrant используется `abs(hash(payload.get('source_file', '') + payload.get('text', ''))))`. Функция `hash()` в Python по умолчанию не гарантирует детерминированность между запусками интерпретатора и даже в рамках одного процесса, если переменная окружения `PYTHONHASHSEED` не установлена. Это означает, что при повторной индексации одного и того же файла (или его чанка) может быть сгенерирован новый ID, что приведёт к дублированию точек вместо их обновления, или к некорректному удалению старых точек.
    *   **Расположение:** `src/vector_store.py:100` (`VectorStore.upsert`)
    *   **Критичность:** **Critical (GT-3)**
    *   **Обоснование:** Прямо влияет на целостность данных в векторной базе. Может привести к экспоненциальному росту объёма индекса, некорректным результатам поиска (дубликаты), и невозможности надёжной инкрементальной индексации. `abs()` дополнительно уменьшает пространство хэшей, увеличивая вероятность коллизий.

*   **Название:** Отсутствие блокировок для инкрементальной индексации (Race Condition)
    *   **Описание:** Метод `IndexingService.run_indexing` запускается асинхронно через FastAPI `BackgroundTasks`. Если несколько запросов на индексацию одного и того же проекта будут отправлены одновременно (например, два пользователя нажимают "Index"), то `run_indexing_task` будет запущен параллельно. Это приведёт к гонке данных при обновлении (удалении/вставке) одних и тех же точек в Qdrant. Векторная база данных Qdrant, хотя и оптимизирована для параллельных запросов, не может самостоятельно разруливать логику инкрементального обновления, которая включает в себя "удалить старые чанки X, вставить новые чанки Y" для одного и того же файла. Это может привести к потере данных, частичным обновлениям или ошибкам.
    *   **Расположение:** `src/main.py:214` (`index_project_endpoint`), `src/services.py:180` (`IndexingService.run_indexing`)
    *   **Критичность:** **High (GT-4)**
    *   **Обоснование:** Может привести к серьёзным проблемам с целостностью индекса и непредсказуемому поведению системы при конкурентной нагрузке.

*   **Название:** Неоптимальная стратегия чанкирования
    *   **Описание:** В `IndexingService._process_files_in_batches` используется простая стратегия чанкирования: разделение файлов на чанки фиксированного размера (15 строк) с фиксированным перекрытием (2 строки). Для кода и документации такой подход часто неэффективен. Код обычно лучше разделять по функциям, классам или блокам логики, а документацию — по секциям или абзацам. Простой по строкам метод может разбивать логически связанные фрагменты или оставлять слишком много бессмысленного контекста, что снижает качество эмбеддингов и, как следствие, релевантность поиска.
    *   **Расположение:** `src/services.py:149` (`_process_files_in_batches`)
    *   **Критичность:** **Medium**
    *   **Обоснование:** Снижает качество поиска и потенциально увеличивает размер индекса (если чанки слишком мелкие или неинформативные).

#### 2. Дефекты архитектуры и масштабируемости

*   **Название:** Хранение состояния задач в памяти (In-Memory Task Status)
    *   **Описание:** Статус фоновых задач индексации (`TASK_STATUS`) хранится в глобальном словаре в памяти (`Dict[str, Dict[str, Any]]`). Это делает систему неустойчивой к перезапускам (все статусы задач будут потеряны) и несовместимой с масштабированием API (при запуске нескольких инстансов FastAPI каждый инстанс будет иметь свой `TASK_STATUS`, и запросы на `/tasks/status/{task_id}` могут попадать не на тот инстанс, где была запущена задача).
    *   **Расположение:** `src/main.py:40` (`TASK_STATUS`)
    *   **Критичность:** **High**
    *   **Обоснование:** Блокирует горизонтальное масштабирование API и приводит к потере информации о состоянии задач при любом перезапуске сервиса, ухудшая UX и затрудняя отладку.

*   **Название:** Неоптимальное управление worker-ами Uvicorn
    *   **Описание:** В `manage.py` команда `UVICORN_CMD` всегда запускает Uvicorn с `--workers 1`. Хотя это может быть оправдано для моделей на GPU (чтобы каждый worker не пытался загрузить свою копию модели в ограниченную GPU-память), для CPU-ориентированных рабочих нагрузок (или если модель достаточно мала) это значительно ограничивает параллелизм обработки HTTP-запросов. Если ML-модель не использует GPU, или если основная нагрузка — это I/O-bound операции Qdrant, то использование всего одного worker-а может создать узкое место.
    *   **Расположение:** `manage.py:24` (`UVICORN_CMD`)
    *   **Критичность:** **Medium**
    *   **Обоснование:** Ограничивает производительность и отзывчивость API при высокой нагрузке на CPU или I/O, если модель не требует исключительного доступа к GPU.

#### 3. Дефекты безопасности

*   **Название:** Потенциальная уязвимость "Path Traversal" в `get_projects_base_dir`
    *   **Описание:** Функция `get_projects_base_dir` в `src/config.py` по умолчанию устанавливает базовую директорию для проектов как `Path(os.getenv("PROJECTS_BASE_DIR", ".."))`. Значение по умолчанию `..` (т.е. родительская директория относительно запуска `main.py`) в сочетании с тем, что `project_name` берётся из URL (`/projects/{project_name}/...`), может быть уязвимо к атакам Path Traversal. Злоумышленник может попытаться передать `project_name` типа `../../etc/passwd` или `../.git/hooks` для доступа к файлам вне разрешенной директории, если система не будет корректно разрешать и проверять пути. Хотя `Path.is_dir()` и `Path.is_relative_to()` используются, необходимо тщательно проверить все места, где `project_path` строится из `projects_base_dir` и `project_name`.
    *   **Расположение:** `src/config.py:13` (`get_projects_base_dir`), `src/main.py` (`project_path` usage in graph/search endpoints)
    *   **Критичность:** **High**
    *   **Обоснование:** Позволяет доступ к файлам вне предполагаемой директории проектов, что может привести к раскрытию чувствительной информации или удаленному выполнению кода.

#### 4. Дефекты обработки данных и логики ML

*   **Название:** Эвристическое бустирование релевантности поиска
    *   **Описание:** В `main.py` реализована эвристическая логика "бустирования" результатов поиска на основе ключевых слов в запросе (`is_doc_query`, `is_code_query`). Это бустирование применяется *после* получения результатов от Qdrant, модифицируя `score` только для *первого* хита в каждой группе файлов. Такой подход является пост-фильтрацией/ре-ранжированием, а не настоящим бустированием в процессе векторного поиска. Он может давать ложноположительные или неоптимальные результаты, так как реальная релевантность чанков не изменяется в векторной базе. Более эффективным было бы интегрировать бустирование в запрос к Qdrant, используя его возможности для весов или дополнительные фильтры.
    *   **Расположение:** `src/main.py:311` (логика бустирования в `search_project_endpoint`)
    *   **Критичность:** **Medium**
    *   **Обоснование:** Может приводить к неточному ранжированию результатов поиска, так как "бустинг" применяется не к исходной релевантности векторного поиска, а к уже полученным результатам, изменяя их поверхностно.

*   **Название:** Несоответствие описания модели
    *   **Описание:** В `src/config.py` комментарий к `MODEL_NAME` (paraphrase-multilingual-MiniLM-L12-v2) ошибочно утверждает, что модель "trained on a massive Russian corpus". Это мульти-языковая модель, но не специализированная на русском языке. Хотя это не является функциональной ошибкой, это может ввести в заблуждение разработчиков относительно ожидаемых характеристик производительности модели.
    *   **Расположение:** `src/config.py:27` (`MODEL_NAME`)
    *   **Критичность:** **Low**
    *   **Обоснование:** Вводит в заблуждение относительно характеристик и оптимального использования модели.

#### 5. Инфраструктурные и операционные дефекты

*   **Название:** Жёстко закодированный путь проекта в `index.py`
    *   **Описание:** Скрипт `src/index.py` имеет жёстко закодированный `PROJECT_ROOT_PATH = Path(__file__).parent.parent.parent / 'autowarp'`. Это делает скрипт негибким для индексации других проектов без изменения кода. Хотя `IndexingService` (используемый API) использует настраиваемый `projects_base_dir`, этот отдельный скрипт является примером или утилитой, которая не соответствует принципам конфигурируемости.
    *   **Расположение:** `src/index.py:10` (`PROJECT_ROOT_PATH`)
    *   **Критичность:** **Low**
    *   **Обоснование:** Ограничивает переиспользуемость скрипта, требуя модификации кода для изменения цели индексации.

*   **Название:** Использование `qdrant/qdrant:latest` в `docker-compose.yml`
    *   **Описание:** Использование тега `latest` для образа Qdrant в `docker-compose.yml` является плохой практикой для production-сред и даже для стабильной разработки. При каждом `docker-compose pull` или `up`, может быть скачана новая версия, потенциально содержащая несовместимые изменения или баги, что нарушит работу приложения.
    *   **Расположение:** `docker-compose.yml:5` (`image: qdrant/qdrant:latest`)
    *   **Критичность:** **Low**
    *   **Обоснование:** Может привести к неожиданным поломкам и нестабильности из-за неконтролируемых обновлений зависимостей. Рекомендуется фиксировать версию образа.

#### 6. Дефекты пользовательского интерфейса (Dashboard)

*   **Название:** Инвертирование масштабирования графа для тачпадов/мыши
    *   **Описание:** В `static/app.js` опция `zoomSpeed: -1` для `vis.Network` инвертирует направление масштабирования. Это может быть желаемым поведением для некоторых пользователей (например, на Mac-тачпадах), но не для всех (стандартное колесо мыши на Windows/Linux) и может быть неинтуитивным или нежелательным по умолчанию. Лучше предоставить пользователю возможность выбора или придерживаться стандартного поведения.
    *   **Расположение:** `static/app.js:154` (`interaction.zoomSpeed`)
    *   **Критичность:** **Low**
    *   **Обоснование:** Несоответствие ожидаемому поведению масштабирования может ухудшить удобство использования для некоторых пользователей.

### Заключение

Проект `isearch` обладает хорошей основой, но требует доработки в ключевых областях, особенно касающихся **целостности данных** в векторном хранилище и **масштабируемости** API. Недетерминированные ID для точек Qdrant (`GT-3`) и отсутствие блокировок при инкрементальной индексации (`GT-4`) являются наиболее критичными проблемами, которые могут привести к серьёзным сбоям и потере данных. Устранение этих проблем должно быть приоритетом.
Кроме того, необходимо обратить внимание на потенциальную уязвимость Path Traversal и улучшить обработку ошибок и хранение состояния для фоновых задач, чтобы система была более надёжной и масштабируемой.
