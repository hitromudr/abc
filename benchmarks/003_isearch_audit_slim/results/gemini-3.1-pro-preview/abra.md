# Концептуальный Протокол: Аудит Системы Поиска isearch

## 1. Топология задачи (Ontology)
- **Мета-тип:** Аудит конвейера обработки данных, векторного хранения и контрактов API.
- **Ключевой конфликт:** Математическая детерминированность векторного хранилища против недетерминированной среды исполнения Python. Изоляция файловой системы против неконтролируемого пользовательского ввода.

## 2. Инварианты (Invariants)
1. **Закон сохранения идентификаторов:** Идентификатор чанка (Point ID) обязан быть математически детерминированным и зависеть исключительно от контента и пути.
2. **Предел изоляции периметра:** Внешний пользовательский ввод (параметры URL) не имеет права напрямую конкатенироваться с путями файловой системы без проверки границ.
3. **Принцип единственного писателя:** На один домен данных (проект) может существовать только один активный процесс мутации состояния.

## 3. Точка опоры (Leverage Point)
- **Фокус внимания:** Генерация идентификаторов (`src/vector_store.py`) и маршрутизация ввода API (`src/main.py`).
- **Игнорируемый шум:** Оптимизация ML-моделей. Архитектура деградирует на уровне инфраструктуры и безопасности задолго до того, как модель станет узким местом.

## 4. Векторы энтропии (Degradation Paths)

### Архитектурные и логические уязвимости (L2, L3)
1. **Path Traversal / Arbitrary File Read (Критический):**
   - В `src/main.py` параметр `project_name` принимается напрямую из URL и без валидации конкатенируется с `projects_base_dir`.
   - *Вектор:* Злоумышленник может передать `../` (или URL-encoded эквивалент), заставив `IndexingService` проиндексировать системные директории (например, `/etc`). Последующий запрос к `/search` вернёт содержимое этих файлов через `extract_snippet`.
2. **Недетерминированность State (GT-3) (Высокий):**
   - В `src/vector_store.py` ID генерируется как `abs(hash(p.get('source_file') + p.get('text')))`.
   - Встроенная функция `hash()` использует случайный seed (`PYTHONHASHSEED`) при каждом запуске интерпретатора.
   - *Симптом:* Инкрементальное обновление невозможно. Каждый перезапуск API и вызов `/index` создаёт 100% дубликатов в Qdrant. Функция `abs()` дополнительно удваивает вероятность коллизий.
3. **Гонка данных (Race Condition) (GT-4) (Высокий):**
   - Роутер `/index` запускает фоновую задачу `run_indexing_task` без проверки наличия уже идущей индексации для этого проекта.
   - *Симптом:* Конкурентные запросы вызовут параллельное чтение ФС и запись в БД, что приведёт к повреждению коллекции и OOM.
4. **Утечка памяти (OOM) и потеря данных в `/clusters` (Средний):**
   - Используется `scroll` с `limit=10000`, но игнорируется возвращаемый `next_page_offset`.
   - *Симптом:* Для проектов с >10k чанками метод тихо отбросит данные. Отсутствие пагинации приведёт к загрузке гигантских массивов в RAM, убивая процесс.
5. **Деградация скорости поиска (Гомеостаз) (Низкий):**
   - В `vector_store.py` создан Payload-индекс по `source_file`, но поиск в `main.py` фильтрует по `file_type`. Qdrant вынужден выполнять ресурсоемкое последовательное сканирование.
6. **Мёртвый код (Энтропия) (Низкий):**
   - Скрипты `src/index.py` и `src/search.py` — это устаревшие дубликаты логики, содержащие хардкод путей (`../autowarp`). Кандидаты на удаление.

## 5. Алгоритм стабилизации (The Protocol)
- **Фаза 0: Изоляция.** Удаление `src/index.py` и `src/search.py`.
- **Фаза 1: Оцифровка базиса.** Замена `hash()` на стабильную криптографическую функцию: `int(hashlib.md5(text.encode()).hexdigest()[:15], 16)`. Это гарантирует стабильные ID, валидные для формата uint64 Qdrant.
- **Фаза 2: Применение рычага.**
  - Внедрение санитайзера путей в эндпоинты (проверка `pathlib.Path.resolve().is_relative_to(projects_base_dir)`).
  - Использование `next_page_offset` в цикле метода `/clusters`.
  - Добавление `client.create_payload_index(..., field_name="file_type")`.
- **Фаза 3: Интеграция и Стабилизация.** Введение In-Memory `Set` (реестра активных задач) для блокировки запуска дублирующих `BackgroundTasks` для одного и того же проекта.

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** Перезапуск API и повторный вызов `/index` не увеличивает количество точек в коллекции. Запросы с `../` в `project_name` возвращают 400 Bad Request. Кластеризация успешно работает на 50k+ чанках.
- **Ложный сигнал успеха:** Попытка закрыть Path Traversal примитивным `replace("../", "")`, который обходится кодировками.

## 7. Эвристики и Мета-Контекст (Heuristics & Meta-Context)
- **Наблюдение оператора:** Qdrant инициализирован с `datatype="float16"`, однако из-за использования HTTP API (`prefer_grpc=False`) метод `.tolist()` сериализует векторы как обычные JSON floats. Qdrant сам делает даункастинг на своей стороне, но сеть все равно перегружается float64. Рекомендуется переключить `prefer_grpc=True`.
- **Тактика:** При бустинге в `search_project_endpoint` стоит применять `boost_factor` ко всем хитам в группе, а не только к `hits[0]`.

## 8. Резолюция и Архитектурная Развилка (Approval Gate)
- **Фундаментальный диагноз:** Уязвимость произвольного чтения файлов (Path Traversal) и математическое разрушение БД из-за недетерминированных идентификаторов.
- **Стратегические варианты (The Forks):**
  - **Вариант А:** Точечный патч. Смена хеш-функции, строгая валидация путей `is_relative_to`, In-Memory локи, исправление пагинации.
  - **Вариант Б:** Изоляция. Вынос процесса индексации в фоновый микросервис (Celery), переход Qdrant на gRPC, использование UUIDv5 для идентификаторов.
- **Рекомендация ИИ:** Вариант А. Позволяет закрыть все уязвимости (GT-3, GT-4, Path Traversal) и стабилизировать ядро без смены архитектурной парадигмы.
- **Ближайшие физические шаги (Next Steps):**

  2. **[ЗАПРЕТ]** ИИ не имеет права генерировать `EXECUTION_STATE.md` или рабочий код до явного утверждения варианта оператором.
