# Аудит кодовой базы `isearch` (Baseline Report)

## 📊 Executive Summary
Система `isearch` представляет собой рабочий прототип гибридного поиска с графовым анализом, однако содержит **критические архитектурные и инфраструктурные изъяны**. Главные проблемы сконцентрированы в области детерминированности данных (векторные ID), потокобезопасности и валидации пользовательского ввода (Path Traversal).

## 🚨 Классификация дефектов

### [CRITICAL] 1. Инфраструктура: Недетерминированная генерация ID векторов (Data Corruption / Memory Leak)
* **Локация:** `src/vector_store.py` (метод `upsert`).
* **Описание:** Для генерации ID точек в Qdrant используется конструкция: 
  `ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]`
* **Суть проблемы:** В Python >= 3.3 встроенная функция `hash()` **рандомизируется при каждом запуске процесса** (из-за `PYTHONHASHSEED`). При перезапуске API сервера (или запуске фонового воркера в другом процессе) хэши для тех же самых строк будут другими. 
* **Последствия:** Инкрементальная индексация сломана. Вместо обновления существующих чанков, система будет плодить дубликаты. Использование `abs()` дополнительно удваивает шанс коллизий (уменьшая пространство хэшей).
* **Решение:** Использовать криптографически стабильный хэш, например, `hashlib.md5(text.encode()).hexdigest()`, и конвертировать его в UUID (Qdrant поддерживает UUID в качестве ID).

### [CRITICAL] 2. Безопасность: Directory/Path Traversal (CWE-22)
* **Локация:** `src/main.py` (`get_projects_base_dir`, эндпоинты `/projects/{project_name}/*`), `src/graph_builder.py`.
* **Описание:** Имя проекта (`project_name`) берется напрямую из URL и конкатенируется с базовой директорией: `projects_base_dir / project_name`. 
* **Суть проблемы:** Нет проверки на выход за пределы базовой директории. Злоумышленник может передать `project_name` вида `..%2F..%2Fetc`, и система проиндексирует или выдаст содержимое системных файлов. То же самое касается параметра `start_path` в POST-запросах для графа.
* **Решение:** Добавить строгую валидацию:
  ```python
  resolved_path = (projects_base_dir / project_name).resolve()
  if not resolved_path.is_relative_to(projects_base_dir.resolve()):
      raise HTTPException(400, "Invalid project name")
  ```

### [HIGH] 3. Инфраструктура: Состояние гонки (Race Condition) при фоновой индексации
* **Локация:** `src/main.py` (эндпоинт `/projects/{project_name}/index`).
* **Описание:** Фоновая задача `run_indexing_task` запускается через `BackgroundTasks` без каких-либо блокировок (Mutex/Lock) на уровне проекта.
* **Последствия:** Если вызвать эндпоинт индексации несколько раз подряд для одного проекта, несколько потоков начнут параллельно читать файлы, вычислять diff'ы, удалять и вставлять данные в Qdrant. Это приведет к Race Condition, повреждению данных в коллекции и потенциальному падению сервиса.
* **Решение:** Внедрить In-Memory лок (`asyncio.Lock` или словарь с активными блокировками по `project_name`) или распределенный лок (Redis), если планируется масштабирование.

### [HIGH] 4. Инфраструктура: Игнорирование частичных отказов батчей в Qdrant
* **Локация:** `src/vector_store.py` (метод `upsert`).
* **Описание:** Запись в БД разбивается на батчи по 500 элементов. Если `self.client.upsert()` падает на одном из батчей (например, из-за сетевого таймаута, OOM в Qdrant или ошибки сериализации), исключение прервет весь процесс индексации.
* **Последствия:** Индекс остается в неконсистентном состоянии (часть файлов обновлена, часть — нет). Механизма Retry для транзиентных ошибок нет.

### [MEDIUM] 5. Логика: Падение поиска из-за рассинхронизации исходников (IndexError)
* **Локация:** `src/main.py` (функция `extract_snippet`).
* **Описание:** При формировании сниппета функция читает файл с диска и обращается к строкам: `lines[i]`. `i` берется из метаданных (Payload) Qdrant.
* **Последствия:** Если файл на диске был изменен или урезан после индексации, но до поиска (инкрементальный индекс еще не отработал), произойдет `IndexError: list index out of range`, и весь эндпоинт `/search` отдаст `500 Internal Server Error`.
* **Решение:** Добавить `try/except IndexError` внутри цикла или жестко проверять границы массива строк перед доступом.

### [MEDIUM] 6. Архитектура: In-Memory стейт задач `TASK_STATUS` не масштабируется
* **Локация:** `src/main.py` (`TASK_STATUS: Dict[str, Dict[str, Any]] = {}`).
* **Описание:** Статусы фоновых задач хранятся в глобальном словаре внутри памяти Python-процесса.
* **Последствия:** Как только Uvicorn будет запущен с параметром `--workers > 1` (сейчас захардкожено 1 в `manage.py`, но это узкое место), клиенты при поллинге `/tasks/status/{task_id}` будут получать `404 Not Found`, если запрос попадет на другой воркер.

### [LOW] 7. Технический долг: Конфликтующие стратегии чанкирования в `index.py`
* **Локация:** `src/index.py` и `src/search.py`.
* **Описание:** Это устаревшие (legacy) CLI-скрипты, которые дублируют функционал `services.py`, но используют *абсолютно другую* логику: чанкирование по двойному переносу строки `\n\n` (вместо по 15 строк) и не используют ID на основе хэшей.
* **Последствия:** Если разработчик случайно запустит `python src/index.py`, он полностью повредит коллекцию Qdrant, сделав ее несовместимой с API-частью приложения. Скрипты нужно удалить.
