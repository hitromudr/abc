# Аудит кодовой базы проекта `isearch`

## 1. Резюме (Executive Summary)
Был проведен глубокий аудит архитектуры, логики обработки данных и инфраструктурных контрактов проекта `isearch`. Архитектура системы (FastAPI + Qdrant + SentenceTransformers) в целом спроектирована логично, однако выявлены **критические уязвимости** на стыке управления состоянием, безопасности (Path Traversal) и конкурентного доступа.

Наибольшую угрозу представляет механизм генерации ID векторов и отсутствие изоляции путей, что позволяет как сломать консистентность базы данных, так и осуществить произвольное чтение файлов хоста.

---

## 2. Классификация и разбор дефектов

### 🔴 КРИТИЧЕСКИЕ (CRITICAL)

#### C1. Path Traversal -> Arbitrary File Read (Утечка данных)
* **Локация:** `src/main.py` (эндпоинты `/projects/{project_name}/*`)
* **Описание:** Имя проекта (`project_name`) берется напрямую из URL и конкатенируется с `projects_base_dir` (`projects_base_dir / project_name`). Отсутствует проверка через `resolve().is_relative_to()`. 
* **Вектор атаки:** Злоумышленник может передать `project_name` в виде `../../etc`, инициировать индексацию системной директории, а затем через эндпоинт `/search` читать системные файлы по кускам.
* **Решение:** Внедрить строгую валидацию имени проекта (например, regex `^[a-zA-Z0-9_-]+$`) и проверку пути через `resolve().is_relative_to(projects_base_dir.resolve())`.

#### C2. Недетерминированная генерация ID векторов (Уничтожение консистентности БД)
* **Локация:** `src/vector_store.py` (метод `upsert`)
* **Описание:** Для генерации Point ID в Qdrant используется встроенная функция Python: `ids = [abs(hash(...)) for p in payloads]`.
* **Проблема:** В Python 3 `hash()` рандомизируется при каждом запуске интерпретатора (через `PYTHONHASHSEED`). При перезапуске API-сервера (например, через `make restart`) хеши тех же самых файлов изменятся.
* **Последствия:** Инкрементальная индексация сломается. Вместо перезаписи (upsert) существующих кусков текста, Qdrant будет бесконечно дублировать их в БД, что приведет к переполнению памяти и деградации качества поиска (будут выдаваться дубликаты).
* **Решение:** Использовать детерминированный алгоритм хеширования (например, SHA-256 или MD5), обрезая его до 64-bit unsigned integer, как того требует Qdrant.

---

### 🟠 ВЫСОКАЯ КРИТИЧНОСТЬ (HIGH)

#### H1. Состояние гонки (Race Condition) при фоновой индексации
* **Локация:** `src/main.py` (`index_project_endpoint`)
* **Описание:** При вызове `/index` запускается фоновая задача `run_indexing_task`. Ничто не мешает вызвать эндпоинт 10 раз за секунду для одного и того же проекта.
* **Последствия:** Множественные потоки начнут параллельно читать локальные файлы, делать вызовы `client.scroll` для сверки стейта и параллельно слать `delete/upsert` в Qdrant. Это вызовет конфликты версий (Optimistic Concurrency Control) в Qdrant и логическое повреждение индекса.
* **Решение:** Реализовать in-memory лок (mutex) или статус блокировки (например, в словаре `TASK_STATUS` или Redis) для каждого `project_name`.

#### H2. Отсутствие обработки частичных сбоев (Silent Batch Failures)
* **Локация:** `src/vector_store.py` (`upsert`), `src/services.py`
* **Описание:** При записи батчей в `vector_store.py` используется `client.upsert(..., wait=True)`. Если Qdrant отклонит батч (например, из-за таймаута или поврежденного Payload), выкинется исключение.
* **Последствия:** Процесс индексации в `_process_files_in_batches` аварийно прервется, статус задачи перейдет в "failed", но часть батчей уже будет закоммичена, а локальный стейт не синхронизирован. Откатить транзакцию невозможно.
* **Решение:** Реализовать try/except блок на уровне батчей с механизмом retry.

### 🟡 СРЕДНЯЯ КРИТИЧНОСТЬ (MEDIUM)

#### M1. Усечение данных при кластеризации (Ограничение пагинации)
* **Локация:** `src/main.py` (`perform_clustering`)
* **Описание:** Для получения всех векторов вызывается `vector_store.client.scroll(..., limit=10000)[0]`. 
* **Проблема:** Если в проекте больше 10 000 чанков, остальные будут молча проигнорированы. `scroll` возвращает `next_page_offset`, который здесь никак не используется.
* **Решение:** Написать `while` цикл с использованием `next_page_offset`, аналогично тому, как это сделано в `_get_indexed_state`.

#### M2. Пропуск относительных импортов в графе (False Orphans)
* **Локация:** `src/graph_builder.py` (`parse_python_file`)
* **Описание:** При парсинге AST используется условие: `if isinstance(node, ast.ImportFrom) and node.module:`. 
* **Проблема:** Для относительных импортов вида `from . import config` атрибут `node.module` равен `None`. Этот импорт полностью пропускается.
* **Последствия:** Граф зависимостей получается неполным, что напрямую ведет к ложному определению файлов как "orphaned" (сироты).
* **Решение:** Изменить логику для обработки `node.level > 0` даже при отсутствии `node.module`.

#### M3. Наивный чанкинг кода (OOM & Model Truncation Risk)
* **Локация:** `src/services.py` (`_process_files_in_batches`)
* **Описание:** Чанкинг файлов производится по количеству строк (`chunk_size_lines = 15`).
* **Проблема:** Минифицированные файлы (JS/CSS) или файлы с длинными base64-строками могут содержать тысячи символов в одной строке. Модель MiniLM имеет ограничение в 512 токенов. Слишком длинная строка приведет либо к молчаливой обрезке контекста внутри SentenceTransformers, либо к OOM на GPU.
* **Решение:** Использовать токенизатор (например, из `transformers`) или хотя бы жестко ограничить максимальную длину чанка в символах.

#### M4. Игнорирование якорных ссылок в Markdown
* **Локация:** `src/graph_builder.py`
* **Описание:** `target_path = (file_path.parent / link_target).resolve()`. Если `link_target` имеет вид `guide.md#section-1`, `target_path.exists()` вернет `False`, и ребро не будет создано.

### 🟢 НИЗКАЯ КРИТИЧНОСТЬ И КОДСМЕЛЛЫ (LOW)

1. **Необработанное исключение в CLI (`manage.py`)**: При вызове API, если uvicorn завис, таймаут на 15 секунд в `_api_request` может отвалиться с `ReadTimeout`. `RequestException` ловится, но `JSONDecodeError` внутри 4xx/5xx обработчика упадет, если API ответит сырым HTML (например, от Nginx).
2. **`cudnn.benchmark = True` для динамических батчей**: В `src/embedder.py` включен `cudnn.benchmark`. Это полезно при фиксированном размере батча. Так как размер батча (особенно последнего) плавает, GPU будет каждый раз тратить время на поиск оптимального алгоритма свертки, что замедлит инференс.
3. **Хранение секретов**: Использование `.env` для порта/QDRANT_URL — хорошо, но нет механизма управления API ключами (CORS открыт `allow_origins=["*"]`). Если сервис выставлен наружу, любой может удалить индекс (эндпоинт `DELETE`).

## 3. Итог

Для стабилизации системы необходимо **в первую очередь** исправить механизм хеширования `ids` (заменить `hash()` на MD5/SHA-256) и закрыть уязвимость `Path Traversal` в путях API. Без этих правок система опасна для деплоя даже в защищенном контуре.
