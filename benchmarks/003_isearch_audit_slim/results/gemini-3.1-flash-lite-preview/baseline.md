## Отчёт об аудите кодовой базы проекта `isearch`

### 1. Резюме
Проект представляет собой API для семантического поиска и анализа графа зависимостей кода. Основные архитектурные риски связаны с отсутствием механизмов консистентности данных (race conditions), неоптимальной схемой идентификации объектов в векторной БД и потенциальными узкими местами в логике индексации.

---

### 2. Классификация дефектов

| ID | Категория | Описание | Критичность |
| :--- | :--- | :--- | :--- |
| **D-01** | **Инфраструктура** | Детерминизм ID в `vector_store.py`: `abs(hash(...))` нестабилен между процессами/запусками (PYTHONHASHSEED) и может приводить к коллизиям. | **Critical** |
| **D-02** | **Архитектура** | Отсутствие изоляции (locks) при записи в Qdrant. Параллельная индексация одного проекта приведет к повреждению данных (race condition). | **High** |
| **D-03** | **Безопасность** | `Path Traversal` в `src/main.py`: передача аргументов напрямую в `Path()` без валидации `resolve()` относительно `projects_base_dir`. | **High** |
| **D-04** | **Логика** | Использование `KMeans` на средних значениях эмбеддингов файлов при попытке кластеризации — семантическая потеря информации. | **Medium** |
| **D-05** | **Производительность** | `scroll` по всей коллекции без использования `filter` в `_get_indexed_state` при больших объемах данных создаст огромную нагрузку на RAM. | **Medium** |

---

### 3. Детальный разбор критических уязвимостей

#### D-01: Недетерминированные ID (Critical)
В `vector_store.py`:
```python
ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]
```
- **Проблема**: Функция `hash()` в Python рандомизируется при каждом запуске интерпретатора (security feature). Это означает, что при перезапуске API или воркера indexing-задачи ID одних и тех же чанков изменятся, что вызовет "дубли" в базе при инкрементальной индексации.
- **Рекомендация**: Использовать `hashlib.sha256` и конвертировать первые 8-16 байт в целочисленный тип.

#### D-02: Race Condition (High)
Сервис `IndexingService` не проверяет состояние `RUNNING` для конкретного проекта.
- **Проблема**: Если пользователь отправит два запроса на `/index` для одного проекта, оба процесса начнут `recreate_collection` или `upsert` одновременно. Это приведет к непредсказуемому состоянию индекса.
- **Рекомендация**: Добавить `FileLock` или Redis-блокировку на имя проекта на время выполнения метода `run_indexing`.

#### D-03: Path Traversal (High)
В `main.py` и `manage.py` пути к проектам формируются динамически.
- **Проблема**: Если `project_name` содержит `../`, можно выйти за пределы разрешенной директории.
- **Рекомендация**: Использовать `Path(base).joinpath(name).resolve()` и проверять, что результат начинается с `base` (см. `is_relative_to`).

### 4. Инфраструктурные рекомендации
1. **Qdrant**: Оценить необходимость `grpc` (сейчас `prefer_grpc=False` из-за таймаутов).
2. **Типизация**: Добавить `mypy` в CI, так как использование `Any` и слабые контракты API маскируют потенциальные ошибки рантайма (например, в кластеризации).
3. **Мониторинг**: Внедрить `healthcheck` для Qdrant, который реально проверяет доступность коллекции, а не просто наличие соединения.
