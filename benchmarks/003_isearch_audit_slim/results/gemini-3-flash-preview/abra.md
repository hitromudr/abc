Проект `isearch` представляет собой систему гибридного поиска (семантика + графы). 
Ключевые компоненты: FastAPI, Qdrant, SentenceTransformers, Pytest, Click CLI.

**Анализ по Октагону и Фазам:**

1.  **Фаза 0 (Эмпирика):** 
    *   `src/vector_store.py`: Используется `abs(hash(...))` для генерации ID. В Python `hash()` рандомизирован (PYTHONHASHSEED) между перезапусками процесса. Это делает "incremental indexing" невозможным после перезапуска сервера — ID изменятся, и система начнет двоить данные. Это критический баг инфраструктуры.
    *   `src/main.py`: `TASK_STATUS` — глобальный dict в памяти. При перезапуске все данные о задачах теряются.
    *   `src/main.py`: `perform_clustering` выгружает 10,000 векторов в память (`all_points`). При размерности 384 это ~15МБ, но при росте базы или размерности модели (E5-large=1024) и использовании обычного `np.array` это приведет к OOM.

2.  **Фаза 2 (Инварианты):**
    *   Консистентность ID: Нарушен закон идемпотентности. Одинаковый контент должен давать одинаковый ID всегда, а не только внутри сессии.
    *   Изоляция: `project_name` в эндпоинтах используется для формирования пути: `projects_base_dir / project_name`. Без валидации через `.resolve()` и проверки `is_relative_to` возможна атака **Path Traversal** (чтение/индексация файлов вне базовой директории).

3.  **Фаза 3 (Leverage):**
    *   Точка рычага — механизм генерации ID и управление конкурентностью. Сейчас нет блокировок на индексацию одного и того же проекта. Два параллельных запроса на `/index` вызовут Race Condition в Qdrant.

4.  **Red Teaming (Октагон):**
    *   *Наследственность (L2):* Состояние задач (`TASK_STATUS`) и детерминизм ID (hash) провалены.
    *   *Иммунитет (L3):* Path Traversal через API.
    *   *Метаболизм (L2):* OOM в кластеризации.

**Формирование протокола:**
Класс задачи: Система индексации и поиска векторов. 
Главный конфликт: Скорость (инкрементальность) vs Точность (детерминизм).

# Концептуальный Протокол: Аудит Системы `isearch`

## 1. Топология задачи (Ontology)
- **Мета-тип:** Инкрементальная синхронизация физического состояния (файлы) с векторным представлением (Qdrant).
- **Ключевой конфликт:** Детерминизм ID против коллизий при использовании быстрых хеш-функций в распределенной среде.

## 2. Инварианты (Invariants)
1.  **Закон Идемпотентности:** `ID(content, salt) == ID(content, salt)` всегда. Текущая реализация через `hash()` нарушает этот закон.
2.  **Закон Изоляции (Sandbox):** Операции над проектом `X` не должны иметь доступа к файловой системе выше `PROJECTS_BASE_DIR`.
3.  **Закон Лимитов (OOM Protection):** Загрузка векторов в RAM должна быть ограничена (streaming/batching).

## 3. Резолюция: Найденные Дефекты

### [CRITICAL] Инфраструктурный Хаос: Недетерминированные ID
- **Локация:** `src/vector_store.py`, метод `upsert`.
- **Суть:** Используется встроенный `hash()`. В Python по умолчанию включена рандомизация хешей (`PYTHONHASHSEED`). После перезапуска API-сервера те же самые куски текста получат другие ID.
- **Следствие:** "Инкрементальное" индексирование превращается в бесконечное дублирование данных в Qdrant. База будет расти экспоненциально при каждом перезапуске и переиндексации.
- **Root Cause:** Использование сессионного хеша для персистентного хранилища.

### [HIGH] Security: Path Traversal в API
- **Локация:** `src/main.py`, эндпоинты `/projects/{project_name}/index` и `/projects/{project_name}/search`.
- **Суть:** Параметр `project_name` напрямую конкатенируется с путем: `projects_base_dir / project_name`.
- **Следствие:** Злоумышленник может передать `project_name="../../../../etc"` и проиндексировать конфиги системы или пароли, получив к ним доступ через поиск.
- **Root Cause:** Отсутствие санитизации путей и проверки `is_relative_to`.

### [HIGH] Ресурсная смерть (OOM) при кластеризации
- **Локация:** `src/main.py`, метод `perform_clustering`.
- **Суть:** Вызов `vector_store.client.scroll` с лимитом 10,000 и немедленная конвертация всего объема в `np.array`.
- **Следствие:** На боевых данных или моделях с большой размерностью (1024+) процесс будет убит OOM-киллером.
- **Root Cause:** Отсутствие пакетной обработки (batching) при аналитике больших весов.

### [MEDIUM] Состояние-фантом (TASK_STATUS)
- **Локация:** `src/main.py`, `TASK_STATUS = {}`.
- **Суть:** Статусы длительных задач индексации хранятся в глобальной переменной в памяти.
- **Следствие:** При перезагрузке API (или worker-а) клиент никогда не узнает результат задачи. Task ID становится "мертвым".
- **Root Cause:** Отсутствие персистентного слоя для метаданных задач (Redis/SQLite/File).

### [LOW] Коллизии `abs(hash())`
- **Суть:** Использование `abs()` на результате `hash()` увеличивает вероятность коллизий (два разных хеша `N` и `-N` схлопываются в один ID). 
- **Следствие:** Редкая потеря чанков данных при индексации.

## 4. Архитектурная Развилка (Approval Gate)

### Вариант А: "Стабилизация текущей архитектуры" (Рекомендуется)
1.  **Fix ID:** Заменить `hash()` на `hashlib.blake2b(chunk.encode()).hexdigest()` (или первые 8-16 байт для int64).
2.  **Safe Path:** Внедрить проверку: `path.resolve().is_relative_to(base_dir)`.
3.  **Locking:** Добавить `asyncio.Lock` по `project_name` в `main.py`, чтобы предотвратить гонки при индексации.
4.  **Task Persistence:** Сбрасывать `TASK_STATUS` в JSON-файл в `.run/tasks.json` при каждом обновлении.

### Вариант Б: "Переход на Production-ready"
1.  **Celery/Redis:** Вынос задач индексации в Celery для управления очередями и состоянием.
2.  **Database:** Использование SQLite для хранения маппинга файлов и метаданных задач.

**Рекомендация ИИ:** Выбрать **Вариант А** как наиболее быстрый и закрывающий 100% критических уязвимостей без усложнения стека.
