# Отчёт по результатам глубокого аудита кодовой базы `isearch`

## 1. Резюме (Executive Summary)
Проведен аудит поисковой системы `isearch`. Система демонстрирует хорошую функциональную полноту (индексация, семантический поиск, анализ графов), однако содержит критические дефекты в механизме обеспечения целостности данных, архитектурные ограничения при масштабировании и потенциальные уязвимости безопасности.

**Основные находки:**
*   **Критическая потеря данных:** Использование встроенной функции `hash()` для генерации ID в Qdrant делает индекс несовместимым между перезапусками процесса.
*   **Архитектурная блокировка:** Выполнение тяжелых задач индексации (ML) в `BackgroundTasks` FastAPI без изоляции процессов.
*   **Риск повреждения индекса:** Отсутствие блокировок на уровне проектов при параллельном доступе к индексации.

## 2. Классификация найденных дефектов

### 2.1. Логические ошибки и целостность данных

#### [CRITICAL] Недетерминированные ID векторов (Python Hash Randomization)
*   **Файл:** `src/vector_store.py`, метод `upsert`.
*   **Суть:** Код использует `abs(hash(p.get('source_file', '') + p.get('text', '')))` для генерации ID точки в Qdrant. В Python 3 по умолчанию включена рандомизация хэшей (`PYTHONHASHSEED`). 
*   **Последствие:** После перезапуска API-сервера хэши для тех же самых строк будут другими. При попытке "инкрементальной" индексации система не найдет старые ID, не обновит их, а создаст дубликаты. Индекс будет бесконечно расти, а поиск будет выдавать по 10 одинаковых результатов. 
*   **Дополнительно:** Использование `abs()` увеличивает вероятность коллизий в два раза (отрицательные и положительные хэши схлопываются).

#### [HIGH] Ненадежный сбор состояния индекса (`_get_indexed_state`)
*   **Файл:** `src/services.py`.
*   **Суть:** Для получения списка уже проиндексированных файлов используется `scroll` всей коллекции. 
*   **Последствие:** На больших проектах (десятки тысяч файлов/чанков) это вызовет огромный оверхед по памяти и сети при каждой проверке "нужно ли обновляться". Это сводит на нет преимущества инкрементальной индексации.

---

### 2.2. Архитектура и Конкурентность

#### [HIGH] Отсутствие механизмов синхронизации (Race Conditions)
*   **Файл:** `src/main.py` / `src/services.py`.
*   **Суть:** Нет никаких блокировок (locks/semaphores) на запуск индексации конкретного проекта. 
*   **Последствие:** Если два пользователя (или один случайно дважды) нажмут "Index", запустятся два параллельных процесса `run_indexing_task`. Они будут одновременно читать `_get_indexed_state`, одновременно вычислять диффы и одновременно слать `upsert` в Qdrant. Это приведет к непредсказуемому состоянию индекса.

#### [MEDIUM] Злоупотребление FastAPI BackgroundTasks для ML
*   **Файл:** `src/main.py`.
*   **Суть:** Индексация (тяжелая модель SentenceTransformer) запускается как `BackgroundTasks`. 
*   **Последствие:** В Python `BackgroundTasks` выполняются в том же процессе. Тяжелые вычисления на GPU/CPU будут блокировать event loop или приводить к OOM (Out Of Memory) всего API-сервера. Для таких задач обязателен вынос в Celery/RQ или отдельный микросервис.

#### [LOW] Хранение состояния задач в RAM
*   **Файл:** `src/main.py`, переменная `TASK_STATUS`.
*   **Суть:** Статусы задач хранятся в обычном Python-словаре.
*   **Последствие:** При перезагрузке сервера или падении процесса вся история задач и их прогресс теряются. Клиент, опрашивающий `/tasks/status/`, получит 404.

### 2.3. Инфраструктура и API

#### [HIGH] Лимиты Qdrant в запросах на удаление
*   **Файл:** `src/vector_store.py`, метод `delete_points_by_source`.
*   **Суть:** Используется `models.Filter(should=[FieldCondition(...) for file in source_files])`. 
*   **Последствие:** Если в проекте изменилось/удалилось 5000 файлов, будет сформирован гигантский JSON-запрос. У Qdrant и HTTP-серверов есть лимиты на размер запроса и количество условий в фильтре. Запрос просто упадет с ошибкой 413 или 400.

#### [MEDIUM] Ненадежная демонизация в `manage.py`
*   **Файл:** `manage.py`.
*   **Суть:** Использование `nohup ... & echo $!`. 
*   **Последствие:** Если uvicorn упадет сразу после старта (например, порт занят), `echo $!` всё равно вернет PID оболочки, который запишется в `api.pid` как валидный, порождая "зомби-статусы" в CLI.

### 2.4. Безопасность

#### [HIGH] Потенциальный Path Traversal
*   **Файл:** `src/config.py` (`PROJECTS_BASE_DIR = ".."`) и `src/main.py`.
*   **Суть:** Использование `project_name` из URL напрямую в `Path(projects_base_dir / project_name)`.
*   **Последствие:** Хотя в некоторых местах есть `resolve()`, в методе `list_projects` или `index_project_endpoint` злоумышленник может передать `project_name` типа `../../etc`. Если права процесса позволяют, система начнет сканировать и индексировать системные файлы. Обязательна валидация, что итоговый путь находится *внутри* базовой директории.

---

### 2.5. Качество поиска (Product Logic)

#### [LOW] Ошибочное ранжирование при бустинге
*   **Файл:** `src/main.py`, логика `boost_factor`.
*   **Суть:** Код увеличивает `score` только у `top_hit` (первого элемента) в сгруппированном результате.
*   **Последствие:** Если в результатах было 10 файлов, и файл №5 — это документация, которая должна была "всплыть" при `is_doc_query`, она может не подняться в топ списка `grouped_list`, потому что сортировка по `score` происходит *после* того, как были взяты только первые хиты из групп.

## 4. Рекомендации по исправлению
1.  **Генерация ID:** Заменить `hash()` на детерминированный алгоритм, например `hashlib.md5(path + text).hexdigest()` (Qdrant поддерживает строковые UUID).
2.  **Concurrency:** Использовать файловые блокировки (через `fcntl` или `portalocker`) в `IndexingService` для предотвращения одновременной индексации одного проекта.
3.  **Безопасность:** Добавить проверку `path.resolve().is_relative_to(base_dir)`.
4.  **Масштабируемость:** Реализовать пакетное удаление (batching) в `delete_points_by_source`.
5.  **Архитектура:** Рассмотреть использование Celery для выноса индексации в отдельные worker-процессы.
