# Отчёт по результатам глубокого аудита проекта `isearch`

**Дата:** 2024-05-22  
**Роль:** Senior Software Engineer  
**Объект аудита:** Поисковая система `isearch` (slim abra)

---

## 1. Резюме (Executive Summary)
В ходе аудита кодовой базы проекта `isearch` выявлены критические дефекты в механизмах обработки данных и безопасности. Ключевые проблемы связаны с недетерминированной генерацией идентификаторов в векторной базе, уязвимостями к обходу директорий (Path Traversal) и отсутствием устойчивости состояния (Persistence) для фоновых задач.

---

## 2. Классификация дефектов

### 2.1. Критические уязвимости (Critical)

#### [C-01] Недетерминированные ID и нарушение идемпотентности индексации
**Файл:** `src/vector_store.py`, метод `upsert`
**Описание:** Идентификаторы точек в Qdrant генерируются с использованием встроенной функции Python `hash()`:
```python
ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]
```
**Проблема:** В Python 3 функция `hash()` использует рандомизацию (SipHash) по умолчанию. Это означает, что при перезапуске процесса или работе в нескольких воркерах одни и те же данные получат разные ID.
**Последствия:** 
1. При попытке переиндексации (incremental indexing) вместо обновления существующих записей будут создаваться дубликаты.
2. Использование `abs()` увеличивает вероятность коллизий хэша в два раза.
3. Поиск будет возвращать один и тот же контент многократно, засоряя выдачу.

#### [C-02] Уязвимость к обходу директорий (Path Traversal)
**Файл:** `src/config.py`, `src/main.py`
**Описание:** Параметр `project_name` в API-запросах (например, `/projects/{project_name}/search`) используется для формирования пути к файлам через `projects_base_dir / project_name`.
**Проблема:** Отсутствует валидация `project_name` на наличие последовательностей `../`.
**Последствия:** Злоумышленник может передать `project_name=../../etc` и через механизм `extract_snippet` прочитать произвольные файлы в системе, к которым имеет доступ пользователь, запустивший API.

### 2.2. Логические и архитектурные дефекты (High)

#### [H-01] Потеря состояния задач (In-Memory Task Storage)
**Файл:** `src/main.py`
**Описание:** Словарь `TASK_STATUS` является глобальной переменной в памяти.
**Проблема:** Любой перезапуск сервера (или сбой) приводит к полной потере информации о текущих и завершенных задачах индексации.
**Последствия:** Клиентские приложения (включая CLI `manage.py progress`) не смогут отследить статус задач после рестарта. В распределенной среде (если воркеров будет > 1) задачи вообще "рассинхронятся".

#### [H-02] Конфликт логики батчинга при эмбеддинге
**Файл:** `src/services.py`, метод `_process_files_in_batches`
**Описание:** В коде вызывается `self.embedder.encode(chunks_for_batch)`. Однако внутри `src/embedder.py` по умолчанию стоит `batch_size=8`, а `IndexingService` пытается пропихивать батчи по 32 файла, которые превращаются в сотни чанков.
**Проблема:** В `src/index.py` (который кажется устаревшим, но содержит важную логику) вызывается `embedder.encode(..., batch_size=len(batch_docs))`.
**Последствия:** Непредсказуемое потребление VRAM. Если чанков в батче слишком много, это приведет к CUDA Out of Memory (OOM), особенно на больших текстовых файлах.

#### [H-03] Ошибка в фильтрации Scope (Mismatched Keys)
**Файл:** `src/services.py` vs `src/main.py`
**Описание:** В `IndexingService` задаются `code_extensions` и `docs_extensions`. Для документов тип устанавливается как `"docs"`.
В `main.py` при поиске используется фильтр:
```python
query_filter = Filter(must=[FieldCondition(key="file_type", match=MatchValue(value=scope))])
```
где `scope` может быть `"docs"` или `"code"`.
**Проблема:** Если в Qdrant поле `file_type` не проиндексировано как `Keyword`, или если константы разойдутся (например, `doc` vs `docs`), фильтрация сломается. В `src/services.py` используется `"docs"`, но в ряде тестов и документации встречается `"doc"`.

### 2.3. Инфраструктурные и системные дефекты (Medium)

#### [M-01] Блокирующий запуск (FastAPI Lifespan)
**Файл:** `src/main.py`
**Описание:** Инициализация `EmbeddingModel` и `VectorStore` происходит в блоке `lifespan`.
**Проблема:** Если база Qdrant недоступна или модель не может загрузиться (нет интернета / памяти), всё приложение падает на этапе старта.
**Рекомендация:** Реализовать ленивую инициализацию или механизм Retry, чтобы API могло отдавать 503 ошибку, а не просто "умирать".

#### [M-02] Риск порчи коллекции при несовпадении размерности
**Файл:** `src/services.py`, метод `run_indexing`
**Описание:** Механизм пересоздает коллекцию, если размерность вектора модели не совпадает с существующей. 
**Проблема:** Это деструктивное действие. Если пользователь случайно поменяет модель в `config.py`, вся существующая база будет удалена без предупреждения.

## 3. Оценка критичности

| ID | Название | Критичность | Слой |
|:---|:---|:---|:---|
| C-01 | Недетерминированные ID (hash) | **Critical** | Data / DB |
| C-02 | Path Traversal в project_name | **Critical** | Security |
| H-01 | In-memory Task State | **High** | Architecture |
| H-02 | Конфликт батчинга (OOM risk) | **High** | Performance |
| H-03 | Scope Filter Mismatch | **High** | Logic |
| M-01 | Blocking Lifespan Init | **Medium** | Infrastructure |
| M-02 | Auto-recreate on Dim Mismatch | **Medium** | Data |

---

## 4. Рекомендации по исправлению

1.  **Заменить `hash()`** на `hashlib.sha256()` для генерации ID чанков. Использовать UUID v5 на основе хэша для Qdrant.
2.  **Валидировать `project_name`**: разрешить только алфавитно-цифровые символы, дефис и подчеркивание.
3.  **Вынести `TASK_STATUS`** в SQLite или Redis, если требуется сохранение состояния между перезапусками.
4.  **Унифицировать константы** типов файлов (`code` / `docs`) в одном конфигурационном файле.
5.  **Добавить проверку типов** в `src/vector_store.py`, чтобы убедиться, что `numpy.float16` корректно передается в Qdrant (текущий `.tolist()` может приводить к потере точности или неожиданным приведениям типов, если не контролировать входной массив).
