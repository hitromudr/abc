# Концептуальный Протокол: Аудит семантической поисковой системы

## 1. Топология задачи (Ontology)
- **Мета-тип:** Управление состоянием распределённой системы с ML-компонентами
- **Ключевой конфликт:** Между интеллектуальностью (семантический поиск) и надёжностью (консистентность, безопасность, производительность)

## 2. Инварианты (Invariants)
1. **Закон сохранения консистентности:** Данные в Qdrant должны отражать текущее состояние файловой системы проекта
2. **Предел детерминизма:** Хэширование файлов должно быть детерминистичным и коллизионно-стойким
3. **Принцип изоляции:** Пользовательские запросы не должны влиять на целостность системы

## 3. Точка опоры (Leverage Point)
- **Фокус внимания:** Механизм инкрементальной индексации и синхронизации с Qdrant
- **Игнорируемый шум:** Микрооптимизации кода vs. архитектурные риски race conditions и data corruption

## 4. Векторы энтропии (Degradation Paths)
- **Анти-паттерн А:** Молчаливое игнорирование ошибок Qdrant (batch failures, connection drops)
- **Когнитивное искажение Б:** Предположение, что "работает на тестовых данных" = "работает в продакшене"

## 5. Алгоритм стабилизации (The Protocol)
- **Фаза 0: Изоляция.** Запустить аудит в изолированной среде с моком Qdrant
- **Фаза 1: Оцифровка базиса.** Собрать метрики: время индексации, потребление памяти, частота ошибок
- **Фаза 2: Применение рычага.** Проанализировать механизм инкрементальной индексации на предмет race conditions
- **Фаза 3: Интеграция и Стабилизация.** Предложить архитектурные патчи для критических уязвимостей

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** Система возвращает релевантные результаты поиска при 100% uptime и нулевой потере данных
- **Ложный сигнал успеха:** "Тесты проходят" при наличии скрытых race conditions

## 7. Эвристики и Мета-Контекст
- **Открытые вопросы:** Как система ведёт себя при одновременной индексации одного проекта из нескольких процессов?
- **Тактические предложения:** Добавить health checks для Qdrant, мониторинг расхождения индекса и файловой системы
- **Комментарии оператора:** Проект зрелый, но есть "тихие" архитектурные риски

## 8. Резолюция и Архитектурная Развилка (Approval Gate)
- **Фундаментальный диагноз:** Система страдает от потенциальных race conditions при индексации, недетерминистичных хэшей и недостаточной security hardening.

- **Стратегические варианты (The Forks):**
  - **Вариант А: Минимальные исправления.** Починить критические security issues (path traversal) и детерминизм хэшей. Плюсы: низкий риск, быстро. Минусы: не решает архитектурные race conditions.
  - **Вариант Б: Архитектурный рефакторинг.** Ввести distributed lock для индексации, переделать механизм хэширования, добавить idempotency keys. Плюсы: фундаментально надёжнее. Минусы: высокая стоимость, риск регрессий.

- **Рекомендация ИИ:** Начать с Варианта А (критические fixes), затем постепенно внедрять элементы Варианта Б через feature flags.

- **Ближайшие физические шаги (Next Steps):**
  1. **[APPROVAL GATE]** Ожидание решения оператора (Выбор Варианта А или Б).
  2. **[ЗАПРЕТ]** ИИ не имеет права генерировать итоговый рабочий код до явного подтверждения выбора оператором.

# Отчёт об аудите проекта `isearch`

## Критические уязвимости (Severity: Critical)

### 1. Path Traversal в API графа
**Файл:** `src/main.py`, эндпоинт `/projects/{project_name}/graph`
**Описание:** Параметр `start_path` передаётся напрямую в `build_graph()` без достаточной валидации. Злоумышленник может указать `start_path="../../etc"` и получить доступ к файлам вне проекта.
**Код:**
```python
# main.py: build_project_graph
project_path = (projects_base_dir / project_name).resolve()
# Нет проверки, что start_path внутри project_path
return build_graph(str(project_path), request.start_path, request.depth)
```
**Исправление:** Добавить проверку `os.path.commonpath([real_start, project_path]) == project_path`.

### 2. Недостаточная валидация имени проекта
**Файл:** `src/services.py`, функция `project_name_to_collection`
**Описание:** Имя проекта преобразуется в имя коллекции Qdrant, но остаются символы `_` и `-`. Qdrant имеет ограничения на имена коллекций. Может привести к инъекции.
**Исправление:** Строгая whitelist-валидация: только буквы, цифры, дефис.

### 3. Race condition при параллельной индексации
**Файл:** `src/services.py`, класс `IndexingService`
**Описание:** Если два процесса одновременно запустят индексацию одного проекта, будет гонка данных в Qdrant. `_get_indexed_state()` читает состояние, затем `delete_points_by_source()` удаляет, потом `upsert()` записывает новое. Между чтением и записью другой процесс может изменить состояние.
**Корень:** Нет механизма блокировок (distributed lock) на уровне проекта.
**Исправление:** Ввести advisory lock через файл или Redis.

## Высокие риски (Severity: High)

### 4. Недетерминистичные ID точек в Qdrant
**Файл:** `src/vector_store.py`, метод `upsert`
**Описание:** ID генерируется как `abs(hash(source_file + text))`. `hash()` в Python недетерминистичен между запусками интерпретатора (PYTHONHASHSEED). При перезапуске сервиса те же данные получат другие ID → дублирование в Qdrant.
**Код:**
```python
ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]
```
**Исправление:** Использовать `hashlib.sha256(content).hexdigest()` или UUID5.

### 5. Молчаливое игнорирование ошибок Qdrant
**Файл:** `src/vector_store.py`, метод `upsert`
**Описание:** Qdrant может отвергнуть часть батча (невалидный ID, размер вектора), но исключение не логируется и не пробрасывается. Потеря данных без уведомления.
**Исправление:** Проверять ответ Qdrant на ошибки, логировать warning/error.

### 6. Отсутствие timeout на операции Qdrant
**Файл:** `src/vector_store.py`, `__init__`
**Описание:** Таймаут установлен только на HTTP-клиент (60 сек), но нет timeout на отдельные операции (scroll, search). При зависании Qdrant весь сервис блокируется.
**Исправление:** Выставить `timeout` для каждого вызова Qdrant.

## Средние риски (Severity: Medium)

### 7. Утечка памяти при batch processing
**Файл:** `src/services.py`, `_process_files_in_batches`
**Описание:** Все эмбеддинги для батча собираются в список `all_embeddings`, затем конкатенируются `np.vstack()`. Для очень больших проектов это может привести к OOM.
**Исправление:** Инкрементально записывать в Qdrant или использовать генераторы.

### 8. Нет graceful degradation при недоступности модели
**Файл:** `src/main.py`, lifespan
**Описание:** Если модель Hugging Face не загрузится при старте, приложение упадёт. Нет fallback на более простую модель или режим "только keyword search".
**Исправление:** Добавить health check и режим degraded.

### 9. Отсутствие аутентификации/авторизации API
**Файл:** `src/main.py`, все эндпоинты
**Описание:** API полностью открыт. В продакшене это позволит любому индексировать/удалять проекты.
**Исправление:** Добавить API key или OAuth.

## Низкие риски (Severity: Low)

### 10. Неоптимальное chunking файлов
**Файл:** `src/services.py`, `_process_files_in_batches`
**Описание:** Чанкинг по фиксированному количеству строк (15) без учёта семантических границ (функции, классы).
**Исправление:** Для кода использовать AST-based chunking.

### 11. Жёсткая привязка к расширениям файлов
**Файл:** `src/services.py`, `__init__`
**Описание:** Списки `code_extensions` и `docs_extensions` захардкожены. Не поддерживаются новые языки (например, `.cpp`, `.java`).
**Исправление:** Сделать конфигурируемым через настройки.

### 12. Отсутствие метрик качества поиска
**Описание:** Нет way to measure recall/precision поиска. Нельзя обнаружить деградацию при смене модели.
**Исправление:** Добавить логгирование relevance feedback и вычисление метрик.

### 13. CLI не проверяет доступность API
**Файл:** `manage.py`, функция `_api_request`
**Описание:** CLI падает с ConnectionError, но не предлагает команду для запуска сервиса.
**Исправление:** Добавить friendly error message с подсказкой `make up`.

### 14. Тесты не покрывают edge cases
**Файл:** `tests/`
**Описание:** Нет тестов на race conditions, обработку битых файлов, Unicode-имена.
**Исправление:** Добавить интеграционные тесты с моком Qdrant.

### 15. Docker-compose без персистентности
**Файл:** `docker-compose.yml`
**Описание:** Том `./qdrant_storage:/qdrant/storage` может быть на tmpfs. Нет backup стратегии.
**Исправление:** Добавить volume driver и инструкции по бэкапу.

### 16. Отсутствие лимитов на размер запросов
**Файл:** `src/main.py`, эндпоинты
**Описание:** Можно отправить огромный JSON в `graph` или `search` и исчерпать память.
**Исправление:** Добавить FastAPI `max_request_size`.

## Рекомендации по приоритету исправлений

1. **Немедленно:** Path traversal (Critical #1) — security issue.
2. **Следующий релиз:** Race condition (#3) и недетерминистичные ID (#4) — риск потери данных.
3. **Планово:** Аутентификация API (#9), timeout Qdrant (#6), graceful degradation (#8).
4. **Технический долг:** Оптимизации chunking, метрики, тесты.
