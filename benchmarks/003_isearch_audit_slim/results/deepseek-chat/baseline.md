# Отчёт по аудиту проекта `isearch`

## Обзор

Проведён глубокий аудит кодовой базы поисковой системы `isearch`. Анализ охватывает архитектурные, логические и инфраструктурные аспекты с фокусом на обработке данных, контрактах API, хранении состояния и пограничных случаях.

## Классификация дефектов

### 1. Архитектурные дефекты

#### 1.1. Несогласованность конфигурации Qdrant
**Критичность:** Высокая  
**Файлы:** `src/config.py`, `src/vector_store.py`, `debug_qdrant.py`  
**Описание:** В проекте присутствует конфликт режимов работы с Qdrant:
- `src/config.py` определяет `QDRANT_URL = os.getenv("QDRANT_URL", "http://localhost:6333")`
- `debug_qdrant.py` использует локальный режим с `QdrantClient(path="./test_qdrant")`
- `docker-compose.yml` поднимает контейнер на порту 6333
- `src/vector_store.py` использует только URL-режим

**Риски:** Непредсказуемое поведение в разных окружениях, сложность отладки.

#### 1.2. Смешение ответственности в `src/main.py`
**Критичность:** Средняя  
**Файл:** `src/main.py`  
**Описание:** Монолитный файл (800+ строк) объединяет:
- Инициализацию приложения и жизненный цикл
- Все эндпоинты API
- Логику поиска и кластеризации
- Фоновые задачи индексации
- Управление состоянием задач

**Риски:** Сложность поддержки, тестирования и масштабирования.

#### 1.3. Отсутствие стратегии обработки ошибок
**Критичность:** Высокая  
**Файлы:** `src/main.py`, `src/services.py`, `src/vector_store.py`  
**Описание:** Разрозненная обработка исключений:
- В некоторых местах используются голые `except Exception`
- В других - специфичные исключения без восстановления
- Нет единого механизма логирования ошибок
- Отсутствуют retry-стратегии для сетевых операций

### 2. Логические дефекты

#### 2.1. Уязвимость Path Traversal в поиске
**Критичность:** Критическая  
**Файл:** `src/main.py` (функция `extract_snippet`)  
**Описание:** В функции `extract_snippet` используется конкатенация путей:
```python
project_path / source_file
```
где `source_file` приходит из ненадёжного источника (векторной БД).

**Риски:** Возможность чтения произвольных файлов на сервере через конструкции типа `../../etc/passwd`.

#### 2.2. Небезопасная генерация ID для Qdrant
**Критичность:** Высокая  
**Файл:** `src/vector_store.py` (метод `upsert`)  
**Описание:** Используется `abs(hash(...))` для генерации ID:
```python
ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]
```
**Проблемы:**
1. Хэш Python зависит от `PYTHONHASHSEED` (недетерминирован между запусками)
2. `abs()` может создавать коллизии (|x| == |-x|)
3. Возможны дубликаты при переиндексации

#### 2.3. Гонка данных при индексации
**Критичность:** Высокая  
**Файл:** `src/services.py` (класс `IndexingService`)  
**Описание:** Отсутствие блокировок при параллельной индексации одного проекта:
- Несколько задач могут одновременно модифицировать одну коллекцию
- Возможна потеря данных или повреждение индекса
- Нет механизма `optimistic locking` или транзакций

#### 2.4. Проблемы с обработкой Unicode
**Критичность:** Средняя  
**Файлы:** `src/graph_builder.py`, `src/index.py`  
**Описание:** Несогласованная обработка кодировок:
- В `graph_builder.py` явно указано `encoding="utf-8"`
- В `index.py` используется `errors='ignore'` без логирования
- Возможна потеря данных для файлов в других кодировках

### 3. Инфраструктурные дефекты

#### 3.1. Утечки памяти в обработке больших проектов
**Критичность:** Высокая  
**Файлы:** `src/services.py`, `src/index.py`  
**Описание:**
- Загрузка всех файлов проекта в память перед обработкой
- Отсутствие потоковой обработки для больших файлов
- Нет ограничений на размер батчей при upsert в Qdrant

#### 3.2. Отсутствие мониторинга и метрик
**Критичность:** Средняя  
**Файлы:** Все основные модули  
**Описание:**
- Нет метрик производительности (latency, throughput)
- Отсутствует мониторинг использования памяти/CPU
- Нет алертинга на сбои индексации или поиска

#### 3.3. Проблемы с управлением зависимостями
**Критичность:** Низкая  
**Файлы:** `pyproject.toml`, `setup.py`  
**Описание:**
- Дублирование конфигурации в `pyproject.toml` и `setup.py`
- Жёсткая привязка к версиям (`transformers==4.38.2`)
- Отсутствие разделения зависимостей для production/dev

### 4. Дефекты API и контрактов

#### 4.1. Несогласованность фильтрации по scope
**Критичность:** Высокая  
**Файлы:** `src/main.py`, `src/services.py`  
**Описание:** В поиске используется фильтр по `file_type`:
- В `services.py` задаются значения `"code"` и `"docs"`
- В `main.py` эндпоинт принимает `scope: Literal["code", "docs"]`
- Но нет гарантии, что все документы имеют корректный `file_type`

#### 4.2. Отсутствие валидации входных данных
**Критичность:** Средняя  
**Файл:** `src/main.py`  
**Описание:**
- Нет ограничений на длину запроса поиска
- Нет валидации параметров `start_path` и `depth` для графов
- Возможны DoS-атаки через большие или сложные запросы

#### 4.3. Проблемы с асинхронностью
**Критичность:** Средняя  
**Файл:** `src/main.py`  
**Описание:** Фоновые задачи индексации используют синхронный код в асинхронном контексте:
- Блокировка event loop при длительных операциях
- Нет механизма отмены задач
- Ограничение на один worker (`--workers 1`)

### 5. Дефекты тестирования

#### 5.1. Хрупкие интеграционные тесты
**Критичность:** Низкая  
**Файлы:** `tests/integration/test_api_*.py`  
**Описание:**
- Зависимость от внешних сервисов (Qdrant)
- Длительное время выполнения из-за индексации в фикстурах
- Отсутствие изоляции между тестами

#### 5.2. Неполное покрытие кода
**Критичность:** Низкая  
**Файлы:** Все тестовые модули  
**Описание:**
- Отсутствуют тесты для `src/embedder.py`
- Нет тестов на обработку ошибок
- Не покрыты edge cases (пустые файлы, бинарные данные)

## Рекомендации по приоритету

### Критический приоритет (необходимо исправить немедленно):
1. Path Traversal уязвимость в `extract_snippet`
2. Гонка данных при параллельной индексации
3. Небезопасная генерация ID для Qdrant

### Высокий приоритет (исправить в ближайшем релизе):
1. Утечки памяти при обработке больших проектов
2. Несогласованность конфигурации Qdrant
3. Отсутствие стратегии обработки ошибок

### Средний приоритет (планировать на следующий квартал):
1. Рефакторинг монолитного `main.py`
2. Добавление валидации входных данных API
3. Улучшение обработки Unicode

### Низкий приоритет (долгосрочные улучшения):
1. Добавление мониторинга и метрик
2. Улучшение тестового покрытия
3. Оптимизация управления зависимостями

## Заключение

Проект `isearch` демонстрирует хорошую архитектурную основу для семантического поиска, но содержит ряд критических уязвимостей безопасности и проблем с надёжностью. Наиболее серьёзные проблемы связаны с безопасностью (Path Traversal) и целостностью данных (гонки, недетерминированные ID). Рекомендуется начать с исправления критических дефектов перед развёртыванием в production-окружении.
