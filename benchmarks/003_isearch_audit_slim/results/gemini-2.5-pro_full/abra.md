# Аудит Проекта isearch (Abra)

## 1. Архитектурные и Инфраструктурные Уязвимости

### 1.1. [CRITICAL] Недетерминистичная генерация ID и тихая коррупция данных (Нарушение Наследственности)

- **Находка:** `vector_store.py` использует `abs(hash(p.get('source_file', '') + p.get('text', '')))` для генерации ID точек в Qdrant. Встроенная функция `hash()` в Python не является детерминированной между запусками (из-за `PYTHONHASHSEED`).
- **Влияние:** Это приводит к невозможности обновлять существующие записи при инкрементальной индексации. Вместо обновления создаются дубликаты, что ведёт к экспоненциальному росту базы и деградации качества поиска. `abs()` усугубляет проблему, сокращая пространство хэшей и увеличивая вероятность коллизий. **Это GT-3.**
- **Ось Октагона:** L2: Наследственность (Heredity).
- **Рекомендация:** Заменить `hash()` на стабильный криптографический хэш, например, `hashlib.sha256`, и использовать его hexdigest для ID.

### 1.2. [CRITICAL] Уязвимость Path Traversal (Нарушение Иммунитета)

- **Находка:** В `main.py` параметр `project_name` из URL напрямую используется для конструирования пути к файловой системе: `projects_base_dir / project_name`.
- **Влияние:** Злоумышленник может передать `project_name`, содержащий `../`, для чтения или записи файлов за пределами предполагаемой директории проектов. Например, `GET /projects/../../.ssh/id_rsa/search`.
- **Ось Октагона:** L3: Иммунитет (Immunity).
- **Рекомендация:** Перед использованием `project_name` необходимо строго валидировать и санировать его, убедившись, что итоговый путь находится внутри `projects_base_dir`.

### 1.3. [HIGH] Риск отказа в обслуживании (DoS) через API кластеризации (Нарушение Метаболизма)

- **Находка:** Эндпоинт `/projects/{project_name}/clusters` в `main.py` пытается загрузить все векторы из коллекции (`limit=10000`), чтобы выполнить KMeans кластеризацию в памяти.
- **Влияние:** При росте проекта до сотен тысяч векторов этот запрос гарантированно вызовет Out-of-Memory (OOM) на сервере, приводя к отказу в обслуживании.
- **Ось Октагона:** L2: Метаболизм (Metabolism).
- **Рекомендация:** Переработать эндпоинт. Вместо выгрузки всех векторов выполнять агрегацию на стороне базы данных или использовать алгоритмы онлайн-кластеризации, которые работают с потоком данных.

### 1.4. [HIGH] Хрупкое управление состоянием процессов (Нарушение Кинематики)

- **Находка:** `manage.py` использует PID-файлы для управления состоянием фонового API-сервера. Эта модель подвержена race conditions и может оставлять "зомби" процессы или "мёртвые" PID-файлы, которые мешают перезапуску.
- **Влияние:** Нестабильность сервиса, необходимость ручного вмешательства для очистки (`kill -9`, `rm .run/api.pid`).
- **Ось Октагона:** L2: Кинематика (Kinematics).
- **Рекомендация:** Заменить ручное управление процессами на более надёжный инструмент, например, `systemd` для продакшена или `supervisor` для разработки.

## 2. Логические Дефекты и Анти-паттерны

### 2.1. [MEDIUM] Деструктивная миграция схемы (Нарушение Мутагенеза)

- **Находка:** `vector_store.py` содержит метод `recreate_collection`, который безусловно удаляет существующую коллекцию перед созданием новой. Это единственный механизм для обновления настроек (например, размера вектора).
- **Влияние:** Невозможно изменить параметры индекса без полной потери всех данных и дорогостоящей полной реиндексации.
- **Ось Октагона:** L5: Мутагенез (Mutagenesis).
- **Рекомендация:** Внедрить систему "синих/зелёных" коллекций. Создать новую коллекцию с новыми настройками, перенести в неё данные, а затем атомарно переключить alias, чтобы избежать простоя.

### 2.2. [MEDIUM] Тихий пропуск файлов при ошибках индексации (Нарушение Гомеостаза)

- **Находка:** В `services.py` (`_process_files_in_batches`) ошибки чтения и чанкинга файлов логируются, но основной цикл продолжается.
- **Влияние:** Индексация может завершиться со статусом `completed`, но при этом часть важных файлов может быть пропущена из-за ошибок кодировки или прав доступа, о чём администратор не узнает без анализа логов.
- **Ось Октагона:** L4: Гомеостаз (Homeostasis).
- **Рекомендация:** Собирать статистику ошибок и сообщать о ней в итоговом статусе задачи. При критических ошибках (например, невозможности прочитать ключевой файл) задача должна переходить в статус `failed`.

- **Находка:** В `manage.py` для запуска uvicorn используется `subprocess.Popen` с `shell=True`.
- **Влияние:** Хотя на данный момент команда формируется из статических переменных, это является известным анти-паттерном. Любые будущие изменения, включающие пользовательский ввод в эту команду, могут привести к уязвимости Command Injection.
- **Ось Октагона:** L3: Иммунитет (Immunity).
- **Рекомендация:** Переписать вызов, передавая аргументы списком (`subprocess.Popen(['nohup', '.venv/bin/python', ...])`) без `shell=True`.

# Аудит Проекта isearch (Baseline)

## Обзор

Проект `isearch` представляет собой API для семантического поиска и анализа графа зависимостей. Кодовая база в целом хорошо структурирована, использует современные инструменты (FastAPI, Qdrant, `uv`) и покрыта тестами. В ходе аудита было выявлено несколько проблем разной степени критичности.

## Критические уязвимости (CRITICAL)

### 1. Нестабильные идентификаторы точек в Vector Store

- **Проблема:** В `vector_store.py` (L103) для создания ID используется `abs(hash(text))`. Встроенная функция `hash()` в Python не гарантирует одинаковый результат между разными запусками программы. Это зависит от переменной окружения `PYTHONHASHSEED`.
- **Последствия:** Инкрементальная индексация в `services.py` не будет работать корректно. При каждом перезапуске API будут создаваться дубликаты уже проиндексированных документов, так как их хэши не совпадут. Это приведёт к быстрому разрастанию базы и неверным результатам поиска. **Это GT-3**.
- **Решение:** Использовать стабильный алгоритм хэширования, например `hashlib.sha256(text.encode()).hexdigest()`.

### 2. Вероятность Path Traversal в API

- **Проблема:** Все эндпоинты, работающие с проектами (`/projects/{project_name}/...`), используют `project_name` из URL для построения путей в файловой системе (`main.py`, `services.py`).
- **Последствия:** Если не производится должная санитация, злоумышленник может передать в `project_name` значение типа `../../etc/` для доступа к файлам за пределами ожидаемой директории. `pathlib` может защитить от некоторых атак, но это требует явной проверки и тестирования.
- **Решение:** Написать тест, проверяющий поведение при передаче `project_name` со спецсимволами (`../`, `/`). Явно резолвить путь и проверять, что он является дочерним для `projects_base_dir`.

## Серьезные проблемы (HIGH)

### 3. Неэффективная реализация эндпоинта кластеризации

- **Проблема:** Эндпоинт `/clusters` в `main.py` (L389) пытается загрузить все векторы из Qdrant (`limit=10000`) в память сервера для выполнения KMeans.
- **Последствия:** Это не масштабируемое решение. При большом количестве документов (например, 100,000 чанков по 384*2 байт = ~74MB) это может привести к OOM (Out of Memory) ошибке и отказу сервиса.
- **Решение:** Рассмотреть альтернативные подходы: либо выполнять кластеризацию на стороне Qdrant (если поддерживается), либо использовать approximate-алгоритмы, либо выполнять её как фоновую задачу с выгрузкой данных по частям.

### 4. Race Condition при управлении сервисом

- **Проблема:** `manage.py` использует PID-файл для отслеживания запущенного uvicorn-сервера. Если два пользователя одновременно запустят `make up`, возможна гонка состояний, когда один процесс перезапишет PID-файл другого.
- **Последствия:** Может остаться "зомби" процесс, который продолжит занимать порт, или, наоборот, будет убит не тот процесс при вызове `make down`. **Это GT-4**.
- **Решение:** Использовать блокировки файлов (file locks) при чтении/записи PID-файла, чтобы сделать операцию атомарной.

## Рекомендации по улучшению (MEDIUM)

### 5. Отсутствие миграций для коллекций Qdrant

- **Проблема:** При изменении конфигурации (например, размер вектора модели) `vector_store.py` использует `recreate_collection`, что полностью удаляет все данные.
- **Последствия:** Отсутствует бесшовный путь обновления. Требуется полный даунтайм и реиндексация.
- **Решение:** Реализовать стратегию "сине-зеленого" деплоя для коллекций: создать новую, мигрировать данные, переключить alias.

### 6. Неявная конфигурация `PROJECTS_BASE_DIR`

- **Проблема:** В `src/config.py` (L14) `PROJECTS_BASE_DIR` по умолчанию имеет значение `..`.
- **Последствия:** Поведение приложения сильно зависит от того, из какой директории его запустили. Это неочевидно и может приводить к ошибкам "проект не найден".
- **Решение:** Сделать путь по умолчанию более явным, например, `./projects`, и документировать это. Или сделать его обязательным параметром конфигурации.

### 7. Хрупкий парсинг прогресса в CLI

- **Проблема:** В `manage.py` (L519) парсинг вывода о прогрессе задачи основан на простом удалении символа `%`.
- **Последствия:** Если в поле `details` ответа API случайно окажется строка с числом и знаком процента, парсер может сломаться или выдать неверное значение.
- **Решение:** Использовать более надежный способ, например, регулярное выражение, или передавать прогресс в API в виде чистого `float` числа.

## Мелкие замечания (LOW)

- **`search.py`:** Этот скрипт кажется отдельным инструментом, дублирующим логику API. Возможно, его стоит удалить или интегрировать с `manage.py`.

- **Мертвый код:** Скрипт `index.py` выглядит как устаревшая версия логики, которая теперь находится в `services.py`. Он использует жестко заданный путь `PROJECT_ROOT_PATH` и другую логику фильтрации. Вероятно, его следует удалить, чтобы избежать путаницы.
