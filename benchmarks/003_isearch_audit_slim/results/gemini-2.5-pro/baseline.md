### Категория: Безопасность (Security)

| ID | Проблема | Критичность | Компоненты |
| :--- | :--- | :--- | :--- |
| **SEC-01** | **Path Traversal в API-эндпоинтах** | **Критический** | `src/main.py`, `src/config.py` |

**Описание:**
Функция `get_projects_base_dir` в `src/config.py` по умолчанию возвращает `Path("..")`. Это значит, что базовый каталог для проектов находится на уровень выше директории, из которой запущено приложение. API-эндпоинты (`/projects/{project_name}/...`) принимают `project_name` как часть URL и конкатенируют его с `projects_base_dir`.

Отсутствует должная санация параметра `project_name`. Злоумышленник может передать в качестве имени проекта значение вида `../../etc` и, если такой путь существует, API попытается его обработать (просканировать, проиндексировать, построить граф).

**Влияние:**
Это классическая уязвимость "Path Traversal". Злоумышленник может получить доступ к файловой системе сервера за пределами предполагаемой директории с проектами. В зависимости от прав пользователя, под которым запущено приложение, это может привести к чтению произвольных файлов (например, `/etc/passwd`), раскрытию исходного кода самого сервиса или других чувствительных данных.

**Рекомендация:**
1.  Изменить значение по умолчанию для `PROJECTS_BASE_DIR` на более безопасное, например, `./projects`.
2.  Реализовать строгую валидацию `project_name`. Перед выполнением любых операций с файлами необходимо убедиться, что разрешённый (resolved) путь `(projects_base_dir / project_name).resolve()` является поддиректорией `projects_base_dir.resolve()`.

### Категория: Целостность данных (Data Integrity)

| ID | Проблема | Критичность | Компоненты |
| :--- | :--- | :--- | :--- |
| **DI-01** | **Недетерминированная генерация ID для точек в Qdrant** | **Критический** | `src/vector_store.py` |

**Описание:**
Метод `upsert` в `VectorStore` генерирует ID для точек в Qdrant, используя `ids = [abs(hash(p.get('source_file', '') + p.get('text', '')))]`. Встроенная функция `hash()` в Python **не является стабильной** между различными запусками интерпретатора по соображениям безопасности (защита от атак на коллизии). Её результат зависит от переменной окружения `PYTHONHASHSEED`.

**Влияние:**
При каждом перезапуске API-сервера или скрипта индексации для одних и тех же чанков будут сгенерированы **разные ID**. Вместо обновления существующих точек (upsert), в Qdrant будут создаваться их дубликаты. Это приводит к раздуванию индекса, неверным результатам поиска (один и тот же результат будет появляться несколько раз) и нарушению логики инкрементальной индексации. Использование `abs()` усугубляет проблему, сокращая пространство хэшей вдвое и увеличивая вероятность коллизий.

**Рекомендация:**
Заменить `hash()` на стабильный и детерминированный алгоритм хеширования, например, `hashlib.sha256` или `hashlib.md5`, с последующим преобразованием в `int`.

### Категория: Архитектура и Логика (Architecture & Logic)

| ID | Проблема | Критичность | Компоненты |
| :--- | :--- | :--- | :--- |
| **ARC-01** | **Состояние гонки (Race Condition) при параллельной индексации** | **Высокий** | `src/main.py`, `src/services.py` |

**Описание:**
API позволяет запускать несколько задач индексации асинхронно. Если две задачи запущены для одного и того же проекта одновременно, отсутствует механизм блокировки. Оба экземпляра `IndexingService` прочитают состояние индекса, вычислят разницу с локальными файлами и начнут операции удаления/добавления.

**Влияние:**
Это приведёт к непредсказуемому состоянию индекса. Например, один процесс может удалить точки для файла, который другой процесс в этот момент переиндексирует, что приведет к потере данных. Или оба процесса могут одновременно пытаться добавить одни и те же точки.

**Рекомендация:**
Внедрить механизм блокировки на уровне проекта. Перед началом индексации сервис должен устанавливать блокировку (например, в Redis, или создавая `.lock` файл в директории проекта). Если блокировка уже существует, новая задача индексации должна быть отклонена или поставлена в очередь.

| ID | Проблема | Критичность | Компоненты |
| :--- | :--- | :--- | :--- |
| **ARC-02** | **Наивный алгоритм разбиения на чанки (Chunking)** | **Высокий** | `src/services.py` |

**Описание:**
Алгоритм чанкинга в `_process_files_in_batches` использует простую нарезку по фиксированному количеству строк (`chunk_size_lines`). Этот метод не учитывает синтаксическую и семантическую структуру исходного кода или текста.

**Влияние:**
Функции, классы, логические блоки и даже отдельные предложения могут быть разорваны на части между разными чанками. Это значительно снижает качество эмбеддингов, так как модель получает на вход семантически неполные фрагменты. Как следствие — деградация качества поиска.

**Рекомендация:**
Использовать более продвинутые стратегии чанкинга:
*   **Для кода:** Рекурсивное разделение на основе AST-дерева (классы, функции) или использование специализированных сплиттеров, знающих синтаксис языка.
*   **Для текста:** Разделение по предложениям или параграфам с использованием библиотек NLP (например, `nltk`).

| ID | Проблема | Критичность | Компоненты |
| :--- | :--- | :--- | :--- |
| **ARC-03** | **Загрузка всех векторов в память для кластеризации** | **Высокий** | `src/main.py` |

**Описание:**
Эндпоинт `/projects/{project_name}/clusters` использует `client.scroll` с лимитом `10000` для загрузки **всех** векторов проекта в память сервера для последующей KMeans-кластеризации.

**Влияние:**
Для больших проектов это гарантированно приведет к исчерпанию оперативной памяти (OOM Killer) и падению сервиса. Это является вектором DoS-атаки. Лимит в 10000 точек может быть недостаточен для полного охвата проекта, что приведёт к нерепрезентативной кластеризации.

**Рекомендация:**
*   Реализовать пагинацию или потоковую обработку векторов.
*   Рассмотреть алгоритмы кластеризации, которые могут работать с данными итеративно (например, `MiniBatchKMeans` из `scikit-learn`).
*   Ввести разумный верхний лимит на количество точек для кластеризации и возвращать ошибку, если он превышен.

| ID | Проблема | Критичность | Компоненты |
| :--- | :--- | :--- | :--- |
| **ARC-04** | **Хранение статуса задач в памяти процесса** | **Средний** | `src/main.py` |

**Описание:**
Глобальный словарь `TASK_STATUS` используется для отслеживания состояния фоновых задач индексации. Это состояние не является персистентным.

**Влияние:**
При любом перезапуске, сбое или масштабировании API-сервера (если будет >1 worker'а), вся информация о текущих и завершенных задачах будет потеряна. Клиенты не смогут узнать статус задач, запущенных до перезапуска.

**Рекомендация:**
Использовать внешнее, персистентное хранилище для статусов задач, например, Redis или простую SQL-таблицу.

| ID | Проблема | Критичность | Компоненты |
| :--- | :--- | :--- | :--- |
| **ARC-05** | **Пост-фильтрация вместо query-time бустинга в поиске** | **Средний** | `src/main.py` |

**Описание:**
Логика повышения релевантности (бустинга) для `code` или `docs` в `search_project_endpoint` применяется **после** получения `top_k` результатов из Qdrant.

**Влияние:**
Если релевантный документ (например, `docs`) получил от векторного поиска оценку, которая поместила его на `k+1` место, он никогда не попадет в итоговый результат, даже если запрос явно ориентирован на документацию. Бустинг не сможет "поднять" его в топ.

**Рекомендация:**
Реализовать бустинг на стороне Qdrant, если это возможно. В противном случае, запрашивать из Qdrant больше результатов (например, `k*3`), применять логику бустинга, а затем возвращать итоговые `top_k` результатов.
