# Концептуальный Протокол: Аудит Кодовой Базы `isearch`

## 1. Топология задачи (Ontology)
- **Мета-тип:** Аудит распределенной системы с ML-компонентами и хранением векторов.
- **Ключевой конфликт:** Баланс между производительностью/удобством разработки и надежностью/целостностью данных.

## 2. Инварианты (Invariants)
1.  **Закон сохранения Целостности Данных:** Любые операции с данными (индексация, удаление) должны быть идемпотентны и не приводить к потере, дублированию или некорректному обновлению данных.
2.  **Предел Пропускной Способности:** Системы с I/O- и CPU/GPU-интенсивными операциями (ML-инференс, файловый ввод-вывод) имеют жесткие лимиты по параллелизму, которые нельзя нарушать без деградации доступности.
3.  **Принцип Минимальной Привилегии/Предсказуемости:** Взаимодействие с файловой системой и внешними сервисами должно быть строго ограничено и предсказуемо.

## 3. Точка опоры (Leverage Point)
- **Фокус внимания:** Устранение источников недетерминированности и обеспечение консистентности на уровне векторной базы данных. Любые усилия здесь многократно окупятся в стабильности поиска.
- **Игнорируемый шум:** Оптимизация незначительных деталей UI/UX или излишняя детализация логирования на данный момент отвлекает от фундаментальных проблем с данными и производительностью.

## 4. Векторы энтропии (Degradation Paths)
-   **Анти-паттерн А (Недетерминированные ID):** Система будет постепенно накапливать дубликаты или терять актуальные данные из-за непредсказуемого поведения хеш-функции.
-   **Анти-паттерн Б (Блокирующая Индексация):** Под нагрузкой API будет "зависать" или отвечать с большой задержкой, что приведет к тайм-аутам клиентов и падению доступности.
-   **Когнитивное искажение В (Implicit Contracts):** Доверие к неявным контрактам (например, неизменности `hash()` или консистентности внешних зависимостей без ретраев) неизбежно приведет к сбоям.

## 5. Алгоритм стабилизации (The Protocol)

-   **Фаза 0: Изоляция.**
    -   Изолировать процесс аудита от рабочей среды.
    -   Зафиксировать текущий `commit SHA` кода для воспроизводимости.
    -   Собрать минимальные метрики по производительности API под нагрузкой во время индексации.
-   **Фаза 1: Оцифровка базиса.**
    -   Провести анализ статического кода.
    -   Выявить все точки, где генерируются идентификаторы данных.
    -   Выявить все пути файлового доступа и параметры их контроля.
    -   Выявить все места, где фоновые задачи могут блокировать основной поток.
-   **Фаза 2: Применение рычага.**
    -   Перейти к детерминированному и стабильному формированию Qdrant Point ID.
    -   Разделить или изолировать ресурсоемкие фоновые задачи (индексация) от основной API-логики.
    -   Ввести явную, строгую валидацию путей для файловых операций.
    -   Унифицировать стратегию чанкирования данных.
-   **Фаза 3: Интеграция и Стабилизация.**
    -   Внедрить механизмы наблюдаемости для всех фоновых задач (прогресс, ошибки).
    -   Реализовать устойчивые механизмы подключения к Qdrant (ретраи, таймауты).
    -   Обновить документацию по best practices развертывания (пиннинг версий, управление процессами).

## 6. Метрика истины (Truth Criteria)
-   **Состояние успеха:**
    -   Qdrant не содержит дубликатов точек для одного и того же контента после множественных операций индексации/реиндексации.
    -   API отвечает на запросы с предсказуемой задержкой даже во время активной индексации.
    -   Система устойчива к некорректным путям, перехватывая их до попытки доступа к файловой системе.
    -   Логи и метрики однозначно отражают статус фоновых задач.
-   **Ложный сигнал успеха:** API успешно отвечает на запросы, но индексация происходит некорректно или данные в Qdrant испорчены.

## 7. Эвристики и Мета-Контекст (Heuristics & Meta-Context)

-   **Открытые вопросы:**
    -   Какова ожидаемая нагрузка на API во время индексации? Допустимо ли блокирование API на несколько секунд/минут?
    -   Планируется ли поддержка нескольких worker'ов FastAPI?
    -   Есть ли требования к устойчивости к кратковременным разрывам связи с Qdrant?
-   **Тактические предложения:**
    -   Для `manage.py` рассмотреть использование `systemd` или `supervisord` для надежного управления демоном `uvicorn` вместо `nohup`.
    -   Разделить `src/index.py` (CLI tool) и `src/services.py` (API service logic) на более явные, непересекающиеся пути. Возможно, `src/index.py` вообще убрать, оставив API как единственный способ индексации.
    -   Пересмотреть стратегию кеширования embeddings для горячих данных, чтобы избежать повторных вычислений.
-   **Комментарии оператора:** Проект `isearch` демонстрирует частые анти-паттерны, возникающие при быстром прототипировании. Ядро семантического поиска и графового анализа функционально, но обвязка вокруг него (операции с файловой системой, управление процессами, целостность данных в БД) требует существенного усиления для производственного использования.

## 8. Резолюция и Архитектурная Развилка (Approval Gate)

-   **Фундаментальный диагноз:** Система `isearch` страдает от фундаментальных проблем с целостностью данных, безопасностью и доступностью, вызванных недетерминированным управлением состоянием, неконсистентными операциями с данными и хрупкой инфраструктурной обвязкой.

-   **Стратегические варианты (The Forks):**

    -   **Вариант А: Инкрементальное Укрепление (Рекомендуется)**
        -   **Описание:** Поэтапное устранение критических и высокоприоритетных уязвимостей без переписывания основной логики. Фокус на стабилизации данных, повышении безопасности и устранении блокировок API.
        -   **Плюсы:** Быстрое снижение рисков, минимальное влияние на существующую кодовую базу.
        -   **Минусы:** Некоторые менее критичные проблемы могут остаться или проявятся позже.
        -   **Цена внедрения:** Низкая-средняя (несколько дней/недель работы).

    -   **Вариант Б: Рефакторинг Архитектуры (Долгосрочно)**
        -   **Описание:** Пересмотр архитектуры, включая использование отдельных процессов/сервисов для индексации, централизованное управление состояниями задач, и создание более надежного слоя абстракции над Qdrant.
        -   **Плюсы:** Высокая надежность, масштабируемость и устойчивость к ошибкам.
        -   **Минусы:** Высокая стоимость, длительный срок внедрения, риск введения новых багов при масштабных изменениях.
        -   **Цена внедрения:** Высокая (несколько недель/месяцев работы).

-   **Рекомендация ИИ:** Рекомендуется **Вариант А (Инкрементальное Укрепление)**. Он обеспечивает наилучшее соотношение затрат и снижения рисков для текущего состояния проекта. Критические проблемы, такие как Path Traversal и недетерминированные ID, должны быть устранены немедленно.

-   **Ближайшие физические шаги (Next Steps):**
    1.  **[APPROVAL GATE]** Ожидание решения оператора (Выбор Варианта А или Б).
    2.  **[ЗАПРЕТ]** ИИ не имеет права генерировать итоговый рабочий код до явного подтверждения выбора оператором.
