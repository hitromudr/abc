Отчёт об аудите кодовой базы проекта `isearch`

## Введение

Проект `isearch` представляет собой систему для семантического поиска и анализа графов зависимостей в кодовых базах. Целью данного аудита является выявление архитектурных, логических и инфраструктурных уязвимостей, а также проблем, связанных с обработкой данных, контрактами API, хранением состояния в векторной базе данных (Qdrant) и пограничными случаями.

Аудит проводился на основе предоставленного исходного кода, без доступа к реальной среде выполнения, с фокусом на потенциальные риски и области для улучшения.

## Обзор и Общие Выводы

Проект имеет четкую структуру, использует современные технологии (FastAPI, Qdrant, SentenceTransformers) и включает базовые механизмы для управления сервисами и взаимодействия с API. Тесты покрывают основные функциональные сценарии.

Однако в ходе аудита был выявлен ряд значительных проблем, затрагивающих надежность данных, масштабируемость и безопасность. Наиболее критичные из них связаны с недетерминированностью генерации ID в Qdrant, отсутствием персистентности состояния задач индексации, потенциальными гонками данных при индексации и недостаточной обработкой больших объемов данных в некоторых API.

### 1. Архитектурные Недостатки (Architectural Deficiencies)

| ID     | Описание проблемы                                                                                                                                                                                                                                                                                                                                                                                              | Критичность | Слой          |
| :----- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------- | :------------ |
| AR-01  | **Неперсистентное состояние задач индексации (`TASK_STATUS`)** <br> Словарь `TASK_STATUS` в `src/main.py` хранится в памяти. Это означает, что состояние всех фоновых задач индексации будет потеряно при перезапуске FastAPI-сервера, его падении или при использовании нескольких воркеров Uvicorn (каждый воркер будет иметь свою копию `TASK_STATUS`, что приведет к несогласованности и гонкам данных). | **Критично** | API/Infra     |
| AR-02  | **Игнорирование параметра `reindex` для полной переиндексации** <br> API-эндпоинт `/projects/{project_name}/index` принимает параметр `reindex: bool`, и на фронтенде есть кнопка "Full Re-index". Однако этот параметр игнорируется `IndexingService.run_indexing`. Сервис всегда выполняет инкрементальное обновление или создает индекс с нуля, если его нет. Это приводит к расхождению между ожиданиями пользователя/UI и фактическим поведением бэкенда. | **Высокая** | Functional/API |
| AR-03  | **Отсутствие пер-проектных блокировок при индексации (GT-4)** <br> Если несколько задач индексации для одного и того же проекта запускаются одновременно, они будут модифицировать одну и ту же коллекцию Qdrant. Это создает состояние гонки, которое может привести к повреждению данных или несогласованности, так как операции удаления/вставки могут накладываться друг на друга непредсказуемым образом. Несмотря на `uvicorn --workers 1`, `BackgroundTasks` могут запускаться параллельно. | **Высокая** | Data/Infra    |
| AR-04  | **Жестко заданный путь `PROJECT_ROOT_PATH` в `src/index.py`** <br> В скрипте `src/index.py` (предназначенном для автономного использования) `PROJECT_ROOT_PATH` жестко зашит на `../autowarp`. Это делает скрипт непригодным для других проектов без изменения кода и противоречит гибкому подходу `project_name`, используемому основным FastAPI-приложением. | **Средняя** | Maintainability |

### 2. Обработка Данных и Состояние Векторной Базы (Data Handling & Vector DB State)

| ID     | Описание проблемы                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     | Критичность | Слой              |
| :----- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------- | :---------------- |
| DH-01  | **Недетерминированные Point ID из `hash()` (GT-3)** <br> В `src/vector_store.py` ID для точек Qdrant генерируются с использованием `abs(hash(p.get('source_file', '') + p.get('text', '')))`. Функция `hash()` в Python по умолчанию недетерминирована между разными запусками интерпретатора (из-за `PYTHONHASHSEED`). Это означает, что один и тот же чанк контента может получить новый ID при перезапуске приложения, что приведет к созданию дубликатов вместо обновления существующих точек во время инкрементальной индексации. `abs()` дополнительно увеличивает вероятность коллизий. Это фундаментальный недостаток для идемпотентных операций upsert. | **Критично** | Data Consistency/Infra |
| DH-02  | **Ограниченный лимит `scroll` при кластеризации** <br> В `src/main.py` эндпоинт `perform_clustering` извлекает точки из Qdrant с `limit=10000`. Если проект содержит более 10 000 чанков документов, остальные чанки будут проигнорированы, что приведет к неполным или неверным результатам кластеризации. Метод `scroll` Qdrant поддерживает пагинацию, но она не используется.                                                                                                                                                                                                                                                                                                                                                                                                                     | **Высокая** | Data Completeness/Functional |
| DH-03  | **Неоптимальный размер батча для эмбеддингов в `_process_files_in_batches`** <br> В `src/services.py` функция `embedder.encode` вызывается с `batch_size=len(batch_docs)`. `batch_docs` может содержать большое количество файлов, каждый из которых, в свою очередь, может быть разбит на множество чанков. Это приводит к тому, что `len(chunks_for_batch)` может стать очень большим (например, 32 файла * 100 чанков/файл = 3200 чанков), что потенциально вызывает ошибки нехватки памяти (OOM), особенно на GPU, или существенно замедляет обработку. `batch_size` должен быть фиксированным, оптимальным для модели и оборудования (например, 64, 128). | **Высокая** | Performance/Memory |
| DH-04  | **Неявное понижение точности в `embedder.encode`** <br> Модель `SentenceTransformer` обычно выводит эмбеддинги в формате `float32`. Преобразование их в `float16` *после* кодирования (`embeddings.to(dtype=torch.float16)`) означает, что само кодирование происходит с полной точностью, а затем происходит потеря точности. Для оптимальной производительности/памяти (если требуется инференс `float16`), модель должна быть загружена или сконфигурирована для выполнения инференса непосредственно в `float16`. | **Низкая**  | Performance/Accuracy |

Это значение является значением по умолчанию для `SentenceTransformer.encode`, поэтому явное указание его здесь избыточно.                                                                                                                                                       | **Низкая**  | Code Quality |

### 4. Инфраструктура и Операции (Infrastructure & Operations)

| ID     | Описание проблемы                                                                                                                                                                                                                                                                                                                                                                                                                    | Критичность | Слой          |
| :----- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :---------- | :------------ |
| IO-01  | **Зависимость от `uv` без альтернативы** <br> `Makefile` использует `uv run` и `uv pip install`. Это жестко связывает проект с инструментом `uv`. Хотя `uv` — современный и хороший инструмент, для большей совместимости и гибкости желательно предоставить альтернативные инструкции по настройке окружения с использованием стандартных `python -m venv` и `pip install`, либо явно это задокументировать. | **Низкая**  | Infrastructure/Ops |
| IO-02  | **Агрессивный `sys.exit(1)` в CLI-скриптах** <br> В `manage.py`, вспомогательные функции `run_command` и `_api_request` вызывают `sys.exit(1)` при ошибке. Хотя это приемлемо для простых скриптов, для более сложного CLI это предотвращает любую очистку или альтернативную обработку ошибок в вызывающих контекстах, что снижает гибкость и отказоустойчивость. | **Низкая**  | Maintainability |
| IO-03  | **Использование bind mount вместо именованных томов Docker для Qdrant** <br> В `docker-compose.yml`, для Qdrant используется `bind mount` `./qdrant_storage:/qdrant/storage`. Хотя это обеспечивает персистентность данных, для управления данными Docker предпочтительнее использовать именованные тома (`named volumes`), так как они предоставляют лучшую изоляцию и управление жизненным циклом данных Docker. | **Низкая**  | Infrastructure/Ops |

### 5. Безопасность (Security Concerns)

| ID     | Описание проблемы                                                                                                                                                                                                                                                                                                                                                                                              | Критичность | Слой          |
| :----- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------- | :------------ |
| SEC-01 | **Потенциальная уязвимость "Path Traversal" в `graph_builder.py`** <br> В `src/graph_builder.py`, использование `Path.resolve()` в сочетании с произвольным пользовательским вводом (например, в `ref_target` или `link_target`) без строгой проверки того, что разрешенный путь остается в пределах `project_root`, может привести к Path Traversal. Злоумышленник может создать файл `.md` или `.py` со ссылками типа `../../../../etc/passwd.md`, которые, если будут проиндексированы, могут привести к раскрытию чувствительных путей или содержимого файлов в ответе API графа. Хотя проверка `is_relative_to(project_root)` присутствует, она применяется *после* `resolve()`. | **Высокая** | Security/Data |

## Общие Рекомендации

1.  **Приоритизация Исправлений Критичных Дефектов:** В первую очередь устранить проблемы DH-01 (недетерминированные ID) и AR-01 (неперсистентное состояние задач), так как они напрямую влияют на целостность данных и надежность системы. AR-03 (гонки при индексации) также требует немедленного внимания.
2.  **Улучшение Механизмов Индексации:**
    *   Реализовать полноценную обработку `reindex=True` в `IndexingService` для выполнения полного сброса и переиндексации.
    *   Внедрить строгие блокировки на уровне проекта в `IndexingService` (например, с использованием файла блокировки или распределенного мьютекса, если планируется кластерная архитектура) для предотвращения гонок данных.
    *   Оптимизировать использование батчей при создании эмбеддингов, чтобы избежать OOM-ошибок при больших объемах данных.
3.  **Повышение Устойчивости и Масштабируемости:**
    *   Перевести хранение состояния задач (`TASK_STATUS`) из памяти в персистентное хранилище (например, Redis, базу данных или даже Qdrant для простых метаданных), чтобы обеспечить устойчивость к перезапускам и поддержку нескольких воркеров.
    *   Доработать `perform_clustering` для использования пагинации Qdrant `scroll`, чтобы обрабатывать проекты с большим количеством чанков без потери данных.
4.  **Улучшение Безопасности:** Внедрить более строгую валидацию и санитаризацию путей, передаваемых в `graph_builder`, а также гарантировать, что все операции с файловой системой остаются строго внутри `project_root` *до* любых попыток чтения или обработки.
5.  **Оптимизация ML-модели:** Рассмотреть возможность загрузки `SentenceTransformer` с `torch_dtype=torch.float16` для сквозного инференса с пониженной точностью, если это приемлемо для качества поиска и требуется для оптимизации производительности/памяти.
6.  **Улучшение UI/UX:** Исправить инвертированное направление масштабирования в UI.

## Заключение

Проект `isearch` демонстрирует хороший потенциал, но требует значительной доработки в критических областях, особенно касающихся целостности и согласованности данных, а также обработки ошибок и масштабируемости. Устранение выявленных проблем позволит существенно повысить надежность, функциональность и безопасность системы.

## Введение

Проект `isearch` представляет собой систему для семантического поиска и анализа графов зависимостей в кодовых базах. Целью данного аудита является выявление архитектурных, логических и инфраструктурных уязвимостей, а также проблем, связанных с обработкой данных, контрактами API, хранением состояния в векторной базе данных (Qdrant) и пограничными случаями.

Аудит проводился на основе предоставленного исходного кода, без доступа к реальной среде выполнения, с фокусом на потенциальные риски и области для улучшения.

## Обзор и Общие Выводы

Проект имеет четкую структуру, использует современные технологии (FastAPI, Qdrant, SentenceTransformers) и включает базовые механизмы для управления сервисами и взаимодействия с API. Тесты покрывают основные функциональные сценарии.

Однако в ходе аудита был выявлен ряд значительных проблем, затрагивающих надежность данных, масштабируемость и безопасность. Наиболее критичные из них связаны с недетерминированностью генерации ID в Qdrant, отсутствием персистентности состояния задач индексации, потенциальными гонками данных при индексации и недостаточной обработкой больших объемов данных в некоторых API.
