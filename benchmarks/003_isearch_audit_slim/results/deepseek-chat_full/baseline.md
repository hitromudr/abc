# Отчёт по аудиту проекта `isearch`

**Проект:** iSearch (поисковая система)
**Дата аудита:** 2024-03-12
**Аудитор:** Senior Software Engineer

## Резюме

Проект `isearch` представляет собой полнофункциональную поисковую систему с семантическим поиском по коду и документации, анализом графов зависимостей и веб-интерфейсом. Архитектура включает FastAPI-сервер, векторную базу данных Qdrant, ML-модели для эмбеддингов и CLI для управления. Обнаружены критические проблемы в области безопасности, обработки данных и инфраструктуры, требующие немедленного внимания.

---

## Классификация дефектов

### 1. КРИТИЧЕСКИЕ (Critical) - Требуют немедленного исправления

#### 1.1. Path Traversal в API поиска
- **Файл:** `src/main.py`, функция `search_project_endpoint`
- **Описание:** Использование пользовательского ввода `source_file` для построения пути файла без валидации позволяет обойти ограничения директории проекта.
- **Код:** `project_path / source_file` (строка ~340)
- **Риск:** Чтение произвольных файлов на сервере, включая системные файлы.
- **Рекомендация:** Добавить проверку `os.path.commonpath([real_path, project_path]) == project_path`.

#### 1.2. Небезопасная генерация ID для точек Qdrant
- **Файл:** `src/vector_store.py`, метод `upsert`
- **Описание:** Использование `hash()` для генерации ID точек приводит к коллизиям и недетерминированному поведению (зависит от PYTHONHASHSEED).
- **Код:** `ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]`
- **Риск:** Потеря данных при реиндексации, дублирование точек, неконсистентность базы.
- **Рекомендация:** Использовать детерминированный хеш (SHA256) или UUID.

#### 1.3. Отсутствие блокировок при параллельной индексации
- **Файл:** `src/services.py`, класс `IndexingService`
- **Описание:** Несколько одновременных задач индексации одного проекта могут привести к гонкам данных и коррупции коллекции.
- **Риск:** Повреждение данных в Qdrant, потеря чанков, неконсистентное состояние.
- **Рекомендация:** Реализовать distributed lock на уровне проекта (Redis, файловая блокировка).

### 2. ВЫСОКИЕ (High) - Серьёзные функциональные проблемы

#### 2.1. Тихая потеря данных при батчевом апсерте
- **Файл:** `src/vector_store.py`, метод `upsert`
- **Описание:** Отсутствие обработки ошибок и валидации ответов Qdrant при батчевых операциях.
- **Риск:** Частичная потеря данных без уведомления пользователя.
- **Рекомендация:** Добавить проверку статуса ответа, ретраи с экспоненциальной задержкой.

#### 2.2. Неправильная обработка прогресса индексации
- **Файл:** `src/main.py`, функция `run_indexing_task`
- **Описание:** Прогресс обновляется в глобальном словаре без thread-safety механизмов.
- **Код:** `TASK_STATUS[task_id] = {**TASK_STATUS.get(task_id, {}), **status_update}`
- **Риск:** Race conditions, потеря обновлений прогресса при высокой нагрузке.
- **Рекомендация:** Использовать `threading.Lock` или `asyncio.Lock`.

#### 2.3. Уязвимость к DoS через большие файлы
- **Файл:** `src/services.py`, метод `_process_files_in_batches`
- **Описание:** Чтение всего файла в память перед чанкингом (`content = f.read()`).
- **Риск:** Исчерпание памяти при обработке больших файлов (например, дампов БД).
- **Рекомендация:** Построчное чтение с буферизацией.

### 3. СРЕДНИЕ (Medium) - Проблемы архитектуры и дизайна

#### 3.1. Смешение ответственностей в `src/index.py`
- **Описание:** Модуль содержит dead code (жестко закодированный `PROJECT_ROOT_PATH`), не используется в основном потоке.
- **Риск:** Путаница при поддержке, дублирование логики с `IndexingService`.
- **Рекомендация:** Удалить или переработать в утилиту для однократной индексации.

#### 3.2. Несогласованность конфигурации Qdrant
- **Файл:** `src/config.py` и `src/vector_store.py`
- **Описание:** `QDRANT_URL` используется только в `vector_store.py`, но не в `debug_qdrant.py` и других местах.
- **Риск:** Разное поведение в разных режимах (local vs remote).
- **Рекомендация:** Унифицировать конфигурацию, добавить режим работы (local/remote).

#### 3.3. Отсутствие валидации входных данных Graph API
- **Файл:** `src/main.py`, эндпоинты графа
- **Описание:** Параметры `start_path` и `depth` не валидируются на корректность.
- **Риск:** Рекурсивный обход произвольных путей, потенциальный DoS.
- **Рекомендация:** Добавить лимиты на глубину и проверку путей.

### 4. НИЗКИЕ (Low) - Вопросы код-стайла и оптимизации

#### 4.1. Избыточные импорты и зависимости
- **Файлы:** `src/embedder.py`, `src/main.py`
- **Описание:** Импорт `torch.backends.cudnn` без использования, `sklearn` для кластеризации.
- **Риск:** Увеличение размера образа, потенциальные конфликты версий.
- **Рекомендация:** Удалить неиспользуемые импорты, рассмотреть lightweight альтернативы.

#### 4.2. Неоптимальная работа с памятью в кластеризации
- **Файл:** `src/main.py`, функция `perform_clustering`
- **Описание:** Загрузка всех векторов коллекции в память (`limit=10000`).
- **Риск:** Исчерпание памяти на больших коллекциях.
- **Рекомендация:** Использовать итеративную обработку или выборку.

#### 4.3. Отсутствие мониторинга и метрик
- **Описание:** Нет сбора метрик производительности, ошибок, использования ресурсов.
- **Риск:** Сложность диагностики проблем в production.
- **Рекомендация:** Интегрировать Prometheus метрики, structured logging.

## Заключение

Проект `isearch` демонстрирует хорошую архитектурную основу с чётким разделением ответственности, но содержит критические уязвимости безопасности и проблемы обработки данных. Наиболее серьёзные проблемы связаны с отсутствием валидации пользовательского ввода и небезопасной работой с файловой системой. Рекомендуется начать исправления с критических проблем, затем перейти к улучшению надёжности и масштабируемости системы.
