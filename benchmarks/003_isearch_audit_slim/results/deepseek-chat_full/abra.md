# Концептуальный Протокол: Аудит семантической поисковой системы

## 1. Топология задачи (Ontology)
- **Мета-тип:** Аудит распределённой системы с ML-компонентами и векторной БД.
- **Ключевой конфликт:** Точность семантического поиска vs производительность и надёжность инфраструктуры.

## 2. Инварианты (Invariants)
1. **Инвариант пространства эмбеддингов:** Векторы запроса и документов должны быть сгенерированы одной моделью с одинаковыми параметрами.
2. **Инвариант идемпотентности индексации:** Повторная индексация тех же данных не должна создавать дубликатов.
3. **Инвариант безопасного ввода:** Пользовательские пути и запросы не должны приводить к инъекциям или чтению запрещённых файлов.

## 3. Точка опоры (Leverage Point)
- **Фокус:** Корректность генерации эмбеддингов и работы с Qdrant (векторная БД).
- **Шум:** Визуализация в UI, мелкие баги в CLI.

## 4. Векторы энтропии (Degradation Paths)
- **Анти-паттерн:** Молчаливое проглатывание ошибок в батчах Qdrant.
- **Когнитивное искажение:** Предположение, что "семантический поиск просто работает" без проверки качества эмбеддингов.

## 5. Алгоритм стабилизации (The Protocol)
**Фаза 0 (Цель и Границы):** Определить, что система должна давать релевантные результаты поиска по коду и документам. Границы — FastAPI + Qdrant + модели.
**Фаза 1 (Механизм):** Проверить корректность эмбеддингов, индексации, поиска. Убедиться в отсутствии race conditions.
**Фаза 2 (Защита и Осознание):** Проверить безопасность, обработку ошибок, наблюдаемость.

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** Поисковый запрос возвращает релевантные фрагменты кода/документов с оценкой >0.7.
- **Телеметрия:** Логи содержат информацию об ошибках, время ответа <1s.

## 7. Эвристики и Мета-Контекст
- **Открытые вопросы:** Насколько модель `paraphrase-multilingual-MiniLM-L12-v2` подходит для кода (vs документации)?
- **Тактические предложения:** Добавить валидацию размерности векторов при индексации.
- **Патч конвейера:** Дефектов не обнаружено.
- **Комментарии оператора:** Код выглядит зрелым, но есть типичные проблемы распределённых систем.

## 8. Резолюция (Executive Summary)
- **Фундаментальный диагноз:** Система имеет солидную архитектуру, но содержит критические уязвимости в обработке ошибок и потенциальные проблемы безопасности.
- **Стратегическое решение:** Исправить silent failures в Qdrant, добавить валидацию ввода, улучшить обработку edge cases.

# Детальный аудит проекта `isearch`

## Критические проблемы (Severity: Critical)

### 1. Небезопасное разрешение путей в `graph_builder.py` (Path Traversal)
**Файл:** `src/graph_builder.py`, функции `parse_python_file` и `parse_markdown_file`
**Описание:** Код использует `(file_path.parent / ref_target).resolve()` без проверки, что итоговый путь остаётся внутри `project_root`. Злонамеренный файл с комментарием `# ../../etc/passwd` может привести к чтению системных файлов.
**Риск:** Утечка конфиденциальных данных, чтение произвольных файлов.
**Исправление:** Добавить проверку `target_path.is_relative_to(project_root)` перед добавлением в зависимости.

### 2. Недетерминированные ID точек в Qdrant
**Файл:** `src/vector_store.py`, метод `upsert`
**Описание:** `ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]` использует `hash()`, который недетерминирован между запусками Python (`PYTHONHASHSEED`). При переиндексации создадутся дубликаты.
**Риск:** Раздувание базы, неконсистентность данных.
**Исправление:** Использовать `hashlib.sha256` или детерминированный хэш.

### 3. Молчаливое проглатывание ошибок батчей в Qdrant
**Файл:** `src/vector_store.py`, метод `upsert`
**Описание:** Если Qdrant отклонит часть батча (невалидный ID или размер вектора), исключение не логируется. Система считает, что данные записаны.
**Риск:** Тихая потеря данных, неконсистентность индекса.
**Исправление:** Проверять ответ `upsert` от Qdrant, логировать ошибки.

## Высокие проблемы (Severity: High)

### 4. Race condition при параллельной индексации
**Файл:** `src/services.py`, класс `IndexingService`
**Описание:** Если два процесса запустят индексацию одного проекта одновременно, будет гонка данных при удалении/добавлении точек. Нет блокировок на уровне проекта.
**Риск:** Повреждение индекса, потеря данных.
**Исправление:** Ввести `filelock` на уровне проекта или использовать атомарные операции Qdrant.

### 5. Отсутствие валидации размерности векторов
**Файл:** `src/vector_store.py`, метод `recreate_collection`
**Описание:** При пересоздании коллекции проверяется `vector_size`, но нет проверки, что `embedder.get_embedding_dim()` возвращает корректное значение (может быть 0 при ошибке загрузки модели).
**Риск:** Создание коллекции с неверной размерностью, поломка поиска.
**Исправление:** Добавить assert `vector_size > 0`.

### 6. Уязвимость к DoS через большие запросы
**Файл:** `src/main.py`, endpoint `/projects/{project_name}/search`
**Описание:** Параметр `top_k` ограничен только `ge=1, le=50`, но 50 — всё ещё много для больших коллекций. Нет лимитов на длину запроса `q`.
**Риск:** Исчерпание памяти/CPU при поиске с большим `top_k` или длинным запросом.
**Исправление:** Уменьшить max `top_k` до 20, добавить лимит на длину запроса.

## Средние проблемы (Severity: Medium)

### 7. Неполная обработка ошибок в `embedder.py`
**Файл:** `src/embedder.py`, метод `_load_and_optimize_model`
**Описание:** При ошибке загрузки модели логируется `log.critical`, но исключение пробрасывается. В `main.py` нет обработки этой ошибки на уровне lifespan.
**Риск:** Падение сервиса при недоступности Hugging Face.
**Исправление:** Добавить retry логику, fallback-модель.

### 8. Отсутствие мониторинга использования памяти
**Файл:** `src/embedder.py`, метод `encode`
**Описание:** `batch_size=8` фиксирован, но для больших моделей на GPU может вызвать OOM. Нет адаптивного батчинга.
**Риск:** OutOfMemory на GPU, падение сервиса.
**Исправление:** Динамический подбор `batch_size` на основе свободной памяти GPU.

### 9. Потенциальная утечка памяти в `services.py`
**Файл:** `src/services.py`, метод `_process_files_in_batches`
**Описание:** `all_embeddings.append(embeddings.cpu().numpy())` накапливает все эмбеддинги в памяти перед конкатенацией. Для больших проектов может исчерпать RAM.
**Риск:** OOM kill процесса индексации.
**Исправление:** Инкрементальная запись в Qdrant или использование генераторов.

## Низкие проблемы (Severity: Low)

### 10. Избыточные зависимости в `pyproject.toml`
**Файл:** `pyproject.toml`
**Описание:** `bitsandbytes`, `accelerate` не используются в коде (нет квантования, нет распределённого обучения). `torchvision` тоже избыточен.
**Риск:** Увеличение размера образа, потенциальные конфликты версий.
**Исправление:** Удалить неиспользуемые зависимости.

### 11. Жёстко закодированные пути в `index.py`
**Файл:** `src/index.py`, переменная `PROJECT_ROOT_PATH`
**Описание:** `PROJECT_ROOT_PATH = Path(__file__).parent.parent.parent / 'autowarp'` — хардкод. Скрипт не reusable.
**Риск:** Нельзя использовать для других проектов без редактирования кода.
**Исправление:** Сделать параметром командной строки.

### 12. Отсутствие типизации в возвращаемых значениях
**Файл:** `src/graph_builder.py`, функция `_resolve_import_path`
**Описание:** Возвращает `Path | None`, но в коде нет type hints для сложных структур.
**Риск:** Сложность рефакторинга, ошибки в runtime.
**Исправление:** Добавить `from typing import ...`, аннотировать возвращаемые типы.

### 13. Неоптимальный чанкинг в `services.py`
**Файл:** `src/services.py`, метод `_process_files_in_batches`
**Описание:** Чанкинг по фиксированному количеству строк (15) без учёта семантических границ.
**Риск:** Разрыв логических блоков кода (например, функция разделена между чанками).
**Исправление:** Умный чанкинг по AST для Python, по заголовкам для Markdown.

### 14. Отсутствие тестов на кириллические пути
**Файл:** `tests/unit/test_graph_builder.py`
**Описание:** Есть тест `test_build_graph_cyrillic_support`, но нет интеграционных тестов для поиска по файлам с кириллицей.
**Риск:** Падение при обработке Unicode в путях.
**Исправление:** Добавить интеграционный тест.

### 15. Устаревший метод `search` в `vector_store.py`
**Файл:** `src/vector_store.py`, метод `search`
**Описание:** В комментарии указано "The `search` method is deprecated. Using `query_points` instead.", но используется `query_points`. Название метода вводит в заблуждение.
**Риск:** Путаница при поддержке кода.
**Исправление:** Переименовать метод в `query`.

### 16. Отсутствие graceful shutdown в `manage.py`
**Файл:** `manage.py`, функция `service_up`
**Описание:** При `KeyboardInterrupt` убивается процесс, но нет гарантии корректного завершения Qdrant.
**Риск:** Повреждение данных Qdrant при аварийной остановке.
**Исправление:** Добавить обработчик SIGTERM, корректное завершение.
