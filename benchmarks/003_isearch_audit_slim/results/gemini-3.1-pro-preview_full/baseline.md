# Аудит кодовой базы проекта `isearch`

В ходе глубокого анализа предоставленной кодовой базы `isearch` были выявлены серьезные архитектурные, логические и инфраструктурные дефекты. Ниже представлен структурированный отчет с классификацией уязвимостей и оценкой их критичности.

---

## 🔴 КРИТИЧЕСКИЕ ДЕФЕКТЫ (CRITICAL)

### 1. Уязвимость Path Traversal (Безопасность)
**Слой:** API / Безопасность
**Локация:** `src/main.py` (эндпоинты `/search`, `/index`, `/graph`), `src/graph_builder.py`
**Описание:** Параметры `project_name` (из URL) и `start_path` (из тела `GraphRequest`) никак не санитизируются. Они конкатенируются с `projects_base_dir`:
`project_path = projects_base_dir / project_name`
**Влияние:** Злоумышленник может передать `project_name = "../../../etc"`, что позволит ему:
1. Проиндексировать и прочитать системные файлы сервера через эндпоинт поиска.
2. Построить граф файловой системы хоста.
3. Удалить произвольные коллекции в Qdrant, если угадает их названия.

### 2. Недетерминированные ID векторов и повреждение состояния (Хранение данных)
**Слой:** Инфраструктура / Векторная БД
**Локация:** `src/vector_store.py` (строка 62, метод `upsert`)
**Код:** `ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]`
**Описание:** Начиная с Python 3.3, функция `hash()` использует рандомизацию (`PYTHONHASHSEED`), что делает ее результаты недетерминированными между перезапусками процесса. 
**Влияние:** При каждом рестарте API-сервера (или при пересоздании воркера) хэши для одних и тех же файлов будут генерироваться разные. Это полностью ломает логику инкрементальной индексации: старые чанки не будут перезаписываться, база начнет дублировать данные. Кроме того, использование `abs()` удваивает вероятность коллизий хэшей.

---

## 🟠 ВЫСОКИЙ УРОВЕНЬ УГРОЗЫ (HIGH)

### 3. Состояние гонки (Race Condition) при индексации
**Слой:** Инфраструктура / Concurrency
**Локация:** `src/services.py` (метод `run_indexing`), `src/main.py` (метод `index_project_endpoint`)
**Описание:** Индексация запускается через `BackgroundTasks` (выполняется в пуле потоков). Логика `run_indexing` выполняет цикл: чтение состояния индекса -> вычисление дифа -> удаление -> вставка. В коде отсутствуют какие-либо блокировки (Locks) на уровне проекта.
**Влияние:** Если два пользователя одновременно запустят индексацию для одного и того же проекта, потоки прочитают одинаковое старое состояние и начнут параллельно удалять и вставлять данные. Это приведет к ошибкам Qdrant, дубликатам или "осиротевшим" векторам.

### 4. Несовпадение портов REST и gRPC (Отказ обслуживания БД)
**Слой:** Инфраструктура
**Локация:** `src/config.py`, `src/vector_store.py`, `docker-compose.yml`
**Описание:** В `docker-compose.yml` Qdrant биндит порты: `6333` (gRPC) и `6334` (REST). В конфиге установлено `QDRANT_URL = "http://localhost:6333"`. Однако инициализация клиента идет с флагом `prefer_grpc=False` (REST API).
**Влияние:** Клиент отправляет HTTP-запросы на gRPC-порт (6333). В зависимости от версии Qdrant это может приводить к таймаутам, сбросу соединений или падению парсера запросов в БД.

### 5. Усечение выборки при кластеризации (Потеря данных)
**Слой:** Логика / Графы
**Локация:** `src/main.py` (метод `perform_clustering`)
**Код:** `all_points = vector_store.client.scroll(... limit=10000)[0]`
**Описание:** Вызов метода `scroll` без использования `next_page_offset` в цикле. Забирается только первая страница результатов.
**Влияние:** Если проект содержит более 10,000 чанков (что стандартно для крупных репозиториев), кластеризация будет выполнена только для случайного подмножества данных. Остальные файлы будут проигнорированы, API вернет неполный ответ.

---

## 🟡 СРЕДНИЙ УРОВЕНЬ УГРОЗЫ (MEDIUM)

### 6. Экспоненциальное потребление памяти (OOM) при инкрементальной синхронизации
**Слой:** Архитектура / Производительность
**Локация:** `src/services.py` (метод `_get_indexed_state`)
**Описание:** Для вычисления того, какие файлы изменились, сервис скачивает полезную нагрузку (`with_payload=True`) *всех* точек из коллекции в память приложения словарем `{source_file: file_hash}`.
**Влияние:** На проекте из 100,000 чанков загрузка всех пейлоадов займет значительный объем оперативной памяти и приведет к перегрузке сети (Data Egress) между приложением и БД.

### 7. Потеря зависимостей (Imports) при построении графа кода
**Слой:** Логика / Парсинг AST
**Локация:** `src/graph_builder.py` (метод `parse_python_file`)
**Описание:** Скрипт использует `ast.walk` и фильтрует ноды только по `isinstance(node, ast.ImportFrom)`. Обычные импорты (вида `import utils` -> класс `ast.Import`) полностью игнорируются.
**Влияние:** Построенный граф зависимостей (`Code -> Code`) является неполным. Потеряются критические связи между файлами.

### 8. Сломанный "мертвый" код (Технический долг)
**Слой:** Логика / Сопровождаемость
**Локация:** `src/index.py`
**Описание:** Оставлен устаревший скрипт, использующий хардкод путей (`Path(__file__).parent.parent.parent / 'autowarp'`).
Вызов `vector_store.upsert(vectors=..., payloads=...)` падает с `TypeError`, так как сигнатура метода в `vector_store.py` требует позиционный аргумент `collection_name`, который сюда не передается.
**Влияние:** Запутывает разработчиков, тратит ресурсы на анализ мертвого кода.

---

## 🟢 НИЗКИЙ УРОВЕНЬ УГРОЗЫ (LOW / QUALITY)

### 9. Скрытие ошибок декодирования
**Слой:** Работа с данными
**Локация:** `src/services.py` и `src/index.py`
**Описание:** Использование `open(..., encoding="utf-8", errors="ignore")`.
**Влияние:** Проглатывает символы из файлов в кодировке cp1251 или бинарных файлов, которые случайно не отфильтровались. Это может привести к поврежденному нечитаемому тексту в векторной базе без логирования ошибки.

### 10. Наивное несемантическое чанкование кода
**Слой:** Продукт / ML
**Локация:** `src/services.py` (строки 136-137)
**Описание:** Код бьется жестко по 15 строк (`chunk_size_lines = 15`). 
**Влияние:** Для исходного кода это катастрофично, так как разрывает определения функций и классов на несвязанные куски. Семантический поиск по фрагментированному коду работает с низкой точностью (Low Recall). Необходимо использовать AST-ориентированный чанкер (например, бьющий по границам методов).

---
**Итог:** Архитектура имеет фундаментальный изъян в работе с состоянием векторной базы (недетерминированные ID) и требует немедленного патча безопасности для устранения Path Traversal перед любым выводом в production.
