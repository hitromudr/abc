# Концептуальный Протокол: Архитектурный Аудит isearch

## 1. Топология задачи (Ontology)
- **Мета-тип:** Аудит распределенной системы (API + Векторная БД + ML-эмбеддинги).
- **Ключевой конфликт:** Гибкость файловой интеграции (индексация локальных путей) конфликтует с изоляцией среды (Path Traversal), а асинхронная обработка данных конфликтует с консистентностью БД (Race Conditions).

## 2. Инварианты (Invariants)
1. **Закон детерминизма данных:** Одинаковый текст должен всегда указывать на один и тот же вектор в пространстве.
2. **Закон изоляции периметра (Мембрана):** Ни один пользовательский ввод (путь, имя проекта) не должен резолвиться за пределы `projects_base_dir`.
3. **Закон эксклюзивного доступа:** Изменение состояния одного агрегата (индекса проекта) должно блокировать параллельные мутации.

## 3. Векторы энтропии (Отчёт об уязвимостях)

Атака по осям Инженерного Октагона выявила 4 критических/высоких дефекта:

### 🔴 [CRITICAL] L3: Иммунитет — Absolute Path Injection / Directory Traversal
*   **Локация:** `src/main.py` (`build_project_graph`), `src/graph_builder.py:169` (`build_graph`).
*   **Симптом:** Переменная `start_path` берется из API (`GraphRequest`) и склеивается как `(project_root / start_path).resolve()`. В Python конструкция `Path('/any') / '/etc'` возвращает `/etc`. Если атакующий передаст `start_path = "/etc"`, система просканирует всю ОС, а эндпоинт поиска вернет куски системных файлов в виде сниппетов (Arbitrary File Read).
*   **Рычаг (Fix):** Строгая валидация `scan_dir.is_relative_to(project_root)` до начала обхода `os.walk`.

### 🟠 [HIGH] L2: Наследственность — Недетерминированные ID (Скрытая дубликация)
*   **Локация:** `src/vector_store.py:68` (`upsert`).
*   **Симптом:** Идентификаторы векторов генерируются через `ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) ...]`. Встроенная функция `hash()` в Python использует случайную соль (`PYTHONHASHSEED`) при каждом перезапуске интерпретатора. При перезагрузке API и повторной индексации старые чанки не будут перезаписаны — они продублируются с новыми ID. Использование `abs()` дополнительно удваивает вероятность коллизий.
*   **Рычаг (Fix):** Заменить `hash()` на детерминированную криптографическую хеш-функцию, например, конвертацию MD5/SHA256 в `int` (u64) или использование UUIDv5 на базе имени файла и текста.

### 🟠 [HIGH] L2: Кинематика — Состояние гонки при индексации (Race Condition)
*   **Локация:** `src/main.py:169` (`index_project_endpoint`).
*   **Симптом:** Метод триггерит `run_indexing_task` в пуле `BackgroundTasks`. Если пользователь (или CI/CD) вызовет эндпоинт дважды подряд, два потока начнут параллельно читать файлы и писать в одну коллекцию Qdrant. Это приведет к порче данных, конфликту удалений/апсертов и OOM.
*   **Рычаг (Fix):** Внедрить мьютекс (Lock) на уровне `project_name`. Запрещать запуск задачи, если статус в `TASK_STATUS` уже `running` или `queued`.

### 🟡 [MEDIUM] L1: Топология — Молчаливое усечение данных при кластеризации
*   **Локация:** `src/main.py:313` (`perform_clustering`).
*   **Симптом:** Запрос `all_points = vector_store.client.scroll(..., limit=10000)[0]` забирает только первые 10,000 чанков и не делает пагинацию по `next_page_offset`. Если в проекте 15,000 чанков, 33% базы будет молча проигнорировано при кластеризации.
*   **Рычаг (Fix):** Использовать цикл `while` с `next_page_offset` для выгрузки всего объема данных (как это сделано в `_get_indexed_state`).

### ⚪ [LOW] Эвристика — Мёртвый код
*   **Локация:** `src/index.py`.
*   **Симптом:** Файл содержит хардкод пути (`autowarp`) и дублирует логику `IndexingService`. Является мусорным артефактом. Подлежит удалению.

## 4. Алгоритм стабилизации (The Protocol)
1.  **Фаза Изоляции:** Немедленно пропатчить `graph_builder.py` и `main.py`, внедрив проверку `is_relative_to` для всех внешних путей (`project_name`, `start_path`).

3.  **Фаза Кинематики:** Добавить проверку `if TASK_STATUS.get(task_id, {}).get("status") in ["queued", "running"]: raise HTTPException(409)` в `index_project_endpoint`.
