**ШАГ 1. ЭМПИРИЧЕСКОЕ ЗАЗЕМЛЕНИЕ И KILL SWITCH**
- Данные (код) загружены из контекста. Метрики потенциального падения очевидны из AST: `hash()` в Python 3+ использует рандомный seed (`PYTHONHASHSEED`), что делает генерацию ID вектора случайной при перезапуске процесса (утечка/дубли в БД). Конкурентные запросы к `/index` не лочатся. Флаг `reindex=True` проигнорирован. Введенный пользователем `project_name` напрямую конкатенируется к пути (Path Traversal).

# Концептуальный Протокол: Аудит Архитектуры isearch

## 1. Топология задачи (Ontology)
- **Мета-тип:** Синхронизация распределенного состояния и изоляция среды.
- **Ключевой конфликт (Диалектика):** Stateless-природа FastAPI конфликтует со stateful-природой фоновых процессов (индексации) и персистентностью графа в Qdrant. Асинхронные задачи пытаются изменять общее состояние без механизма консенсуса (Locking).

## 2. Инварианты (Invariants)
1. **Закон детерминированности ключей:** Идентификаторы (IDs) в персистентном хранилище не должны зависеть от эфемерной памяти процесса (RAM/рандомизации).
2. **Закон непроницаемости мембраны:** Пользовательский ввод никогда не должен напрямую транслироваться в операции с физической файловой системой без резолва и проверки границ (Jail).

## 3. Точка опоры (Leverage Point)
- **Фокус (TOC):** Отсутствие оркестратора фоновых задач и отсутствие детерминированного генератора ID для чанков.
- **Шум:** Выбор конкретной ML-модели или размер батчей для GPU (текущие решения в этих областях работают адекватно и не являются bottleneck).

## 4. Векторы энтропии (Degradation Paths)
- **Анти-паттерн А (Memory Leak):** Сохранение состояния завершенных задач в глобальный словарь `TASK_STATUS` без механизма сборки мусора (TTL/Eviction).
- **Искажение Б (Ложная идемпотентность):** Вера в то, что Python `hash()` выдает одинаковые значения при разных запусках интерпретатора.

## 5. Алгоритм стабилизации (The Protocol)

### Фаза 0: Топология и Безопасность (L1)
- **[CRITICAL] Path Traversal (Arbitrary File Read/Write):** Параметр `project_name` в эндпоинтах `/projects/{project_name}/*` позволяет передать `../`. Это позволяет индексировать и извлекать сниппеты из любых файлов на диске (например, `/etc/` или конфиги других сервисов). 
  *Решение:* Использовать `Path.resolve().is_relative_to(projects_base_dir)` для валидации `project_name`.
- **[HIGH] Коллизия коллекций:** `project_name_to_collection("foo-bar")` и `project_name_to_collection("foo_bar")` генерируют одно и то же имя `proj_foo_bar`. Приводит к слипанию данных разных проектов.

### Фаза 1: Механизм и Наследственность (L2)
- **[HIGH] Недетерминированность ID (Silent batch duplicates / Loss of Data):** В `VectorStore.upsert` ID генерируется как `abs(hash(...))`. В Python 3+ используется `PYTHONHASHSEED`, который рандомизируется при каждом старте. Перезапуск API приведет к генерации совершенно новых ID для тех же самых данных, вызывая их дублирование. Более того, два одинаковых чанка в коде получат одинаковый `hash()` в рамках одного прогона и перезапишут друг друга.
  *Решение:* Использовать детерминированный алгоритм, например `int(hashlib.md5(text.encode()).hexdigest()[:15], 16)` или UUIDv5.
- **[HIGH] Пропуск параметра reindex:** В `index_project_endpoint` принимается флаг `reindex=True`, но он **игнорируется** при передаче в `run_indexing_task`. Выполнить полную переиндексацию сломанного проекта сейчас физически невозможно через API.
- **[HIGH] Утечка памяти (Memory Leak):** Словарь `TASK_STATUS` бесконечно накапливает данные. При активном использовании сервер упадет с OOM.

### Фаза 2: Защита и Гомеостаз (L3, L4)
- **[HIGH] Race Condition (Гонка данных):** Отсутствуют блокировки (Mutex) при вызове `run_indexing_task`. Два пользователя могут одновременно запустить индексацию одного проекта: процессы будут параллельно удалять и писать в базу, разрушая индекс.
- **[MEDIUM] Limit 10,000 в Кластеризации:** В `perform_clustering` хардкод `limit=10000` без механизма пагинации (`offset`). Проекты с более чем 10k чанками будут кластеризоваться только по первой части файлов.
- **[MEDIUM] Падение gRPC Payload:** В `delete_points_by_source` формируется гигантский `Filter` с массивом `should`. При удалении проекта на 10,000+ файлов этот запрос превысит лимиты Qdrant/gRPC. Необходима разбивка удаления на батчи.
- **[LOW] Мертвый код:** Скрипт `src/index.py` падает при запуске, так как вызывает `vector_store.recreate_collection(vector_size=embedding_dim)` без обязательного аргумента `collection_name`.

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** API валидирует все пути; перезапуск сервера не порождает новых ID для старых чанков; попытка запустить вторую индексацию того же проекта возвращает `409 Conflict`.
- **Телеметрия:** Мониторинг RAM процесса Uvicorn (стабилизация вместо линейного роста).

## 7. Эвристики и Мета-Контекст (Heuristics & Meta-Context)
- **Открытые вопросы:** Qdrant Client инициализируется с `prefer_grpc=False` (REST), но в `docker-compose.yml` порт 6333 подписан как `gRPC`. Необходимо синхронизировать документацию портов (обычно 6333 — REST, 6334 — gRPC).
- **Тактические предложения:** Внедрить `asyncio.Lock` в in-memory реестр для защиты `project_name` от конкурентных запусков `run_indexing_task`.

## 8. Резолюция (Executive Summary)
- **Фундаментальный диагноз:** Система страдает от эфемерности состояния (Python hash seed), критических уязвимостей периметра (Path Traversal) и гонок данных при асинхронной обработке.
- **Стратегическое решение:** Заменить генерацию ID на MD5/UUIDv5, внедрить строгую маршрутизацию и изоляцию проектов (Path validation) и добавить семафоры для фоновых задач.
- **Ближайшие физические шаги (Next Steps):**
  1. Исправить `extract_snippet` и `index_project_endpoint`, добавив `project_path.resolve().is_relative_to(projects_base_dir)`.
  2. Переписать генерацию `ids` в `VectorStore.upsert` на `hashlib`.
  3. Передать аргумент `reindex` в `run_indexing_task` и добавить логику полного удаления коллекции при его наличии.
