## Экзекьютив-саммари
Проект имеет хороший фундамент (FastAPI, векторная база, асинхронные задачи), однако текущая реализация содержит **критическую уязвимость безопасности (Path Traversal)**, фундаментальную **архитектурную ошибку в генерации ID для векторной базы** (которая полностью ломает заявленную "инкрементальную индексацию") и **утечки памяти (OOM)** при обработке крупных репозиториев.

## 1. Критические дефекты (CRITICAL)

### 1.1. Path Traversal (Уязвимость безопасности)
**Локация:** `src/main.py` (все эндпоинты, принимающие `project_name`), `src/services.py`
**Описание:** Имя проекта (`project_name`) передается пользователем через URL и напрямую склеивается с базовой директорией: `project_path = projects_base_dir / project_name`. 
Система никак не валидирует выход за пределы директории.
**Вектор атаки:** Злоумышленник может вызвать `POST /projects/..%2f..%2f..%2fetc/index` (где `%2f` — это `/`). Сервер проиндексирует директорию `/etc` в Qdrant. Затем злоумышленник через эндпоинт `/search` сможет извлекать содержимое любых файлов на сервере (например, `/etc/passwd` или `.env` файлы с ключами).
**Как исправить:** 
Внедрить строгую валидацию путей во всех эндпоинтах:
```python
resolved_path = (projects_base_dir / project_name).resolve()
if not resolved_path.is_relative_to(projects_base_dir.resolve()):
    raise HTTPException(status_code=400, detail="Invalid project name.")
```

### 1.2. Недетерминированная генерация ID векторов (Потеря данных / Дубликация)
**Локация:** `src/vector_store.py` (метод `upsert`)
**Описание:** Для генерации уникального ID векторного чанка используется встроенная функция `hash()`:
`ids = [abs(hash(p.get('source_file', '') + p.get('text', ''))) for p in payloads]`
В Python (начиная с 3.3) хеши строк рандомизируются при каждом перезапуске интерпретатора (из-за `PYTHONHASHSEED`). 
**Последствия:** При перезапуске API сервера один и тот же текст будет получать **абсолютно новые ID**. "Инкрементальная индексация" сломана: Qdrant не перезапишет старые чанки, а создаст дубликаты. База данных будет экспоненциально раздуваться, а качество поиска деградирует из-за одинаковых результатов. Кроме того, `abs(hash())` удваивает вероятность коллизий.
**Как исправить:** Использовать криптографический хеш или UUIDv5:
```python
import uuid
ids = [str(uuid.uuid5(uuid.NAMESPACE_URL, p.get('source_file', '') + p.get('text', ''))) for p in payloads]
```

## 2. Дефекты высокой критичности (HIGH)

### 2.1. Out of Memory (OOM) при индексации крупных проектов
**Локация:** `src/services.py` (метод `_process_files_in_batches`)
**Описание:** Метод разбивает файлы на чанки и кодирует их батчами по 32 файла. Однако вместо того, чтобы сразу отправлять обработанный батч в Qdrant и очищать память, метод **аккумулирует** все векторы и пейлоады в массивы `all_embeddings` и `all_payloads`, возвращая их целиком в `run_indexing`.
**Последствия:** Для проекта из 5000 файлов этот метод попытается удержать в оперативной памяти сотни тысяч векторов `float16` и строк перед тем, как отдать их в базу. Это гарантированно "положит" сервер.
**Как исправить:** Изменить метод на генератор (`yield`) или передавать ссылку на `vector_store` прямо в `_process_files_in_batches`, чтобы делать `upsert` внутри цикла каждые N батчей, освобождая память.

### 2.2. Data Race в фоновых задачах (Concurrency)
**Локация:** `src/main.py` (метод `index_project_endpoint`)
**Описание:** Эндпоинт добавляет `run_indexing_task` в `BackgroundTasks`. Нет никаких блокировок на уровне проекта.
**Последствия:** Если пользователь быстро нажмет кнопку "Index" дважды, запустятся две параллельные горутины (потоки), которые будут одновременно читать и писать в одну и ту же коллекцию Qdrant. Это приведет к Race Condition, падениям Qdrant (из-за конфликта версий шардов) и "грязному" стейту индекса.
**Как исправить:** Внедрить `asyncio.Lock` по ключу `project_name` или хранить статус блокировки в словаре `TASK_STATUS` (отклонять новые запросы с 409 Conflict, если статус `running`).

## 3. Дефекты средней и низкой критичности (MEDIUM / LOW)

### 3.1. [MEDIUM] DoS-вектор в алгоритме кластеризации
**Локация:** `src/main.py` (метод `perform_clustering`)
**Описание:** Метод запрашивает векторы из БД через `scroll` с жестким лимитом `limit=10000` (что обрезает данные для больших проектов). Затем он загружает их **все в RAM** для KMeans.
**Последствия:** Блокировка пула потоков FastAPI тяжелой математикой (sklearn KMeans) и высокое потребление памяти на каждый вызов эндпоинта `/clusters`. Нет кэширования результатов кластеризации.

### 3.2. [MEDIUM] Логическая ошибка в отображении прогресса CLI
**Локация:** `manage.py` (команда `api progress`)
**Описание:** Логика конвертации прогресса содержит фатальную ошибку:
`display_percent = progress if progress > 1 else progress * 100`
Если API возвращает строку `"0.50%"` (что означает полпроцента), `manage.py` вырежет `%`, получит float `0.5`. Так как `0.5` не больше 1, он умножит его на 100, и выведет `50%`. Это ломает UX.

### 3.3. [LOW] Рассинхронизация форматов контрактов (Мертвый код)
**Локация:** `src/index.py`
**Описание:** В проекте есть скрипт `src/index.py`, который дублирует логику `src/services.py`, но делает это хуже. Важно: `src/index.py` **не добавляет** поле `file_type` в payload Qdrant.
Если кто-то проиндексирует проект через `src/index.py`, а затем попытается искать через API (с фильтром `scope=code`), он получит пустой результат (Filter Mismatch), так как в базе не будет нужных ключей. Скрипт необходимо удалить как technical debt.

### 3.4. [LOW] Микс локальных и облачных конфигураций Qdrant
**Локация:** `debug_qdrant.py` vs `src/config.py`
`debug_qdrant.py` пытается использовать локальную базу `path="./test_qdrant"`, в то время как весь код завязан на gRPC/REST URL инстанс (Docker контейнер). Это вводит в заблуждение разработчиков.

## 4. Итоговые рекомендации

1. **Безопасность (Блокер):** Немедленно добавить проверку `resolve().is_relative_to(...)` для `project_name`.
2. **Инфраструктура:** Заменить `hash()` на `uuid.uuid5()`.
3. **Память:** Переписать пайплайн `_process_files_in_batches` на потоковую (streaming) обработку с вызовом `upsert` внутри цикла.
4. **Конкурентность:** Ввести механизм мьютексов (locks) для `run_indexing_task`, чтобы предотвратить параллельное мутирование одного индекса.
