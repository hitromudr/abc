# Аудит проекта `isearch` - Отчёт

## Обзор проекта
Проект `isearch` представляет собой поисковую систему с семантическим поиском и анализом графа зависимостей для кодовых баз. Система включает индексацию файлов, векторный поиск через Qdrant, анализ зависимостей и API на FastAPI.

## Классификация дефектов

### 1. Архитектурные проблемы (Критичность: Высокая)

#### 1.1. Несогласованность конфигурации путей
- **Файлы**: `src/config.py`, `src/main.py`, `src/index.py`
- **Описание**: В проекте используется несколько различных способов определения путей к проектам:
  - `src/config.py`: `get_projects_base_dir()` возвращает `Path(os.getenv("PROJECTS_BASE_DIR", ".."))`
  - `src/index.py`: Жёстко закодированный путь `PROJECT_ROOT_PATH = Path(__file__).parent.parent.parent / "autowarp"`
  - `src/main.py`: Использует зависимость от `get_projects_base_dir()`
- **Риск**: Несогласованность может привести к ошибкам при развёртывании и тестировании.
- **Рекомендация**: Унифицировать подход к определению путей через конфигурацию окружения.

#### 1.2. Отсутствие стратегии обработки ошибок в фоновых задачах
- **Файл**: `src/main.py`
- **Описание**: Фоновые задачи индексации (`run_indexing_task`) не имеют механизма восстановления после сбоев. При падении задачи состояние остаётся в `TASK_STATUS` без очистки.
- **Риск**: Накопление "зомби" задач, утечка памяти.
- **Рекомендация**: Реализовать механизм очистки устаревших задач, добавить повторные попытки для временных сбоев.

#### 1.3. Прямая зависимость от глобальных переменных
- **Файл**: `src/main.py`
- **Описание**: Компоненты `embedder` и `vector_store` инициализируются как глобальные переменные, что усложняет тестирование и может привести к состоянию гонки.
- **Риск**: Проблемы с параллельным выполнением, сложность модульного тестирования.
- **Рекомендация**: Использовать dependency injection через FastAPI Depends или фабрики.

### 2. Проблемы безопасности (Критичность: Средняя)

#### 2.1. Отсутствие валидации путей в API
- **Файлы**: `src/main.py`, `src/graph_builder.py`
- **Описание**: API принимает параметры `start_path` и `project_name` без достаточной валидации, что может привести к path traversal атакам.
- **Пример**: В `build_graph` используется `(project_root / start_path).resolve()` без проверки, что результат остаётся внутри `project_root`.
- **Риск**: Доступ к файлам за пределами целевого проекта.
- **Рекомендация**: Добавить проверку `is_relative_to(project_root)` после `resolve()`.

#### 2.2. Уязвимость в парсинге Markdown
- **Файл**: `src/graph_builder.py`
- **Описание**: В функции `parse_markdown_file` есть комментарий о фиксации абсолютных путей, но код продолжает обрабатывать относительные пути без достаточной санитизации.
- **Риск**: Path traversal через специально сформированные ссылки в документации.
- **Рекомендация**: Реализовать строгую проверку всех разрешённых путей.

### 3. Проблемы производительности (Критичность: Средняя)

#### 3.1. Неоптимальное использование памяти при индексации
- **Файл**: `src/index.py`
- **Описание**: Функция `load_documents` загружает все файлы в память перед обработкой, что может привести к OOM на больших проектах.
- **Риск**: Падение при индексации крупных репозиториев.
- **Рекомендация**: Использовать потоковую обработку или батчинг с освобождением памяти.

#### 3.2. Отсутствие кэширования эмбеддингов
- **Файл**: `src/embedder.py`
- **Описание**: Модель эмбеддингов генерирует векторы для одних и тех же запросов повторно.
- **Риск**: Избыточная нагрузка на GPU/CPU при повторяющихся запросах.
- **Рекомендация**: Реализовать LRU-кэш для часто запрашиваемых фраз.

#### 3.3. Потенциальные deadlocks в управлении процессами
- **Файл**: `manage.py`
- **Описание**: В функции `service_up` используется `nohup` и `&` для демонизации, что может привести к "осиротевшим" процессам при быстром завершении.
- **Риск**: Накопление фоновых процессов, конфликты портов.
- **Рекомендация**: Использовать более надёжные механизмы управления процессами (systemd, supervisor).

### 4. Проблемы надёжности (Критичность: Высокая)

#### 4.1. Ненадёжное определение работающего API
- **Файл**: `manage.py`
- **Описание**: Функция `is_api_running` проверяет PID и порт, но не проверяет работоспособность эндпоинтов здоровья.
- **Риск**: CLI может считать API работающим, когда тот фактически не отвечает.
- **Рекомендация**: Добавить проверку HTTP-запросом к `/system/health`.

#### 4.2. Отсутствие транзакционности в операциях с Qdrant
- **Файлы**: `src/vector_store.py`, `src/services.py` (отсутствует в предоставленном коде)
- **Описание**: Операции удаления и вставки точек выполняются отдельно, что может привести к inconsistent state при сбоях.
- **Риск**: Потеря данных или дублирование записей при прерывании индексации.
- **Рекомендация**: Реализовать паттерн "compare-and-swap" или использовать транзакционные возможности Qdrant.

#### 4.3. Хрупкий механизм отслеживания задач
- **Файл**: `src/main.py`
- **Описание**: `TASK_STATUS` хранится в памяти, что приводит к потере состояния при перезапуске приложения.
- **Риск**: Пользователи теряют возможность отслеживать долгие задачи индексации.
- **Рекомендация**: Сохранять состояние задач в persistent storage (Redis, база данных).

### 5. Проблемы кода и поддерживаемости (Критичность: Низкая)

#### 5.1. Дублирование логики chunking
- **Файл**: `src/chunker.py`
- **Описание**: Несколько стратегий chunking (Python, Markdown, brace-based) имеют перекрывающуюся логику.
- **Риск**: Усложнение поддержки, возможные расхождения в поведении.
- **Рекомендация**: Выделить общую абстракцию для chunking с плагинами для разных типов файлов.

#### 5.2. Смешение ответственности в main.py
- **Файл**: `src/main.py`
- **Описание**: Файл содержит логику API, поиска, кластеризации и управления задачами (1000+ строк).
- **Риск**: Сложность тестирования и понимания кода.
- **Рекомендация**: Разделить на отдельные модули (search_service, clustering_service, task_service).

#### 5.3. Жёсткая привязка к конкретной модели эмбеддингов
- **Файл**: `src/config.py`
- **Описание**: `MODEL_NAME = "intfloat/multilingual-e5-large"` захардкожена.
- **Риск**: Сложность смены модели, тестирования альтернатив.
- **Рекомендация**: Сделать модель конфигурируемой через environment variables.

### 6. Инфраструктурные проблемы (Критичность: Средняя)

#### 6.1. Отсутствие мониторинга и метрик
- **Описание**: Нет сбора метрик производительности, ошибок, использования ресурсов.
- **Риск**: Сложность диагностики проблем в production.
- **Рекомендация**: Интегрировать Prometheus метрики, логирование structured JSON.

#### 6.2. Потенциальные проблемы с версиями зависимостей
- **Файл**: `pyproject.toml`
- **Описание**: Жёсткая привязка к конкретным версиям `transformers==4.38.2` и `huggingface-hub==0.22.2`.
- **Риск**: Конфликты с другими зависимостями, проблемы безопасности в зафиксированных версиях.
- **Рекомендация**: Использовать диапазоны версий с регулярным обновлением.

#### 6.3. Отсутствие конфигурации для production
- **Описание**: Нет примеров конфигурации для развёртывания (Dockerfile, k8s manifests, production .env).
- **Риск**: Сложность deployment, возможные security misconfigurations.
- **Рекомендация**: Предоставить примеры production-конфигураций.

### 7. Пограничные случаи (Критичность: Средняя)

#### 7.1. Обработка бинарных и больших файлов
- **Файл**: `src/index.py`
- **Описание**: `ALLOWED_EXTENSIONS` включает `.json`, `.yml` и другие, но нет проверки на размер файла.
- **Риск**: Попытка обработки multi-GB JSON файлов может привести к OOM.
- **Рекомендация**: Добавить лимит на размер файла, пропускать бинарные файлы.

#### 7.2. Проблемы с кодировками
- **Файлы**: `src/index.py`, `src/graph_builder.py`
- **Описание**: Используется `errors="ignore"` при чтении файлов, что может привести к потере информации.
- **Риск**: Некорректная обработка файлов с не-UTF8 кодировками.
- **Рекомендация**: Определять кодировку автоматически или предоставлять конфигурацию.

#### 7.3. Рекурсивные симлинки
- **Файл**: `src/index.py`
- **Описание**: `root_path.rglob("*")` может попасть в бесконечный цикл при наличии рекурсивных симлинков.
- **Риск**: Зависание процесса индексации.
- **Рекомендация**: Добавить проверку на симлинки и максимальную глубину.

## Рекомендации по приоритизации

1. **Немедленно исправить** (Критичность: Высокая):
   - Path traversal уязвимости в API
   - Ненадёжное управление фоновыми процессами
   - Отсутствие persistent storage для состояния задач

2. **Исправить в ближайшем релизе** (Критичность: Средняя):
   - Проблемы с памятью при индексации
   - Добавление валидации всех входных данных
   - Реализация мониторинга и метрик

3. **Улучшить в долгосрочной перспективе** (Критичность: Низкая):
   - Рефакторинг архитектуры
   - Улучшение тестируемости
   - Документация production-развёртывания

## Заключение

Проект `isearch` имеет прочную основу с хорошо реализованным core functionality (семантический поиск, анализ графов). Однако присутствуют значительные архитектурные и security проблемы, требующие внимания перед production-развёртыванием. Наиболее критичные issues связаны с безопасностью и надёжностью системы.

Рекомендуется начать с исправления security vulnerabilities и улучшения управления состоянием системы, затем перейти к оптимизации производительности и улучшению архитектуры.
