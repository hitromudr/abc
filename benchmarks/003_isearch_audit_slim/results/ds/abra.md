# Концептуальный Протокол: Аудит семантической поисковой системы с векторной БД

## 1. Топология задачи (Ontology)
- **Мета-тип:** Аудит распределённой системы с stateful хранилищем (векторная БД) и тяжёлыми ML-моделями.
- **Ключевой конфликт:** Между производительностью (инференс на GPU, низкая задержка поиска) и надёжностью/безопасностью (консистентность индекса, контроль доступа, обработка сбоев).

## 2. Инварианты (Invariants)
1. **Закон сохранения консистентности:** Состояние индекса (файл → хэш) и векторы в Qdrant должны быть синхронизированы. Нарушение → коррупция результатов поиска.
2. **Предел ресурсов:** Модель embedder ограничена памятью GPU/CPU. Превышение → OOM, падение сервиса.
3. **Принцип минимальных привилегий:** Векторная БД и API не должны быть открыты без аутентификации в production.

## 3. Точка опоры (Leverage Point)
- **Фокус внимания:** Управление состоянием индекса и обработка ошибок в цепочке: чтение файлов → chunking → embedding → upsert в Qdrant.
- **Игнорируемый шум:** Микрооптимизации кода (например, скорость chunking) при наличии системных рисков (например, отсутствие retry при сетевых сбоях).

## 4. Векторы энтропии (Degradation Paths)
- **Анти-паттерн А:** Тихий сбой индексации (например, Qdrant недоступен, но задача помечена как completed из-за отсутствия проверки).
- **Когнитивное искажение Б:** Предположение, что локальная файловая система и Qdrant всегда надёжны, что приводит к отсутствию механизмов восстановления.

## 5. Алгоритм стабилизации (The Protocol)
- **Фаза 0: Изоляция.** Ограничить зону поражения: добавить circuit breaker для вызовов к Qdrant, лимиты на размер батчей embedder.
- **Фаза 1: Оцифровка базиса.** Ввести метрики: latency поиска, размер индекса, количество ошибок индексации, доступность Qdrant.
- **Фаза 2: Применение рычага.** Внедрить idempotent-операции индексации с проверкой консистентности (например, checksum векторов). Добавить аутентификацию/авторизацию для Qdrant.
- **Фаза 3: Интеграция и Стабилизация.** Настроить алерты на расхождение состояния, автоматическое перестроение индекса при обнаружении коррупции.

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** Система возвращает релевантные результаты поиска при 99.9% доступности, индекс консистентен, нет утечек данных.
- **Ложный сигнал успеха:** Поиск работает, но индекс постепенно расходится (старые файлы не удаляются, дублируются чанки).

## 7. Эвристики и Мета-Контекст
- **Открытые вопросы:** Как обрабатывать обновление модели embedder без полной переиндексации? Как масштабировать Qdrant?
- **Тактические предложения:**
  1. Добавить health check для Qdrant в `/system/health`.
  2. Валидировать входные параметры (project_name, пути) для предотвращения traversal.
  3. Добавить timeout и retry для операций Qdrant.
  4. Логировать ошибки embedding с stack trace.
- **Комментарии оператора:** Код хорошо структурирован, но lacks production hardening.

## 8. Резолюция и Архитектурная Развилка (Approval Gate)
- **Фундаментальный диагноз:** Система обладает функциональной полнотой, но критически недостаёт production-ready надёжности, безопасности и наблюдаемости.
- **Стратегические варианты (The Forks):**
  - **Вариант А (Incremental Hardening):** Постепенно добавлять исправления по мере выявления проблем. Плюсы: низкий риск, можно внедрять параллельно с разработкой. Минусы: временные окна уязвимости остаются. Цена: средняя (недели работы).
  - **Вариант Б (Security & Reliability Sprint):** Выделить спринт на закрытие всех critical issues (аутентификация Qdrant, консистентность индекса, обработка ошибок). Плюсы: быстрое приведение системы в production-ready состояние. Минусы: требует остановки разработки фич. Цена: высокая (2-3 недели).
- **Рекомендация ИИ:** Вариант Б, так как текущие уязвимости (открытый Qdrant, отсутствие обработки ошибок) могут привести к инцидентам с высоким impact.
- **Ближайшие физические шаги (Next Steps):**
  1. **[APPROVAL GATE]** Ожидание решения оператора (Выбор Варианта А или Б).
  2. **[ЗАПРОС]** Если оператор выбирает вариант, ИИ может сгенерировать детализированный план исправлений (например, список конкретных issues в коде).
