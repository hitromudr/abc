# Концептуальный Протокол: Аудит инфраструктуры поиска (isearch)

## 1. Топология задачи (Ontology)
- **Мета-тип:** Анализ надежности распределенных систем с векторным хранилищем.
- **Ключевой конфликт:** Детерминизм обработки данных vs динамическая природа Python-среды.

## 2. Инварианты (Invariants)
1. **Закон детерминизма идентификаторов:** ID объектов в БД обязаны быть стабильными между перезапусками процесса.
2. **Принцип атомарности индексации:** Операция записи в БД обязана завершаться проверкой статуса ответа сервера.

## 3. Точка опоры (Leverage Point)
- **Фокус:** Замена `hash()` на стабильную хеш-функцию (`hashlib.sha256`).
- **Шум:** Поиск "улучшений" в модели, пока инфраструктура записи нестабильна.

## 4. Векторы энтропии (Degradation Paths)
- **Анти-паттерн А:** Использование `abs(hash(...))` как ключа для `qdrant`.
- **Когнитивное искажение Б:** Предположение, что `Python` сохраняет хеш-значения между сессиями.

## 5. Алгоритм стабилизации (The Protocol)
- **Фаза 0: Изоляция.** Фиксация идентификаторов.
- **Фаза 1: Оцифровка базиса.** Внедрение хеширования контента.
- **Фаза 2: Применение рычага.** Блокировки потоков при записи.

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** Повторный `index` того же проекта не создает дубликатов в `Qdrant`.
- **Ложный сигнал успеха:** Индексация "проходит без ошибок", но часть данных пропадает (silent loss).

## 7. Эвристики и Мета-Контекст
- **Открытые вопросы:** Нужна ли глобальная блокировка на уровне `indexing_service` при параллельных запросах?
- **Тактические предложения:** Замена `hash()` на `hashlib.sha256(...).hexdigest()`.

## 8. Резолюция
- **Фундаментальный диагноз:** Инфраструктура записи данных недетерминирована, что ведет к потере консистентности индекса при каждом перезапуске.
- **Стратегические варианты:**
  - **Вариант А (Быстрый):** Замена `hash()` на `hashlib.sha256` (рекомендуется).
  - **Вариант Б (Архитектурный):** Внедрение версионности индекса (snapshotting).
- **Рекомендация ИИ:** Вариант А — это blocker для любой стабильной работы системы.

# Отчёт об аудите (abra.md)

### 1. Критические дефекты (Critical)
1. **GT-3: Детерминизм идентификаторов (Qdrant).** 
   - `src/vector_store.py`: `abs(hash(...))` — использование встроенного `hash()` недопустимо, так как оно рандомизируется.
   - **Root Cause:** Недетерминированные ID вызывают дублирование точек при каждом повторном запуске индексации.
   - **Fix:** `import hashlib`; использовать `hashlib.sha256(data.encode()).hexdigest()` и конвертировать в `int` для `Qdrant`.

2. **GT-4: Race Condition при индексации.**
   - При вызове `/index` из нескольких потоков/запросов `vector_store.upsert` выполняется без блокировок.
   - **Fix:** Реализовать `FileLock` или `threading.Lock` на уровне `IndexingService`.

### 2. Высокоприоритетные дефекты (High)
1. **Ошибка логирования в `manage.py`.**
   - Процесс `nohup` может умереть, и скрипт `manage.py` продолжает считать, что API работает.
   - **Fix:** Добавить проверку `psutil.pid_exists` в цикле мониторинга.

2. **Отсутствие проверки ответа Qdrant `upsert`.**
   - Метод `upsert` не проверяет, были ли ошибки при вставке батча.
   - **Fix:** Проверять `update_result.status` в Qdrant.

### 3. Инфраструктурные риски (Medium)
1. **Embedder performance.**
   - `batch_size=8` в `src/embedder.py` жестко задан. На GPU это слишком мало, на CPU может быть ОК, но не учитывает VRAM.
   - **Fix:** Сделать `batch_size` конфигурационным параметром или динамически вычисляемым.
