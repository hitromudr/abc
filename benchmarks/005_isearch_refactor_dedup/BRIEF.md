# Bench 005: Refactor — Консолидация дублированной логики файловой обработки

## Контекст

Проект `isearch` — поисковый движок с индексацией файлов и векторным поиском. В кодовой базе 3 независимые реализации одной и той же логики загрузки `.gitignore` и классификации типов файлов.

## Проблема

Три файла содержат дублированную логику:

1. **`src/services.py`** — `IndexingService._load_gitignore_spec()` (строки ~135–152), классификация `code_extensions` / `docs_extensions` (строки ~62–64), обработка файлов `_process_files_in_batches()` с inline чанкингом
2. **`src/graph_builder.py`** — inline загрузка `.gitignore` (строки ~159–166) при построении графа зависимостей
3. **`src/index.py`** — `load_gitignore_spec()` (строки ~21–39), `ALLOWED_EXTENSIONS` (строки ~15–18), `load_documents()` с альтернативной стратегией чанкинга

Три разных реализации `.gitignore` с немного разным поведением. Два набора расширений файлов. Два алгоритма чанкинга. Изменение в одном месте не распространяется на остальные.

## Задание

Выполни рефакторинг: создай единый модуль `src/file_utils.py` (или аналогичный) который консолидирует:

1. **Загрузку .gitignore** — одна функция `load_gitignore_spec(root_path)`, используемая всеми тремя модулями
2. **Классификацию типов файлов** — единый набор расширений и функция `get_file_type(path) -> "code" | "docs" | "other"`
3. **Сканирование файлов** — единая функция для обхода директории с учётом `.gitignore` и допустимых расширений

Требования:
- **Все существующие тесты должны проходить** (`pytest tests/`)
- **Публичный API модулей не должен измениться** — `IndexingService`, `build_graph()`, `load_documents()` сохраняют свои сигнатуры
- **Количество реализаций .gitignore: 3 → 1**
- **Количество реализаций file type classification: 2 → 1**

Выдай результат как unified diff (`\`\`\`diff ... \`\`\``) с изменениями всех затронутых файлов и новым файлом `src/file_utils.py`.
