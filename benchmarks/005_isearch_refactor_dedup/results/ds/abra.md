# Концептуальный Протокол: Консолидация дублированной логики файловой обработки

## 1. Топология задачи (Ontology)
- **Мета-тип:** Устранение дублирования кода в модульной системе
- **Ключевой конфликт:** Независимая эволюция одинаковых реализаций vs необходимость единой точки истины

## 2. Инварианты (Invariants)
1. **Закон сохранения публичного API:** Сигнатуры `IndexingService`, `build_graph()`, `load_documents()` не должны измениться
2. **Предел обратной совместимости:** Все существующие тесты должны проходить
3. **Принцип единственной ответственности:** Каждая функция должна иметь одну четкую цель

## 3. Точка опоры (Leverage Point)
- **Фокус внимания:** Создание единого модуля `file_utils.py` как источника истины для:
  - Загрузки .gitignore
  - Классификации типов файлов
  - Сканирования файлов с фильтрацией
- **Игнорируемый шум:** Внутренние различия в деталях реализации между тремя дубликатами

## 4. Векторы энтропии (Degradation Paths)
- **Анти-паттерн А:** Создание ещё одного дубликата вместо консолидации
- **Когнитивное искажение Б:** Изменение публичного API "для упрощения"

## 5. Алгоритм стабилизации (The Protocol)
- **Фаза 0: Изоляция.** Создать `src/file_utils.py` с чистыми функциями
- **Фаза 1: Оцифровка базиса.** Определить единые наборы расширений для code/docs
- **Фаза 2: Применение рычага.** Заменить вызовы в трёх модулях на импорт из file_utils
- **Фаза 3: Интеграция и Стабилизация.** Запустить тесты, убедиться в обратной совместимости

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** 
  - Количество реализаций .gitignore: 3 → 1
  - Количество реализаций file type classification: 2 → 1
  - Все тесты проходят
- **Ложный сигнал успеха:** Создание нового модуля без реального удаления дубликатов

## 7. Эвристики и Мета-Контекст
- **Открытые вопросы:** Нужно ли кэшировать загруженный .gitignore для производительности?
- **Тактические предложения:** Использовать `functools.lru_cache` для `load_gitignore_spec`
- **Комментарии оператора:** Проект использует `pathspec` для .gitignore, что упрощает унификацию

## 8. Резолюция и Архитектурная Развилка (Approval Gate)
- **Фундаментальный диагноз:** Три независимые реализации одной логики обработки файлов создают риск рассинхронизации и усложняют поддержку.

- **Стратегические варианты (The Forks):**
  - **Вариант А: Полная консолидация**
    - Плюсы: Максимальное устранение дублирования, единая точка истины
    - Минусы: Требует изменения трёх файлов, риск сломать что-то
    - Цена внедрения: Средняя (несколько часов на реализацию и тестирование)
  
  - **Вариант Б: Частичная консолидация (только .gitignore)**
    - Плюсы: Меньший риск, быстрее реализовать
    - Минусы: Оставляет дублирование классификации файлов и сканирования
    - Цена внедрения: Низкая (1-2 часа)

- **Рекомендация ИИ:** Вариант А, так как он полностью решает проблему, а не откладывает её. Риск управляем через существующие тесты.

- **Ближайшие физические шаги (Next Steps):**
  1. **[APPROVAL GATE]** Ожидание решения оператора (Выбор Варианта А или Б).
  2. **[ЗАПРЕТ]** ИИ не имеет права генерировать итоговый рабочий код до явного подтверждения выбора оператором.
