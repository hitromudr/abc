- Факт: тест проверяет стабильность ID при одинаковом input.

L3 Иммунитет: sha256 криптографически стойкий, коллизии крайне маловероятны.

# Концептуальный Протокол: Генерация детерминистичных идентификаторов из строковых данных

## 1. Топология задачи (Ontology)
- **Мета-тип:** Детерминистичное отображение строки в ограниченное числовое пространство
- **Ключевой конфликт:** Требование стабильности (одинаковый результат при одинаковом входе) vs. безопасность (рандомизация для предотвращения атак)

## 2. Инварианты (Invariants)
1. **Закон сохранения уникальности:** Разные входные строки должны с высокой вероятностью давать разные ID
2. **Предел диапазона:** ID должен быть положительным 64-битным целым (0..2^63-1)
3. **Принцип детерминизма:** Одинаковый вход → одинаковый выход независимо от времени, процесса, окружения

## 3. Точка опоры (Leverage Point)
- **Фокус внимания:** Алгоритм хеширования, гарантирующий детерминизм
- **Игнорируемый шум:** Абсолютная криптографическая стойкость (не требуется для данной задачи)

## 4. Векторы энтропии (Degradation Paths)
- **Анти-паттерн А:** Использование недетерминистичных функций (Python hash(), random)
- **Когнитивное искажение Б:** Преждевременная оптимизация (выбор слишком слабого хеша для экономии CPU)

## 5. Алгоритм стабилизации (The Protocol)
- **Фаза 0: Изоляция.** Определить точку генерации ID (метод `upsert` в `vector_store.py`)
- **Фаза 1: Оцифровка базиса.** Выбрать криптографический хеш (SHA-256), гарантирующий детерминизм
- **Фаза 2: Применение рычага.** Заменить `abs(hash(...))` на `int.from_bytes(sha256(...).digest()[:8], 'big') & 0x7FFFFFFFFFFFFFFF`
- **Фаза 3: Интеграция и Стабилизация.** Добавить импорт hashlib, обновить тест

## 6. Метрика истины (Truth Criteria)
- **Состояние успеха:** Тест `test_id_determinism_across_seeds` проходит при любом PYTHONHASHSEED
- **Ложный сигнал успеха:** Использование слабого хеша (CRC32), который технически детерминистичен, но имеет высокий риск коллизий

## 7. Эвристики и Мета-Контекст
- **Открытые вопросы:** Нужно ли сохранять обратную совместимость с существующими ID? (Нет — при reindex пересоздаются)
- **Тактические предложения:** Добавить комментарий о выборе SHA-256 и ограничении диапазона
- **Комментарии оператора:** Задача технически простая, но критичная для корректности системы

## 8. Резолюция и Архитектурная Развилка (Approval Gate)
- **Фундаментальный диагноз:** Использование рандомизированного `hash()` для генерации ID приводит к недетерминизму и дублированию данных

- **Стратегические варианты (The Forks):**
  - **Вариант А: SHA-256 (рекомендуется)**
    - Плюсы: Криптографически стойкий, гарантированно детерминистичный, минимальный риск коллизий
    - Минусы: Небольшой overhead (~0.5µs на хеш)
    - Цена внедрения: 3 строки кода (импорт + замена)
  - **Вариант Б: xxhash (альтернатива)**
    - Плюсы: Быстрее SHA-256, всё ещё детерминистичный
    - Минусы: Требует установку дополнительной зависимости, менее стандартный
    - Цена внедрения: Установка пакета + изменение кода

- **Рекомендация ИИ:** Вариант А (SHA-256) — стандартная библиотека, достаточная скорость, гарантированный детерминизм

- **Ближайшие физические шаги (Next Steps):**
  1. **[APPROVAL GATE]** Ожидание решения оператора (Выбор Варианта А или Б)
  2. **[ЗАПРЕТ]** ИИ не имеет права генерировать итоговый рабочий код до явного подтверждения выбора оператором
