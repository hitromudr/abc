# Эмпирические Бенчмарки (Eval Suite)

Контролируемые эксперименты: vanilla LLM vs abracadabra на одном и том же коде.

## Структура эксперимента

```
benchmarks/NNN_название/
├── BRIEF.md        ← задание + условия эксперимента
├── meta.yml        ← модель, версия, Ground Truth
├── baseline.md     ← результат vanilla-агента
├── abra.md         ← результат abra-агента
└── verdict.md      ← арбитраж
```

## Запуск

### Подготовка

Зафиксировать SHA целевого проекта в `meta.yml` → `target_commit_sha`. Оба прогона делаются на одном коммите.

---

### Сессия 1: Baseline

**Цель:** получить аудит от чистой LLM без abracadabra.

**Где:** Любая среда, где правила abracadabra **не загружены**:
- Claude Code / Cursor / Zed — временно убрать ссылку на abracadabra из `.rules` / `.cursorrules` / `CLAUDE.md` (субмодуль остаётся, файлы доступны для чтения)
- Веб-чат (Claude.ai / ChatGPT / AI Studio) — прикрепить BRIEF.md как файл

**Что пишет оператор:**

IDE (файлы доступны):
```
Прочитай .abracadabra/benchmarks/001_isearch_audit/BRIEF.md и выполни задачу.
```

Веб-чат (BRIEF.md прикреплён):
```
Выполни задачу из прикреплённого файла.
```

Задание берётся из того же BRIEF.md, что и для abra — идентичный вход.

**Что делает агент:**
1. Читает BRIEF.md → получает задачу
2. Читает кодовую базу (самостоятельно выбирает файлы)
3. Анализирует без какой-либо методологии — как умеет
4. Выдаёт отчёт

**Сохранение:** Скопировать вывод агента → `benchmarks/NNN_*/baseline.md` в репо abracadabra.

---

### Сессия 2: Abra

**Цель:** получить аудит от LLM, работающей через конвейер abracadabra.

**Где:** Целевой проект с подключённым субмодулем `.abracadabra`. Чистая сессия.

**Что пишет оператор:**

```
abra init
```

Агент загружает базу знаний (sync_context.sh → параллельное чтение файлов). Отвечает: «Ядро загружено. Жду задачу.» Стоп.

```
abra audit 001
```

**Что делает агент:**
1. Находит `.abracadabra/benchmarks/001_*/BRIEF.md` (или `benchmarks/001_*/BRIEF.md` если abracadabra корневой)
2. Читает задание из BRIEF.md
3. Запускает полный конвейер: Пре-чеклист (8 фильтров) → Фаза 0 (заземление) → Фаза 0.5 (ROI) → Фазы 1–4 (анализ) → Фаза 5 (синтез + Октагон) → Фаза 6 (патч процесса)
4. Выдаёт Концептуальный Протокол по шаблону

**Сохранение:** Скопировать вывод агента → `benchmarks/NNN_*/abra.md`.

---

### Сессия 3: Verdict

**Цель:** независимое сравнение обоих результатов с объективной реальностью.

**Где:** Любая сессия (веб-чат или IDE). Нужен доступ к 3 файлам: `baseline.md`, `abra.md`, `meta.yml`.

**Что пишет оператор:**

```
abra verdict 001
```

Или, если abracadabra не загружена — передать 3 файла и попросить:

```
Сравни два аудита одного проекта. Ground Truth (реальные баги) в meta.yml.
Для каждого GT-бага: нашёл baseline? нашёл abra? Посчитай Recall.
Отметь extras (валидные баги сверх GT) и галлюцинации. Вердикт.
```

**Что делает агент:**
1. Читает `meta.yml` → извлекает Ground Truth (список подтверждённых багов)
2. Читает `baseline.md` и `abra.md`
3. Для каждого GT-бага определяет: найден / не найден / частично
4. Считает Recall = найдено / всего GT
5. Считает extras (реальные баги, не вошедшие в GT) и false positives (галлюцинации)
6. Сравнивает структуру, глубину, actionability рекомендаций
7. Выносит вердикт

**Сохранение:** Вывод → `verdict.md`. Метрики (recall, extras, winner) → обновить `meta.yml`.
