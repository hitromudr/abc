**Дата:** 2026-03-12
**Метод:** Полный анализ кодовой базы (чистая LLM, без фреймворков)
**Версия проекта:** 2.0.0 (pyproject.toml), API version 1.8.0 (main.py)

## 1. КРИТИЧЕСКИЕ ДЕФЕКТЫ (P0 - Требуют немедленного исправления)

### 1.1. Коллизии хешей точек в Qdrant (Data Corruption)

**Файл:** `src/vector_store.py`, строки 110-112
**Проблема:** ID точек вычисляются через `abs(hash(source_file + text))`. Python `hash()` -- не криптографическая функция, дает 64-битный результат. `abs()` уменьшает пространство вдвое. При большом количестве чанков (десятки тысяч) коллизии статистически неизбежны. Два разных чанка с одинаковым хешем приведут к тому, что один перезапишет другой -- **потеря данных без уведомления**.

Дополнительно: `hash()` в Python нестабилен между процессами (из-за PYTHONHASHSEED). При перезапуске сервера те же данные получат другие ID, что приведет к дублированию точек в базе.

**Критичность:** КРИТИЧЕСКАЯ. Неидемпотентное поведение, потеря данных.

### 1.2. Дублирование PROJECTS_BASE_DIR в `.env`

**Файл:** `.env`
```
PROJECTS_BASE_DIR="/home/dms/work/"
QDRANT_URL=./qdrant_local_data
PROJECTS_BASE_DIR=..
```

**Проблема:** Переменная `PROJECTS_BASE_DIR` объявлена дважды с разными значениями. `python-dotenv` по умолчанию берет последнее значение (`..`), что означает, что первая строка -- мертвый код. Но поведение зависит от параметра `override` и порядка загрузки. Это источник непредсказуемого поведения при изменении `.env`.

**Критичность:** КРИТИЧЕСКАЯ. Конфигурация неоднозначна.

### 1.3. Отсутствие аутентификации и авторизации на API

**Файл:** `src/main.py`
**Проблема:** API полностью открыт. CORS настроен как `allow_origins=["*"]`. Любой может:
- Запустить индексацию (тяжелая GPU-операция) -- потенциальный DoS.
- Удалить любой индекс (`DELETE /projects/{project_name}/index`).
- Прочитать содержимое файлов через snippet extraction.

**Критичность:** КРИТИЧЕСКАЯ для production deployment. Для локальной разработки -- допустимо, но документировано должно быть.

### 1.4. Path Traversal через `project_name` и `directory`

**Файл:** `src/main.py`, строки 311, 434
**Проблема:** Параметр `project_name` передается без санитизации в `projects_base_dir / project_name`. Злоумышленник может отправить `project_name=../../etc` и получить доступ к произвольным файлам на диске через snippet extraction. Параметр `directory` в search (строка 434) также не валидируется -- `source_file.startswith(directory)` не защищает от `../` в directory.

**Критичность:** КРИТИЧЕСКАЯ. Уязвимость чтения произвольных файлов.

## 2. СЕРЬЕЗНЫЕ ДЕФЕКТЫ (P1 - Требуют исправления в ближайшем цикле)

### 2.1. Несогласованность типов файлов между chunker и search filter

**Файл:** `src/chunker.py` -- `detect_file_type()` возвращает: `code`, `test`, `doc`, `config`, `other`.
**Файл:** `src/main.py`, строка 418 -- search scope принимает: `code`, `docs`.
**Файл:** `src/services.py`, строка 82 -- разделяет на `code_extensions` и `docs_extensions`.

**Проблема:** Фильтр scope `docs` в search endpoint ищет `file_type == "docs"`, но chunker записывает в payload `file_type = "doc"` (без "s"). Поэтому фильтр `scope=docs` **никогда не вернет результаты** для файлов, которые chunker классифицировал как `doc`. Аналогично, файлы с типом `test`, `config`, `other` не фильтруются ни одним scope.

**Критичность:** СЕРЬЕЗНАЯ. Фильтрация по scope сломана. Поисковый scope "docs" не работает.

### 2.2. Scroll API ограничен лимитом 10000 точек при кластеризации

**Файл:** `src/main.py`, строка 527
```python
all_points = vector_store.client.scroll(
    ...
    limit=10000  # High limit to fetch all chunks
)[0]
```

**Проблема:** Для проектов с >10000 чанков кластеризация молча потеряет данные -- вернутся только первые 10000 точек. Нет пагинации.

**Критичность:** СЕРЬЕЗНАЯ. Неполные результаты кластеризации для крупных проектов.

### 2.3. `get_documents_by_source` и `delete_points_by_source` используют `should` с огромным числом условий

**Файл:** `src/vector_store.py`, строки 182-198, 215-238
**Проблема:** При инкрементальной индексации для каждого файла создается отдельный `FieldCondition` внутри `should`. При тысячах файлов фильтр `should` с тысячами условий может вызвать timeout или OOM на стороне Qdrant. Аналогичная проблема в `delete_points_by_source`.

**Критичность:** СЕРЬЕЗНАЯ. Производительность деградирует на крупных проектах.

### 2.4. Состояние задач хранится In-Memory (TASK_STATUS)

**Файл:** `src/main.py`, строка 107
```python
TASK_STATUS: Dict[str, Dict[str, Any]] = {}
```

**Проблема:**
- При перезапуске сервера все статусы задач теряются.
- Нет механизма очистки -- словарь растет бесконечно.
- Нет TTL, нет лимита на количество задач.
- Клиент, отправивший задачу и получивший task_id, теряет возможность отслеживать прогресс после рестарта.

**Критичность:** СЕРЬЕЗНАЯ. Memory leak, потеря состояния.

### 2.5. Race Condition при параллельной индексации одного проекта

**Файл:** `src/main.py`, строки 295-318, `src/services.py`
**Проблема:** Нет блокировки на индексацию проекта. Два запроса `POST /projects/X/index` запустят две параллельные фоновые задачи, которые будут одновременно писать в одну коллекцию и один state_file. Результат: corrupted state, дублирование/потеря данных.

**Критичность:** СЕРЬЕЗНАЯ. Data corruption при конкурентном доступе.

### 2.6. Устаревший standalone-скрипт `index.py` не работает

**Файл:** `src/index.py`
**Проблема:** Скрипт использует `from embedder import EmbeddingModel` (прямой импорт без пакета), вызывает `vector_store.recreate_collection(vector_size=embedding_dim)` без `collection_name`, и `vector_store.get_collection_info()` без аргументов. Также обращается к `vector_store.collection_name`, которого не существует. Метод `embedder.encode()` вызывается с аргументом `show_progress_bar`, отсутствующим в текущей сигнатуре. Скрипт полностью нерабочий.

**Критичность:** СЕРЬЕЗНАЯ. Мертвый код, вводит в заблуждение.

### 2.7. Устаревший standalone-скрипт `search.py` не работает

**Файл:** `src/search.py`
**Проблема:** Аналогично `index.py` -- использует устаревшие прямые импорты (`from embedder import ...`, `from vector_store import ...`). Вызывает `vector_store.search()` без `collection_name`. Не совместим с текущим API `VectorStore`.

**Критичность:** СЕРЬЕЗНАЯ. Мертвый код.

## 3. СРЕДНИЕ ДЕФЕКТЫ (P2 - Стоит исправить)

### 3.1. Несоответствие версий: pyproject.toml vs FastAPI app

**Файлы:** `pyproject.toml` (version `2.0.0`), `src/main.py` (version `1.8.0`)
**Проблема:** Версии проекта рассинхронизированы.

### 3.2. Boost-логика в search мутирует scores

**Файл:** `src/main.py`, строки 469-478
**Проблема:** Boost-фактор (`+0.1`) применяется к уже нормализованным scores, что может нарушить порядок сортировки. Бустится только **первый хит в группе**, но группа сортируется по первому хиту. Это создает неинтуитивное поведение: файл с одним хитом (score 0.75) бустится до 0.85, обгоняя файл с лучшим хитом (score 0.80).

### 3.3. `chunk_brace_based` не учитывает строки и комментарии

**Файл:** `src/chunker.py`, строки 176-214
**Проблема:** Подсчет фигурных скобок (`{`, `}`) ведется по всей строке, включая строковые литералы и комментарии. JSON-данные внутри кода, шаблонные строки с фигурными скобками, и комментарии с примерами кода будут ломать баланс скобок и порождать некорректные чанки.

### 3.4. `extract_snippet` читает весь файл в память

**Файл:** `src/main.py`, строки 147-171
**Проблема:** Для каждого хита поиска читается весь файл (`f.read().splitlines()`). При 50 результатах из крупных файлов это может быть дорого по памяти.

### 3.5. State file хранится в `qdrant_storage` внутри projects_base_dir

**Файл:** `src/services.py`, строки 285-290
```python
state_file = (
    self.projects_base_dir
    / "qdrant_storage"
    / f"{collection_name}_state.json"
)
```
**Проблема:** State file сохраняется в `projects_base_dir / qdrant_storage`, но `projects_base_dir` -- это директория с проектами, а не с данными Qdrant. Если `PROJECTS_BASE_DIR=..`, то state file будет записан в `../qdrant_storage/`, что может не существовать или быть совершенно другим каталогом. Кроме того, при использовании локального Qdrant (`./qdrant_local_data`) state хранится отдельно от данных БД, что создает риск рассинхронизации при очистке одного из них.

### 3.6. Docker-compose порт gRPC/REST перепутаны в комментариях

**Файл:** `docker-compose.yml`
```yaml
ports:
  - "6333:6333" # gRPC port
  - "6334:6334" # REST API port
```
**Проблема:** Порты Qdrant на самом деле: 6333 -- REST/HTTP, 6334 -- gRPC. Комментарии перепутаны.

### 3.7. Дублирование вызовов `clearProjectCache` в frontend

**Файл:** `static/app.js`, строки 511-512, 532-533
```javascript
clearProjectCache(projectName); // Invalidate cache before indexing
clearProjectCache(projectName); // Invalidate cache before indexing
```
**Проблема:** Дублирование вызова -- копипаст-баг. Присутствует и в `handleIndexProject`, и в `handleDeleteIndex`.

### 3.8. Файлы без расширения включаются как `docs`

**Файл:** `src/services.py`, строка 93
```python
self.docs_extensions = { "", ".md", ... }
```
**Проблема:** Пустая строка `""` в `docs_extensions` означает, что все файлы без расширения (`Makefile`, `Dockerfile`, `LICENSE`, бинарные файлы без расширения, etc.) будут индексированы. Бинарные файлы, прочитанные с `errors='ignore'`, создадут мусорные чанки в индексе.

### 3.9. E5 model требует prefix "query:" / "passage:" для оптимальной работы

**Файл:** `src/embedder.py`, `src/services.py`
**Проблема:** Модель `intfloat/multilingual-e5-large` требует префиксов `query: ` для запросов и `passage: ` для документов (согласно официальной документации модели). Без этих префиксов качество embeddings существенно снижается. В коде нигде не добавляются эти префиксы -- ни при индексации (passage), ни при поиске (query).

**Критичность:** СРЕДНЯЯ, но существенно влияет на качество поиска.

## 4. НИЗКИЕ ДЕФЕКТЫ (P3 - Техдолг)

### 4.1. Дублирование импортов в `main.py`

**Файл:** `src/main.py`
- `from .graph_analyzer import find_orphans` (строка 13) и `from src.graph_analyzer import find_orphans` (строка 89)
- `from .graph_builder import build_graph` (строка 14) и `from src.graph_builder import build_graph` (строка 90)

Относительные импорты на строках 13-14 избыточны, так как перезаписываются абсолютными импортами в try-блоке на строках 89-90.

### 4.2. Дублирование комментариев-разделителей

**Файл:** `src/main.py`, строки 619-622
```python
# Graph API Router (Synchronous)
# ==============================================================================
# Graph API Router (Synchronous)
# ==============================================================================
```
Также артефакт комментария на строке 484: `# Graph API Router (Synchronous)` внутри try-блока search endpoint.

### 4.3. Отсутствие rate limiting на API

Нет защиты от DoS. Каждый запрос на индексацию запускает тяжелую GPU-задачу. Отсутствие лимитов позволяет исчерпать ресурсы сервера простым повтором запросов.

### 4.4. Логгирование на уровне INFO для каждого файла при detect_file_type

**Файл:** `src/chunker.py`
**Проблема:** `detect_file_type()` вызывает `log.info()` для каждого обработанного файла. При индексации крупного проекта это тысячи log-сообщений. Должен быть `log.debug()`.

### 4.5. `requirements.txt` и `pyproject.toml` дублируют зависимости с рассинхроном

Два источника зависимостей -- потенциальный рассинхрон. В `requirements.txt` есть `PySocks`, которого нет в `pyproject.toml`. В `pyproject.toml` есть `httpx[socks]` и `scikit-learn`, которых нет в `requirements.txt`.

### 4.6. Двойная система импортов из-за pytest.ini

**Файл:** `pytest.ini`
```ini
pythonpath = . src
```
**Проблема:** Это создает двойную систему импорта -- модули доступны и как `src.X` и как `X`. В `tests/test_indexing_logic.py` используется `from services import ...`, а в остальных тестах -- `from src.services import ...`. Разные пути к одному и тому же модулю могут вызвать проблемы с кэшированием модулей и непредсказуемое поведение.

### 4.7. Нет graceful shutdown для фоновых задач индексации

При остановке сервера (`service down`) фоновая задача прерывается. `state_file` может быть не записан. Данные в Qdrant могут быть неконсистентны с state_file, что приведет к пропуску файлов при следующей инкрементальной индексации.

### 4.8. `start_line` и `end_line` не сохраняются в payload чанков

**Файл:** `src/services.py`, строки 230-236
**Проблема:** Payload содержит `chunk_index`, но не `start_line` / `end_line`. Функция `extract_snippet()` в `main.py` ожидает `start_line` и `end_line` из payload (`payload.get("start_line", 1)`, `payload.get("end_line", 1)`). При их отсутствии используется fallback (1,1), что возвращает первые строки файла вместо реального контекста чанка.

### 4.9. `qdrant_client` не указан в зависимостях

**Файлы:** `pyproject.toml`, `requirements.txt`
**Проблема:** `qdrant_client` активно используется в `vector_store.py` и `main.py`, но не перечислен ни в одном файле зависимостей проекта. Установка проекта с нуля потребует ручного поиска этой зависимости.

## 5. АРХИТЕКТУРНЫЕ НАБЛЮДЕНИЯ

### 5.1. Monolith с Global State

Глобальные переменные `embedder`, `vector_store`, `TASK_STATUS` делают код сложным для тестирования и масштабирования. Правильное использование Dependency Injection через FastAPI Depends частично решает проблему (уже применяется для `projects_base_dir`), но остальные компоненты глобальны. `main.py` разросся до 667 строк и совмещает routing, бизнес-логику (clustering, boosting), и data transfer objects.

### 5.2. Отсутствие миграции коллекций при смене модели

`project_name_to_collection()` в `services.py` добавляет суффикс к имени коллекции при смене модели, но нет механизма обнаружения старых коллекций и их миграции. Пользователь должен вручную переиндексировать все проекты после смены модели. Старые коллекции остаются в базе навсегда как мертвые данные.

### 5.3. Тестовое покрытие

Юнит-тесты покрывают `graph_analyzer` и `graph_builder` хорошо. Интеграционные тесты для API реально запускают ML-модели (нет mock для embedder в search-тестах), что делает тесты медленными и GPU-зависимыми. Нет тестов для `chunker`, `state_manager`, `vector_store` по отдельности. Нет тестов для кластеризации.
