# Концептуальный Протокол: Аудит и Патч архитектуры `../isearch`

**Источник движения:** Изучение системы локального семантического поиска и построения графов на базе Qdrant.
**Прогон через Мета-Конвейер:** Успешно (Атака по 8 осям Октагона).

## 1. Топология задачи (Ontology)
- **Мета-тип:** RAG-система (Retrieval-Augmented Generation) и экстрактор смыслов из высокоэнтропийной среды (локальных репозиториев кода).
- **Ключевой конфликт (Диалектика):** Полнота контекста (желание прочитать каждый файл) *против* Смыслового Шума (загрязнение векторной БД мусорными машинными кодами, логами и lock-файлами).

## 2. Инварианты (Invariants)
1. **Разделение Топологий:** Правила контроля версий (Git) не равны правилам семантического поиска. То, что нужно хранить в Git, не всегда нужно индексировать для поиска.
2. **Презумпция Мусора:** Любой файл в репозитории считается машинным мусором, пока эвристика не докажет, что он содержит человекочитаемый смысл (Код или Документацию).

## 3. Точка опоры и Векторы Энтропии (Атака Октагоном)
Инкрементальная индексация (по хешам) решает проблему перерасхода Метаболизма при повторных запусках. Однако архитектура уязвима на стыке трех осей:

*   **Дыра 1: Дедлок Топологии (Рабство у `.gitignore`).** 
    В `README.md` признается проблема: система слепо доверяет `.gitignore`. Но файлы вроде `package-lock.json`, `poetry.lock`, сгенерированные `.js.map` или SVG-изображения *обязаны* быть в Git. При этом их загрузка в `sentence-transformers` уничтожит поисковую выдачу мусорным семантическим шумом. iSearch не имеет собственных суверенных границ.
*   **Дыра 2: Дедлок Иммунитета (Смерть Чанкера).** 
    Если в проект случайно попадет дамп базы данных `dump.sql` на 100 МБ или минифицированный `bundle.min.js`, модуль `chunker.py` загрузит его в RAM и начнет резать на куски. Это вызовет либо падение FastAPI по OOM (Out Of Memory), либо паралич GPU на 20 минут. Нет аппаратного предохранителя.
*   **Дыра 3: Дедлок Кинематики (Грязный Shutdown).**
    Асинхронные задачи индексации крутятся в `state_manager.py`. Если пользователь нажмет `CTRL+C` (`make down`) во время индексации большого проекта, процесс записи в Qdrant оборвется. Система не описывает механизм graceful degradation: останутся ли в БД "сиротские" (orphan) вектора, которые приведут к дублированию результатов при следующем запуске?
*   **Дыра 4: Лингвистическая Галлюцинация (Слепота AST).**
    Система заявляет себя как "многоязычная" (мультиязычные эмбеддинги), но архитектурно прибита гвоздями к языку Python. `chunker.py` и `graph_builder.py` используют библиотеку `ast`, которая понимает *только* питоновский код. Для файлов `.ts` или `.rs` система слепнет и режет код тупым текстовым сплиттером, разрушая границы функций и классов (фатальная потеря семантики для RAG).

## 4. План исправления (The Protocol)

Для защиты векторной базы и вычислительных ресурсов необходимо внедрить 4 патча:

- **Шаг 1: Суверенитет Границ (Патч Топологии)**
  Внедрить парсинг файла `.isearchignore`. По умолчанию игнорировать файлы по списку "Анти-семантики" (расширения: `*.lock, *.map, *.min.js, *.csv, *.svg, *.sql`). Git игнорирует артефакты сборки, а `isearch` должен игнорировать *машинно-генерируемый текст*.
- **Шаг 2: Эвристический Рубильник (Патч Иммунитета)**
  В `chunker.py` добавить два жестких ограничения *до* загрузки файла в память:
  1. Лимит на размер (например, пропускать файлы > 1 MB).
  2. Детектор энтропии/бинарности (проверка первых 1024 байт на наличие нулевых байтов `\0` или сверхдлинных строк без переноса).
- **Шаг 3: Атомарные Транзакции Индекса (Патч Кинематики)**
  Внедрить механизм "Shadow Index" (Теневого индекса) или транзакционного коммита. Пока файл не проиндексирован целиком (все его чанки), он не помечается как обновленный в SQLite/Redis трекере хешей. При `CTRL+C` прерванный файл должен переиндексироваться заново при следующем старте, а его старые чанки — зачищаться.
- **Шаг 4: Интеграция Tree-sitter (Патч Наследственности и Семантики)**
  Заменить `import ast` на универсальный парсер абстрактных синтаксических деревьев `tree-sitter`. Это позволит чанкеру и графопостроителю понимать границы классов, функций и импортов для десятков языков (TS, Go, Rust), обеспечивая идеальную семантическую нарезку кода без "мусорных" огрызков.

## 5. Чек-лист приемки (Definition of Done)

Инженерная задача по стабилизации проекта `iSearch` будет решена при закрытии этих пунктов:

- [ ] **Тест Суверенитета:** Создан файл `package-lock.json` с 10 000 строк. При запуске `make index` он игнорируется системой, несмотря на отсутствие в `.gitignore`.
- [ ] **Тест Иммунитета:** В папку проекта помещен `test.log` размером 50 МБ. Индексатор моментально его пропускает, не нагружая GPU и не падая по OOM. В лог `api.log` пишется `SKIPPED: file too large`.
- [ ] **Тест Грязного Прерывания:** Во время индексации 1000 файлов нажимается `CTRL+C`. При повторном `make index` система корректно возобновляет работу, база Qdrant не содержит дубликатов векторов для одного и того же файла.
- [ ] **Тест Истинной Многоязычности:** Скармливание системе файла `.ts` (TypeScript) с функциями. Чанкер нарезает файл строго по границам `function` и `class`, а не по случайным 1000 символам.
